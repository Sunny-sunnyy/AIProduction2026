Core idea - Ý chính

Hoàn thiện hệ thống RAG tự động cho Alex bằng cách dựng một container hóa tác tử (Researcher Agent) cào tin tức tài chính qua Playwright MCP, tinh lọc qua Bedrock Nova Pro, và gọi nạp vào database. Hệ thống được lập lịch định kỳ thông qua sự kết hợp giữa EventBridge và một Lambda Scheduler trung gian làm Proxy chống timeout.

Must know - Cần nhớ

  • Medallion Stage trong Alex: Cào web thô (Bronze) -> Trích lọc qua LLM (Silver) -> Vector hóa lập chỉ mục trong S3 Vectors (Gold).
  • Chicken-and-egg problem: ECR repo cần tạo trước, sau đó build/push image từ local, rồi mới deploy Lambda từ image đó.
  • Timeout: EventBridge chỉ chờ 5s. Phải dùng Lambda Scheduler (timeout 180s) để trigger và chờ Agent phản hồi.

Use in practice - Dùng khi nào

Áp dụng khi cần thiết lập các luồng nạp tri thức chạy ngầm định kỳ (cron job) cho chatbot doanh nghiệp từ các nguồn dữ liệu thay đổi liên tục trên internet (báo chí, tài liệu mạng) một cách bền bỉ và bảo mật.

Core Theory - Lý thuyết cốt lõi

Key concepts - Khái niệm chính

  • Data Engineering - Khoa học dữ liệu: Lĩnh vực tập trung vào thiết kế, xây dựng và quản trị các đường ống (data pipelines) thu thập, biến đổi dữ liệu quy mô lớn một cách tin cậy và chịu lỗi tốt.
  • ETL (Extract, Transform, Load) - Trích xuất, Chuyển đổi, Nạp: Quy trình chuẩn gồm thu thập dữ liệu thô (Extract), chuẩn hóa cấu trúc ngữ nghĩa (Transform), và ghi vào kho đích (Load).
  • LiteLLM - Lớp trừu tượng hóa mô hình: Thư viện trung gian cho phép lập trình viên gọi đa dạng mô hình ngôn ngữ (như Bedrock Nova, OpenAI, Anthropic) bằng cách sử dụng một chuẩn API duy nhất.
  • Function URL - Đường dẫn hàm: Endpoint HTTP(S) duy nhất được gán cho một Lambda function để kích hoạt trực tiếp không cần API Gateway.
  • Agent Tracing - Theo dõi vết tác tử: Cơ chế ghi lại chi tiết các bước suy luận, suy nghĩ (thought) và gọi công cụ (tool call) của Agent để phục vụ giám sát và debug.

Medallion Architecture trong Alex RAG Pipeline

Kiến trúc Huy chương (Medallion) thiết kế kho dữ liệu phân tầng nhằm tăng chất lượng thông tin qua từng giai đoạn:

  • Tầng Bronze (Raw Data): Dữ liệu HTML thô được Playwright MCP cào về từ các trang tài chính Yahoo Finance, Bloomberg. Chứa nhiều mã định dạng và quảng cáo rác.
  • Tầng Silver (Cleaned Data): Bedrock LLM (Nova Pro) nhận dữ liệu thô, loại bỏ nhiễu và tiến hành phân tích, tóm tắt cô đọng các ý quan trọng của doanh nghiệp.
  • Tầng Gold (Curated Data): SageMaker Endpoint thực hiện chuyển đổi văn bản đã làm sạch thành vector nhúng 384 chiều, và Ingest Lambda thực thi lưu vào S3 Vectors sẵn sàng cho Agent trả lời câu hỏi truy vấn.
Deep explanation - Các framework xử lý dữ liệu lớn (Spark và Beam)
Kiến thức mở rộng

Đối với hệ thống dữ liệu lớn của doanh nghiệp, các pipeline ETL thường sử dụng các framework tính toán phân tán:

  • Apache Spark: Engine tính toán phân tán trong bộ nhớ (in-memory computing) cực nhanh, phù hợp cho việc xử lý dữ liệu lớn dạng mẻ (batch processing) và phân tích luồng thời gian thực.
  • Apache Beam: Mô hình lập trình thống nhất cho phép định nghĩa các pipeline xử lý dữ liệu cả dạng mẻ (batch) và dạng luồng (stream), có khả năng chạy trên nhiều execution engines khác nhau như Spark, Flink, hoặc Google Cloud Dataflow.
  • *Ghi chú:* Đối với quy mô của Alex, EventBridge phối hợp Lambda Scheduler và Python script là đủ gọn nhẹ và tối ưu chi phí.

  2. Terraform bỏ qua tài nguyên Lambda function và scheduler (do chưa có image) và chỉ khởi tạo ECR Repository rỗng mang tên alex-researcher cùng phân quyền kéo ảnh (image pull) cho Lambda.
Bước 2: Đóng gói và Đẩy Docker Image lên ECR qua deploy.py
  1. Người dùng thực hiện chạy uv run deploy.py tại thư mục [researcher](file:///G:/AIProduction_t6_2026/production/week3/alex/backend/researcher).
  2. Kịch bản thực thi:
    • Đọc file state của Terraform để lấy địa chỉ ECR Repo URL vừa được tạo.
    • Đăng nhập Docker vào ECR registry bằng password sinh ra từ AWS CLI.
    • Build Docker image bằng lệnh: docker build --platform linux/amd64 -t alex-researcher:deploy-... .
    • Nhãn --platform linux/amd64 bắt buộc được truyền để biên dịch đúng tập lệnh tương thích với môi trường AWS compute, tránh lỗi không tương thích tập lệnh chip (như máy Mac ARM64).
    • Đẩy image lên registry bằng lệnh docker push.
    • Ghi đè tệp biến tạm researcher.auto.tfvars.json lưu lại URI ảnh thực tế.
Bước 3: Hoàn tất triển khai toàn bộ hạ tầng
  1. Người dùng chạy lệnh terraform apply toàn phần không target.
  2. Terraform tìm thấy file cấu hình researcher.auto.tfvars.json chứa image URI, thực hiện:
    • Khởi tạo Lambda function alex-researcher trỏ vào ảnh container trên ECR, cấu hình RAM 2GB, bộ nhớ tạm 2GB, timeout 5 phút, và gán đường dẫn Public Function URL.
    • Triển khai Lambda Scheduler trung gian alex-researcher-scheduler với timeout 180s và truyền Function URL của Agent qua biến môi trường.
    • Triển khai EventBridge Scheduler alex-research-schedule với chu kỳ chạy 2 giờ và gán quyền trigger Lambda Scheduler.
Section 6

Techniques - Kỹ thuật sử dụng

Targeted Apply for Dependency Resolution

Purpose: Phá vỡ vòng lặp phụ thuộc chéo (chicken-and-egg problem) trong IaC khi tài nguyên hạ tầng (Lambda) cần container image, nhưng registry lưu trữ image (ECR) lại được quản lý bởi chính code hạ tầng đó.

When to use: Áp dụng cho mọi luồng triển khai lần đầu tiên của các ứng dụng serverless/container chạy trên AWS, GCP, Azure.

Trade-off: Đòi hỏi quy trình deploy gồm nhiều bước chạy lệnh thủ công hoặc viết script tự động hóa bao quanh.

Proxy Lambda Pattern for Timeout Mitigation

Purpose: Vượt qua giới hạn thời gian chờ phản hồi tối đa 5 giây nghiêm ngặt của EventBridge Scheduler đối với các cuộc gọi API ngoài (API Destinations).

When to use: Khi lập lịch định kỳ kích hoạt các tác vụ nặng như AI Agents suy luận, cào dữ liệu web hoặc chạy báo cáo ETL mất từ 30s đến vài phút.

Trade-off: Tốn thêm một chút chi phí duy trì tài nguyên Lambda Proxy trung gian (tuy nhiên không đáng kể vì RAM Lambda proxy được đặt tối thiểu 256MB).

Tenacity retry logic với Exponential Backoff

Purpose: Tự động gửi lại yêu cầu gọi API khi gặp sự cố mạng hoặc lỗi timeout do container SageMaker Serverless Endpoint bị khởi động lạnh (cold start) sau thời gian dài không hoạt động.

When to use: Áp dụng tại hàm gửi dữ liệu ingest_financial_document của Agent hướng tới Ingest API.

Logic hoạt động: Thử lại tối đa 3 lần. Thời gian chờ giữa các lần thử tăng theo lũy thừa cơ số 2 (1s, 2s, 4s,...) giúp backend có đủ thời gian khởi động container và hồi phục tải.

Docker Layer Caching & Multi-Platform

Purpose: Đảm bảo thời gian build container nhanh nhất bằng cách đặt các câu lệnh cài đặt hệ thống ít thay đổi (Node.js, Playwright Chromium) ở phía trên của Dockerfile. Đồng thời, cờ --platform linux/amd64 đảm bảo ảnh chạy ổn định trên CPU đám mây AWS.

When to use: Khi đóng gói container trên các máy trạm local đa kiến trúc (như Apple Silicon M-series chip ARM64) để đẩy lên môi trường AWS.

# Hàm khởi chạy quy trình suy luận của Agent nghiên cứu đầu tư
async def run_research_agent(topic: str = None) -> str:
    # 1. Định nghĩa prompt truy vấn dựa trên chủ đề hoặc mặc định
    query = f"Research this investment topic: {topic}" if topic else DEFAULT_RESEARCH_PROMPT

    # 2. Cấu hình các biến môi trường bắt buộc để LiteLLM nhận diện Bedrock Region
    region = os.environ.get("BEDROCK_REGION", "us-west-2")
    os.environ["AWS_REGION_NAME"] = region   # LiteLLM bắt buộc sử dụng tham số này
    os.environ["AWS_REGION"] = region
    os.environ["AWS_DEFAULT_REGION"] = region
    
    # 3. Khởi tạo mô hình Bedrock Nova Pro (hoặc GPT OSS 120B) qua lớp LitellmModel
    model_name = os.environ.get("RESEARCHER_MODEL", "bedrock/global.openai.gpt-oss-120b-1:0")
    model = LitellmModel(model=model_name)

    # 4. Kích hoạt context tracing để ghi log
    with trace("Researcher"):
        # Khởi tạo Playwright MCP Server trong container với timeout 2 phút
        async with create_playwright_mcp_server(timeout_seconds=120) as playwright_mcp:
            agent = Agent(
                name="Alex Investment Researcher",
                instructions=get_agent_instructions(), # Tải system prompt
                model=model,
                tools=[ingest_financial_document],     # Công cụ nạp dữ liệu lên S3 Vectors
                mcp_servers=[playwright_mcp],          # Công cụ duyệt web tự động
            )

            # Thực thi vòng lặp Thought-Action Loop của tác tử với giới hạn tối đa 15 lượt
            result = await Runner.run(
                agent,
                input=query,
                max_turns=15,
                hooks=ResearchLoggingHooks() if MCP_LOGGING_ENABLED else None,
            )

    return result.final_output

[tools.py](file:///G:/AIProduction_t6_2026/production/week3/alex/backend/researcher/tools.py) - Tool nạp dữ liệu với cơ chế Retry

Mã nguồn định nghĩa công cụ nạp dữ liệu tri thức của Agent kèm theo bộ lọc retry tự động ngăn lỗi cold start của SageMaker.

# Hàm thô thực hiện gửi yêu cầu POST HTTP tới Ingest API Gateway
def _ingest(document: Dict[str, Any]) -> Dict[str, Any]:
    with httpx.Client() as client:
        response = client.post(
            ALEX_API_ENDPOINT,                # Endpoint API Gateway từ Day 4
            json=document,
            headers={"x-api-key": ALEX_API_KEY}, # Truyền API Key để bypass Usage Plan
            timeout=30.0
        )
        response.raise_for_status()
        return response.json()

# Đính kèm bộ decorator retry từ thư viện tenacity
@retry(
    stop=stop_after_attempt(3),                              # Thử lại tối đa 3 lần trước khi báo lỗi thực sự
    wait=wait_exponential(multiplier=1, min=1, max=10)       # Chờ lũy thừa: 1s, 2s, 4s...
)
def ingest_with_retries(document: Dict[str, Any]) -> Dict[str, Any]:
    """Ingest với retry logic đề phòng lỗi SageMaker Endpoint đang khởi động lạnh (cold start)"""
    return _ingest(document)

# Khai báo decorator function_tool để Agent nhận diện được đây là một công cụ khả dụng
@function_tool
def ingest_financial_document(topic: str, analysis: str) -> Dict[str, Any]:
    """Ingest a financial document into the Alex knowledge base."""
    if not ALEX_API_ENDPOINT or not ALEX_API_KEY:
        return {"success": False, "error": "Alex API not configured. Running in local mode."}
    
    document = {
        "text": analysis,
        "metadata": {
            "topic": topic,
            "timestamp": datetime.now(UTC).isoformat()
        }
    }
    
    try:
        result = ingest_with_retries(document)
        return {
            "success": True,
            "document_id": result.get("document_id"),
            "message": f"Successfully ingested analysis for {topic}"
        }
    except Exception as e:
        return {"success": False, "error": str(e)}

[mcp_servers.py](file:///G:/AIProduction_t6_2026/production/week3/alex/backend/researcher/mcp_servers.py) - Spawn Playwright MCP Stdio

Hàm spawn tiến trình Playwright MCP Server chạy dưới dạng stdio trong container, tự động quét nhị phân trình duyệt Chromium.

def create_playwright_mcp_server(timeout_seconds=120):
    # Cấu hình các tham số chạy Chromium headless bảo mật trong Lambda container
    args = [
        "--headless",
        "--isolated",
        "--no-sandbox",
        "--ignore-https-errors",
        "--user-agent",
        PLAYWRIGHT_USER_AGENT,
    ]

    # Quét thư mục tìm đường dẫn cài đặt thực tế của Chromium binary
    chrome_paths = glob.glob(PLAYWRIGHT_BROWSER_GLOB)
    if chrome_paths:
        chrome_path = chrome_paths[0]
        args.extend(["--executable-path", chrome_path])
    else:
        chrome_path = PLAYWRIGHT_BROWSER_FALLBACK
        args.extend(["--executable-path", chrome_path])

    # Thiết lập file config tạm thời cấu hình các launchOptions cho trình duyệt
    config_path = "/tmp/playwright-mcp.config.json"
    config = {
        "browser": {
            "launchOptions": {
                "args": [
                    "--single-process",   # Lambda chỉ hỗ trợ chạy đơn tiến trình tốt
                    "--no-zygote",
                    "--disable-gpu",       # Lambda container không có GPU vật lý
                ]
            }
        }
    }
    with open(config_path, "w", encoding="utf-8") as config_file:
        json.dump(config, config_file)

    args.extend(["--config", config_path])

    params = {
        "command": "playwright-mcp",       # Lệnh CLI kích hoạt playwright-mcp toàn cục
        "args": args,
        "env": {"DEBUG": "pw:api,pw:browser*"},
    }

    # Trả về đối tượng MCPServerStdio để SDK quản lý giao tiếp qua stdin/stdout
    return MCPServerStdio(params=params, client_session_timeout_seconds=timeout_seconds)

  architectures = ["x86_64"]

    size = 2048                        # Cấp phát 2GB dung lượng ổ đĩa /tmp

    variables = {

      ALEX_API_KEY      = var.alex_api_key

      RESEARCHER_MODEL  = var.researcher_model
    }
  }
}

# 2. Lambda Scheduler làm nhiệm vụ Proxy trung gian tránh timeout 5 giây của EventBridge
resource "aws_lambda_function" "scheduler_lambda" {
  count         = local.scheduler_active ? 1 : 0
  function_name = "alex-researcher-scheduler"

  runtime       = "python3.12"

    variables = {

    }
  }

resource "aws_scheduler_schedule" "research_schedule" {

  target {
    arn      = aws_lambda_function.scheduler_lambda[0].arn

  }

Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ lỗi

  • Lỗi crash loop container trên AWS (Exit code 255): Image build thành công trên local và đẩy lên ECR, nhưng Lambda báo lỗi crash ngay khi start. Do build trên máy Mac M1/M2/M3 (chip ARM64) mà quên không cấu hình platform Linux AMD64.
  • Lỗi Timeout 3 phút của Lambda Scheduler: Lịch trình chạy báo thất bại do Lambda Scheduler bị chết trước khi Agent kịp trả về phản hồi.
  • Lỗi ModelNotSupported từ LiteLLM: Agent báo lỗi không tìm thấy mô hình hoặc tham số không hợp lệ khi gọi Bedrock.
  • Phát sinh chi phí AWS khổng lồ ngoài ý muốn: Học viên để scheduler chạy liên tục qua đêm/cuối tuần khi không học.

Fix / prevention - Cách khắc phục

  • Khắc phục platform: Bắt buộc dùng cờ --platform linux/amd64 khi thực thi lệnh build Docker trong deploy.py và dòng FROM trong Dockerfile.
  • Khắc phục timeout: Cấu hình thuộc tính timeout của Lambda Scheduler trong Terraform lên tối thiểu 180 giây (3 phút) hoặc 300 giây.
  • Khắc phục region LiteLLM: Khai báo và gán giá trị đồng thời cho cả 3 biến môi trường AWS_REGION_NAME, AWS_REGION và AWS_DEFAULT_REGION trong mã nguồn Python.
  • Khắc phục chi phí rác: Đặt cấu hình scheduler_enabled = false trong terraform.tfvars và chạy terraform apply để hủy scheduler. Hoặc chạy terraform destroy toàn bộ khi nghỉ cuối tuần.

Flashcards / Q&A - Thẻ ôn tập

Q: Tại sao AWS EventBridge không thể gọi trực tiếp public Function URL của Researcher Agent trong thực tế?

A: Do giới hạn timeout nghiêm ngặt của EventBridge Scheduler đối với các endpoint gọi HTTP tối đa chỉ là 5 giây, trong khi Researcher Agent cần từ 30s đến 90s để hoàn thành tác vụ duyệt web và phân tích.

Q: Sự khác nhau trong cấu hình của LiteLLM so với Boto3 khi chỉ định Region cho Bedrock là gì?

A: Boto3 sử dụng biến môi trường tiêu chuẩn `AWS_REGION` hoặc `AWS_DEFAULT_REGION`. Trong khi đó, thư viện LiteLLM bắt buộc lập trình viên phải khai báo biến `AWS_REGION_NAME` để nhận dạng đúng endpoint Bedrock.

Q: Cơ chế retry lũy thừa (tenacity) giải quyết vấn đề gì của SageMaker Serverless Endpoint?

A: Giải quyết vấn đề container bị tắt khi không có tải. Lần gọi đầu tiên của Agent sẽ đánh thức container (mất khoảng 10-15s khởi động lạnh gây lỗi timeout), cơ chế retry sẽ thử lại tự động ở giây thứ 1, 2, 4,... để hoàn tất cuộc gọi khi container đã active.

Q: Giải thích bài toán "chicken-and-egg" khi deploy Lambda container và cách giải quyết?

A: Vòng lặp: Lambda cần image trên ECR để khởi tạo, nhưng ECR repository lại được quản lý bởi chính Terraform. Cách giải quyết: dùng lệnh `terraform apply -target=...` chỉ tạo riêng ECR repository trước, build/push image lên, sau đó mới apply toàn phần.
