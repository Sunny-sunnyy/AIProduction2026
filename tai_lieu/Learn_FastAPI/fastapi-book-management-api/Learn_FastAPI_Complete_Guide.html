Foundation

1. API căn bản cho người mới hoàn toàn

Trước khi học FastAPI, bạn cần hiểu API là gì. Nếu phần nền này mơ hồ, những khái niệm như route, request body, response model, status code sẽ rất khó tiêu hóa.

API là gì?

Hiểu đơn giản, API là một "cổng giao tiếp" giữa hai phần mềm. Một bên gửi yêu cầu, bên kia nhận yêu cầu, xử lý, rồi trả kết quả về.

  • Client: bên gọi API, ví dụ browser, mobile app, frontend app, Postman.
  • Server: bên cung cấp API, nhận yêu cầu và trả dữ liệu.
  • JSON: định dạng dữ liệu phổ biến nhất khi trao đổi qua API.
Mental model: frontend giống như người phục vụ gọi món, backend giống như bếp xử lý món ăn, còn API là tờ phiếu order quy định hai bên trao đổi với nhau thế nào.

Request và Response

Mỗi lần client gọi API, thường sẽ có các thành phần sau:

  • URL: gọi endpoint nào.
  • HTTP method: hành động gì, ví dụ xem, tạo, sửa, xóa.
  • Headers: metadata đi kèm.
  • Body: dữ liệu gửi lên, thường dùng khi tạo hoặc sửa.
example-response.json
{
  "id": 1,
  "name": "George Orwell",
  "bio": "Author of 1984"
}
HTTP Method Ý nghĩa Ví dụ trong project
GET Lấy dữ liệu GET /authors, GET /books/1
POST Tạo mới dữ liệu POST /categories, POST /books
PUT Cập nhật dữ liệu PUT /authors/1
DELETE Xóa dữ liệu DELETE /books/1

200 OK

Request thành công và có phản hồi bình thường.

201 Created

Tạo mới thành công, rất hay dùng với POST.

204 No Content

Thành công nhưng không cần trả body, thường dùng khi xóa.

400 Bad Request

Client gửi dữ liệu không hợp lệ về mặt business logic.

404 Not Found

Không tìm thấy resource theo ID hoặc URL.

422 Unprocessable Entity

Dữ liệu gửi lên không khớp schema validation của FastAPI/Pydantic.

FastAPI Core

2. FastAPI là gì và vì sao nó dễ học

FastAPI là Python web framework chuyên để xây API. Điểm mạnh nhất của nó là tận dụng type hints của Python để vừa viết code gọn, vừa tự validation, vừa tự sinh tài liệu API.

Type hints

Khi bạn viết book_id: int hoặc name: str, FastAPI không chỉ dùng để gợi ý code, mà còn dùng để parse dữ liệu và validate luôn.

Pydantic

Pydantic giúp định nghĩa schema dữ liệu. Bạn mô tả data bằng class Python, FastAPI dùng class đó để validate input và format output.

Docs tự sinh

FastAPI tự tạo Swagger UI ở /docs và ReDoc ở /redoc. Đây là lý do nó cực kỳ hợp để học API.

FastAPI đứng trên gì?

  • Starlette: lo phần ASGI/web layer, routing, request/response, middleware.
  • Pydantic: lo phần data validation và serialization.
Nói ngắn gọn: FastAPI là framework "ghép" một web layer mạnh với một data validation layer rất tiện.

Khi nào nên dùng FastAPI?

  • Khi bạn làm backend API bằng Python.
  • Khi bạn muốn docs tự sinh rõ ràng.
  • Khi bạn muốn code sạch nhờ schemas và type hints.
  • Khi bạn cần phát triển nhanh một service hoặc MVP backend.
Lưu ý: FastAPI không tự thay thế database, ORM, auth, queue hay caching. Nó là framework web/API, không phải toàn bộ hệ thống.
First App

3. App FastAPI đầu tiên

Phần này là bước tối thiểu để bạn hiểu cách một FastAPI app được tạo ra và chạy bằng uvicorn.

main.py - ví dụ nhỏ nhất 
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "Hello World"}

Giải thích từng dòng

  • FastAPI() tạo application object trung tâm.
  • @app.get("/") đăng ký route cho URL gốc với method GET.
  • Hàm read_root() là hàm sẽ chạy khi route này được gọi.
  • FastAPI tự chuyển dict Python thành JSON response.

Chạy app

Terminal 
uvicorn app.main:app --reload
  • app.main: module chứa app.
  • :app: object tên app trong module đó.
  • --reload: tự restart server khi code thay đổi.

Micro-example -> Project thật

Trong repo khóa học, file khởi chạy chính là app/main.py. Nó cũng tạo app = FastAPI(...), nhưng ngoài route gốc còn mount static files và include 3 router lớn: authors, categories, books.

Routes

5. Routes, path params, query params và request body

Đây là phần cốt lõi của FastAPI: làm sao định nghĩa URL, lấy dữ liệu từ URL, từ query string, hoặc từ body JSON.

Path parameter

Path parameter nằm ngay trong URL, ví dụ:

Path param 
@app.get("/books/{book_id}")
def get_book(book_id: int):
    return {"book_id": book_id}

book_id được FastAPI parse từ URL sang kiểu int.

Query parameter

Query parameter nằm sau dấu ?, ví dụ:

Query param 
@app.get("/books")
def list_books(skip: int = 0, limit: int = 10):
    return {"skip": skip, "limit": limit}

URL ví dụ: /books?skip=20&limit=10.

Request body

Khi tạo mới dữ liệu, client thường gửi JSON trong body:

Body JSON 
class AuthorCreate(BaseModel):
    name: str
    bio: str | None = None

@app.post("/authors")
def create_author(author: AuthorCreate):
    return author

Trong repo khóa học

  • GET /authors/{author_id} dùng path param.
  • GET /books?author_id=...&year=... dùng query params để filter.
  • POST /books dùng body JSON theo schema BookCreate.
  • POST /books/{book_id}/cover dùng path param + file upload.
Schemas

6. Pydantic schemas: lý do FastAPI vừa gọn vừa chặt chẽ

Schema là nơi bạn mô tả "dữ liệu hợp lệ trông như thế nào". Nếu mô hình này rõ, API của bạn sẽ dễ hiểu, dễ validate, và docs tự sinh cũng đẹp hơn.

Create

Schema dùng khi tạo dữ liệu mới. Các field thường là bắt buộc nếu business logic cần.

Update

Schema dùng khi sửa dữ liệu. Thường để field optional để hỗ trợ partial update.

Response

Schema dùng để trả ra cho client. Đây là lớp rất quan trọng để kiểm soát output.

Ví dụ rút gọn từ project 
from pydantic import BaseModel

class CategoryBase(BaseModel):
    name: str
    description: str | None = None

class CategoryCreate(CategoryBase):
    pass

class CategoryUpdate(BaseModel):
    name: str | None = None
    description: str | None = None

class CategoryInDBBase(CategoryBase):
    id: int

    class Config:
        from_attributes = True

class Category(CategoryInDBBase):
    pass

Tại sao phải tách nhiều schema?

  • Create và Update không có cùng yêu cầu validation.
  • Response model thường có thêm id, timestamp, nested object...
  • Tách schema giúp API sạch, rõ ràng, dễ mở rộng.

Điểm đáng chú ý ở project

Schema Book trong repo dùng nested response: nó không chỉ trả author_id hay category_id, mà còn trả cả object author và category.

Lợi ích: frontend lấy được dữ liệu giàu hơn.
Đánh đổi: response lớn hơn một chút.
Dependencies

7. Depends và get_db(): cơ chế cực quan trọng của FastAPI

Rất nhiều người mới học FastAPI chưa thấy ngay tầm quan trọng của dependency injection. Trong project này, dependency quan trọng nhất là database session.

app/api/deps.py 
from typing import Generator
from app.db.session import SessionLocal

def get_db() -> Generator:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

Nó hoạt động thế nào?

  1. FastAPI thấy route cần db: Session = Depends(get_db).
  2. Nó gọi get_db() trước khi chạy route function.
  3. SessionLocal() tạo session kết nối database.
  4. yield db đưa session đó vào route function.
  5. Sau khi route chạy xong, khối finally luôn đóng session.

Tại sao cách này tốt?

  • Không phải lặp code mở/đóng DB ở mọi route.
  • Giảm nguy cơ quên đóng session.
  • Rất hợp để sau này override trong test.
ORM

8. SQLAlchemy models và quan hệ dữ liệu

Trong project này, FastAPI lo phần web/API, còn SQLAlchemy lo việc map class Python sang bảng database.

Author

authors table chứa id, name, bio. Một tác giả có thể có nhiều sách.

Category

categories table chứa id, name, description. Một danh mục có thể chứa nhiều sách.

Book

books chứa dữ liệu sách và giữ author_id, category_id để liên kết.

erDiagram AUTHORS ||--o{ BOOKS : writes CATEGORIES ||--o{ BOOKS : groups AUTHORS { int id string name text bio } CATEGORIES { int id string name text description } BOOKS { int id string title text description int published_year int author_id int category_id string cover_image datetime created_at datetime updated_at }

Ví dụ từ model Book

app/models/book.py 
author_id = Column(
    Integer,
    ForeignKey("authors.id", ondelete="RESTRICT"),
    nullable=False
)

category_id = Column(
    Integer,
    ForeignKey("categories.id", ondelete="RESTRICT"),
    nullable=False
)

author = relationship("Author", back_populates="books")
category = relationship("Category", back_populates="books")

Đọc ý nghĩa

  • ForeignKey buộc sách phải tham chiếu sang tác giả và danh mục tồn tại.
  • nullable=False nghĩa là không được bỏ trống.
  • relationship() giúp bạn truy cập object liên quan thuận tiện hơn.
  • ondelete="RESTRICT" nói rằng không được xóa record cha khi record con còn tham chiếu.
Migration

9. Alembic migration: biến model thành database schema

Model Python chưa tự động tạo bảng cho bạn trong môi trường bài bản. Alembic giúp quản lý lịch sử thay đổi schema một cách có kiểm soát.

Chu trình cơ bản

Commands 
alembic revision --autogenerate -m "init tables"
alembic upgrade head
  • revision --autogenerate sinh file migration dựa trên model metadata.
  • upgrade head áp migration mới nhất vào database.

Vai trò file

  • alembic.ini: cấu hình chung cho Alembic.
  • migrations/env.py: nơi lấy DB URL và metadata.
  • migrations/versions/*.py: từng revision cụ thể.
flowchart LR A[Edit SQLAlchemy models] --> B[Base.metadata exposes models] B --> C[alembic revision --autogenerate] C --> D[New migration file in versions/] D --> E[alembic upgrade head] E --> F[SQLite app.db schema updated]
Repo note: các điểm đáng lưu ý trong project khóa học
  • app/db/base.py hiện đang comment phần import model. Trong nhiều setup Alembic, nếu metadata không "thấy" model thì autogenerate sẽ không ra bảng như mong muốn.
  • Repo có một revision đầu gần như rỗng, rồi revision sau mới chứa lệnh create_table. Đây là ví dụ rất thực tế cho chuyện migration có thể bị sinh chưa đúng ở lần đầu.
  • Vì vậy, về mặt học tập, bạn nên hiểu concept đúng là: model phải được Alembic nhìn thấy qua metadata trước khi autogenerate.
Practical API

10. CRUD thực chiến trong mini project Book Management API

Project khóa học đi qua đầy đủ CRUD cho Author, Category, và Book. Đây là phần giúp bạn nối các mảnh kiến thức lại với nhau.

Create

Tạo mới thường có flow:

  1. Nhận dữ liệu qua schema Create.
  2. Kiểm tra business rule nếu cần, ví dụ duplicate name.
  3. Tạo ORM object.
  4. db.add(), db.commit(), db.refresh().
  5. Trả object đã tạo theo response model.

Read

Đọc dữ liệu thường có 2 dạng:

  • List: lấy nhiều record, thường có skip, limit, filter.
  • Detail: lấy 1 record theo ID, nếu không có thì trả 404.

Update

Update trong project thường theo chiến lược "chỉ sửa field nào được gửi lên". Vì vậy schema update dùng nhiều field optional.

Delete

Delete sẽ tìm record trước. Nếu có thì xóa và commit. Nếu không có thì trả 404. Nhiều route delete trong repo dùng 204 No Content.

Ví dụ rút gọn từ create_book() 
@router.post("/", response_model=Book, status_code=status.HTTP_201_CREATED)
def create_book(book_in: BookCreate, db: Session = Depends(get_db)):
    author = db.query(models.Author).filter(
        models.Author.id == book_in.author_id
    ).first()
    if not author:
        raise HTTPException(status_code=400, detail="Author does not exist")

    category = db.query(models.Category).filter(
        models.Category.id == book_in.category_id
    ).first()
    if not category:
        raise HTTPException(status_code=400, detail="Category does not exist")

    book = models.Book(
        title=book_in.title,
        description=book_in.description,
        published_year=book_in.published_year,
        author_id=book_in.author_id,
        category_id=book_in.category_id,
    )
    db.add(book)
    db.commit()
    db.refresh(book)
    return book
Điểm hay để học: hàm này cho bạn thấy FastAPI không chỉ nhận request và trả response. Nó còn là nơi phối hợp schema input, ORM model, dependency DB, validation business rule và response model trong cùng một flow rõ ràng.
Query Flow

11. Filter, search và pagination

Đây là phần rất thực tế vì API list hiếm khi chỉ "trả hết dữ liệu". Bạn thường muốn giới hạn số lượng và lọc theo điều kiện.

Logic list_books() rút gọn 
query = db.query(models.Book)

if author_id is not None:
    query = query.filter(models.Book.author_id == author_id)

if category_id is not None:
    query = query.filter(models.Book.category_id == category_id)

if year is not None:
    query = query.filter(models.Book.published_year == year)

if keyword is not None:
    like_pattern = f"%{keyword}%"
    query = query.filter(
        or_(
            models.Book.title.ilike(like_pattern),
            models.Book.description.ilike(like_pattern),
        )
    )

books = query.offset(skip).limit(limit).all()

Tại sao filter nằm trong query params?

Vì filter là cách "chọn dữ liệu nào cần xem" chứ không phải tạo hoặc sửa dữ liệu. Query params rất hợp cho các tình huống này.

Ví dụ URL

sample-request
GET /books?author_id=1&category_id=2&year=1949&keyword=dystopia&skip=0&limit=20
Files

12. Upload file và serve static files

Phần này là điểm nổi bật của project khóa học: upload ảnh bìa sách, lưu file xuống ổ đĩa, rồi trả URL ảnh để frontend hoặc browser mở lại được.

flowchart TD A["Client sends POST /books/{book_id}/cover"] --> B["FastAPI receives UploadFile"] B --> C["Check book exists"] C --> D["Validate content type and extension"] D --> E["Read bytes"] E --> F["Check max size 2MB"] F --> G["Generate unique filename"] G --> H["Write file to app/static/covers"] H --> I["Update book.cover_image in database"] I --> J["Return updated book response"]

2 phần phải có

  • Upload API: nhận file từ request.
  • Static mount: cho phép browser đọc lại file đã lưu.
Static mount 
app.mount("/static", StaticFiles(directory="app/static"), name="static")

Điểm cần nhớ

  • UploadFile phù hợp hơn file bytes thô trong nhiều trường hợp.
  • Luôn validate content_type và extension.
  • Luôn giới hạn size khi cần.
  • Tên file nên có yếu tố unique, project dùng uuid.
Repo note: một rough edge nhỏ trong upload endpoint

Trong upload_book_cover(), repo hiện tại có dòng db.refresh nhưng chưa gọi thành db.refresh(book). Về mặt học tập, ý tưởng của hàm vẫn đúng, nhưng khi làm project thật bạn nên gọi hàm đầy đủ để refresh object sau commit.

13. Swagger UI, ReDoc và cách test API nhanh

/docs

/redoc

14. Architecture và System Design của project

Phần này giúp bạn nhìn project ở góc độ "hệ thống" thay vì chỉ từng file lẻ. Hãy xem kiến trúc này như một sơ đồ đường đi của dữ liệu.

Deep Walkthrough

15. End-to-End Detailed Workflow Walkthrough

Đây là phần quan trọng nhất nếu bạn muốn thật sự hiểu project chạy thế nào từ đầu đến cuối. Chúng ta sẽ đi đúng thứ tự: từ lúc chạy server, tới lúc request đi vào app, tới lúc query database, tới lúc response quay lại client.

1

Giai đoạn khởi chạy server

Bạn chạy lệnh kiểu uvicorn app.main:app --reload. Uvicorn sẽ import module app.main và tìm object tên app trong module đó.

  • File tham gia đầu tiên: app/main.py.
  • Việc quan trọng nhất: tạo ra đối tượng FastAPI.
  • Hàm liên quan: route gốc read_root() cũng được đăng ký ở đây.
2

Giai đoạn import và lắp ráp app

Khi app/main.py được import, nó tiếp tục import các router từ app.api.endpoints. Điều này kéo theo việc Python đọc thêm các file route như authors.py, categories.py, books.py.

  • authors.py khai báo router cho tài nguyên tác giả.
  • categories.py khai báo router cho tài nguyên danh mục.
  • books.py khai báo router cho tài nguyên sách và upload cover.

Sau khi import xong, app.include_router(...) trong main.py gắn các router đó vào app chính với prefix riêng.

3

Giai đoạn cấu hình nền

Các route không hoạt động một mình. Chúng còn phụ thuộc vào config, session và models.

File Vai trò thực tế trong flow
app/core/config.py Tạo object settings, cung cấp PROJECT_NAME và SQLALCHEMY_DATABASE_URI.
app/db/session.py Dùng settings để tạo engine, rồi tạo SessionLocal.
app/api/deps.py Tạo dependency get_db() để cấp session cho từng request.
app/models/*.py Định nghĩa shape của các bảng authors, categories, books.
app/schemas/*.py Định nghĩa input và output shape cho API.
4

Giai đoạn route nhận request

Giả sử client gọi GET /authors. Khi đó FastAPI nhìn vào route map đã đăng ký và tìm ra hàm list_authors() trong app/api/endpoints/authors.py.

Ví dụ route file và các hàm chính:

File route Hàm Nhiệm vụ
authors.py list_authors() Lấy danh sách tác giả với skip và limit.
authors.py get_author() Lấy chi tiết một tác giả theo ID.
authors.py create_author() Kiểm tra trùng tên, tạo tác giả mới.
authors.py update_author() Cập nhật tên hoặc bio, có check duplicate name.
categories.py list_categories(), create_category()... CRUD cho danh mục.
books.py list_books() Lấy danh sách sách kèm filter.
books.py create_book() Check author_id, category_id, rồi tạo sách.
books.py upload_book_cover() Nhận file upload, validate, lưu file, cập nhật URL ảnh.
5

Giai đoạn FastAPI resolve dependencies

Trước khi route function thật sự chạy, FastAPI nhìn vào chữ ký hàm. Nếu nó thấy tham số như db: Session = Depends(get_db), nó sẽ gọi dependency trước.

  • get_db() tạo một session mới từ SessionLocal().
  • Session này được truyền vào tham số db của route function.
  • Sau khi route xong, session được đóng ở khối finally.

Điều này có nghĩa là mỗi request thường có vòng đời DB session riêng.

6

Giai đoạn parse input dữ liệu

Nếu route nhận body JSON, FastAPI sẽ dùng schema Pydantic tương ứng để parse dữ liệu.

  • create_author(author_in: AuthorCreate, ...) dùng AuthorCreate.
  • create_category(category_in: CategoryCreate, ...) dùng CategoryCreate.
  • create_book(book_in: BookCreate, ...) dùng BookCreate.
  • update_* thường dùng schema với field optional.

Nếu dữ liệu không khớp schema, FastAPI trả 422 trước cả khi vào logic route.

7

Giai đoạn chạy business logic trong từng route

Đây là lúc logic "thật sự" của API xảy ra. Route function dùng session để query database, kiểm tra điều kiện, và commit thay đổi nếu cần.

create_author()

  • Query xem tên tác giả đã tồn tại chưa.
  • Nếu trùng thì raise HTTPException(400).
  • Nếu không trùng thì tạo ORM object mới.
  • Add -> commit -> refresh -> return.

create_category()

  • Flow gần giống create_author.
  • Check unique name.
  • Tạo category mới và commit.

create_book()

  • Không check duplicate title.
  • Check author_id có tồn tại không.
  • Check category_id có tồn tại không.
  • Nếu 2 foreign key hợp lệ thì tạo record sách.

update_book()

  • Tìm sách theo ID.
  • Chỉ update field nào người dùng gửi lên.
  • Nếu đổi author_id hoặc category_id thì phải check tồn tại trước.
8

Giai đoạn query database

Session SQLAlchemy dùng các ORM model như models.Author, models.Category, models.Book để tạo câu query. Bạn có thể hình dung:

  • db.query(models.Author) tương đương "làm việc với bảng authors".
  • .filter(models.Author.id == author_id) thêm điều kiện WHERE.
  • .first() lấy một record đầu tiên hoặc None.
  • .all() lấy toàn bộ danh sách kết quả.

Với SQLite, tất cả dữ liệu đang nằm trong file app.db.

9

Giai đoạn serialize response bằng Pydantic

Sau khi route trả ORM object, FastAPI dùng response_model=... để biến object đó thành JSON chuẩn.

  • Ví dụ response_model=Author nghĩa là response phải theo schema Author.
  • Schema nào có from_attributes = True sẽ đọc dữ liệu từ ORM object.
  • Schema Book trong repo trả nested author và category, nên JSON trả về khá giàu dữ liệu.
10

Giai đoạn đóng DB session

Sau khi response đã được tạo, FastAPI quay lại dependency get_db(). Khối finally chạy và gọi db.close().

Đây là bước dọn dẹp tài nguyên rất quan trọng để request sau không dùng chung session theo cách không kiểm soát.

11

Workflow riêng của upload ảnh bìa

Upload khác CRUD JSON ở chỗ nó nhận file nhị phân thay vì body JSON thường. Trong repo, upload_book_cover() là hàm xử lý chính.

  • Tìm sách theo book_id.
  • Kiểm tra MIME type.
  • Kiểm tra extension file.
  • Đọc bytes của file upload.
  • Giới hạn size tối đa 2MB.
  • Tạo filename unique bằng uuid.
  • Ghi file vào app/static/covers.
  • Cập nhật book.cover_image thành URL kiểu /static/covers/....jpg.
  • Commit thay đổi, rồi trả object sách đã có URL ảnh.
12

Workflow của static file serving

Sau khi ảnh đã được lưu vào đĩa, route ảnh không cần viết thủ công nữa. Vì main.py đã mount:

Static files 
app.mount("/static", StaticFiles(directory="app/static"), name="static")

Nghĩa là browser có thể gọi trực tiếp URL như /static/covers/book_1_xxx.jpg, và FastAPI/Starlette sẽ trả file đó ra làm response.

13

Workflow của migration setup

Đây là flow không xảy ra ở mỗi request, nhưng rất quan trọng lúc khởi tạo project hoặc thay đổi schema database.

  1. Bạn sửa model trong app/models.
  2. Alembic chạy migrations/env.py.
  3. env.py lấy DB URL từ app/core/config.py.
  4. target_metadata = Base.metadata giúp Alembic nhìn vào models.
  5. revision --autogenerate sinh file mô tả thay đổi schema.
  6. upgrade head áp thay đổi đó xuống database thật.

Tóm gọn toàn bộ luồng chạy

Uvicorn gọi app.main -> FastAPI app được tạo -> routers được include -> client gửi request -> FastAPI tìm đúng route function -> resolve Depends(get_db) -> route query/commit qua SQLAlchemy Session -> dữ liệu ORM được serialize bằng Pydantic schema -> response trả về client -> session được đóng.

Common Pitfalls

16. Lỗi thường gặp và cách nghĩ đúng

Phần này đặc biệt hữu ích vì nó nối những lỗi của người mới với chính rough edges trong repo khóa học.

1. Nhầm giữa SQLAlchemy model và Pydantic schema

Model dùng để map sang bảng database. Schema dùng để định nghĩa dữ liệu vào/ra của API. Hai thứ này liên quan nhưng không giống nhau.

2. Path parameter đặt tên không nhất quán

Trong project có route như "/{cate_id}" nhưng hàm lại nhận category_id. Về mặt thiết kế, bạn nên luôn đồng nhất tên path param với tên tham số để tránh lỗi và tránh gây nhiễu khi đọc code.

3. Tưởng rằng migration luôn sinh đúng ngay lần đầu

Thực tế không phải vậy. Nếu metadata không được expose đúng, file migration có thể rỗng hoặc thiếu. Vì thế bạn nên luôn đọc migration file sau khi autogenerate.

4. Quên rằng FastAPI tự validate trước khi vào route logic

Nếu body không khớp schema, bạn sẽ nhận 422 và route function có thể còn chưa chạy. Đây là behavior chuẩn của FastAPI, không phải lỗi ngẫu nhiên.

5. Nhầm giữa validation kỹ thuật và validation business

name: str là validation kỹ thuật về kiểu dữ liệu. Còn "author name không được trùng" hay "author_id phải tồn tại" là validation business logic, và bạn phải tự viết trong route hoặc service layer.
