Bản chất của Infrastructure as Code (IaC)§

Infrastructure as Code (IaC) là phương pháp quản lý và thiết lập hạ tầng máy tính thông qua các tệp mã nguồn được định nghĩa rõ ràng, thay vì thực hiện các thao tác thủ công trên giao diện đồ họa Web hoặc sử dụng các kịch bản cài đặt ad-hoc không đồng nhất.

Why this matters: Nỗi đau của Click-Ops

Khi bạn tạo một máy ảo EC2 trên AWS Console bằng cách click tay: Bạn chọn AMI, Instance Type, tạo Security Group, đính kèm IAM Role. Lần đầu thì dễ. Nhưng nếu bạn cần tạo lại chính xác hạ tầng đó ở vùng (region) khác, hoặc khi xảy ra sự cố thảm họa (Disaster Recovery)? Bạn sẽ rất dễ quên hoặc click sai một tùy chọn nhỏ, dẫn đến cấu hình không nhất quán và mất nhiều giờ để gỡ lỗi.

What this means: Hạ tầng của bạn là mã nguồn

Với IaC, toàn bộ VPC, Subnet, máy chủ, cơ sở dữ liệu của bạn được mô tả chi tiết trong các file text. Bạn có thể lưu chúng vào Git, xem lịch sử thay đổi (version control), review code (Pull Request) trước khi triển khai và tạo lại y hệt 100% môi trường chỉ bằng một dòng lệnh.

Khía cạnh Quản lý thủ công (Click-Ops) Hạ tầng dưới dạng mã (IaC)
Tính nhất quán Thấp (Dễ sai lệch cấu hình giữa các môi trường dev/prod). Tuyệt đối (Mã nguồn chạy nhiều lần đều ra kết quả giống nhau).
Tốc độ triển khai Chậm (Phụ thuộc vào tốc độ nhấp chuột của con người). Nhanh (Thực thi tự động chỉ trong vài phút).
Khôi phục thảm họa Phức tạp (Phải cấu hình lại từ đầu dựa trên tài liệu giấy). Tức thì (Chạy lệnh apply lại từ mã nguồn lưu trên Git).

Terraform là gì và mô hình Declarative§

HashiCorp Terraform là một công cụ IaC mã nguồn mở cực kỳ phổ biến giúp bạn định nghĩa, xây dựng và thay đổi hạ tầng một cách an toàn và hiệu quả trên nhiều nhà cung cấp đám mây khác nhau (như AWS, Azure, GCP).

What this means: Mô hình Khai báo (Declarative Model)

Trong lập trình thông thường (Imperative), bạn viết mã mô tả từng bước thực hiện: "Bước 1 tạo đĩa S3, Bước 2 gán quyền cho Lambda...". Nhưng với Terraform, bạn viết theo kiểu Declarative (Khai báo trạng thái mong muốn): "Tôi muốn có một S3 Bucket tên là A, một Lambda tên là B có quyền đọc Bucket A". Terraform sẽ tự tính toán xem thứ tự tạo thế nào cho đúng và chạy các API tương ứng.

Why this matters: Resource Dependency Graph

Khi bạn định nghĩa hàng chục tài nguyên liên kết với nhau, Terraform tự động xây dựng một biểu đồ phụ thuộc (Dependency Graph) dưới dạng đồ thị có hướng không chu trình (DAG). Nó tự biết tài nguyên nào không phụ thuộc nhau để tạo song song nhằm tối ưu thời gian, và tài nguyên nào phụ thuộc nhau để tạo theo trình tự bắt buộc.

Sơ đồ quy trình hoạt động cốt lõi của Terraform

1. Viết cấu hình (.tf files)
↓
2. Khởi tạo môi trường (terraform init)
↓
3. Lên kế hoạch (terraform plan)
↓
4. Áp dụng thực tế (terraform apply)

So sánh Terraform với các công cụ khác§

Người mới thường bị nhầm lẫn giữa Terraform với các công cụ tự động hóa hoặc IaC khác như AWS CloudFormation, Ansible, hoặc Shell Script.

Beginner note: Phân biệt Provisioning vs Configuration Management

Provisioning (Khởi dựng hạ tầng): Tạo ra phần cứng, ảo hóa như mạng VPC, máy chủ EC2, ổ cứng S3. Terraform cực kỳ mạnh ở mảng này.
Configuration Management (Cấu hình hệ điều hành): Cài đặt phần mềm, cấu hình nginx, thiết lập database bên trong máy chủ ảo. Ansible, Chef, Puppet mạnh ở mảng này.

Tiêu chí Terraform CloudFormation Ansible Shell Script
Phạm vi Multi-cloud (AWS, GCP, Azure, Kubernetes) Chỉ dành riêng cho AWS Cấu hình phần mềm bên trong OS Kịch bản lệnh chạy cục bộ ad-hoc
Cơ chế Declarative (Khai báo) Declarative (Khai báo) Hỗn hợp (Hybrid) Imperative (Tuần tự từng bước)
Quản lý State Có (Lưu trong tệp tin State) Có (AWS quản lý ngầm) Không có tệp tin State lưu trữ Không có quản lý State
Why this matters: Multi-Cloud Awareness

Mặc dù mỗi nhà cung cấp đám mây có một bộ Provider riêng và bạn không thể lấy trực tiếp code AWS chạy sang Azure được, nhưng việc dùng chung một cú pháp HCL và cùng một quy trình (init, plan, apply) giúp đội ngũ kỹ sư tiết kiệm rất nhiều thời gian học tập khi làm việc trên môi trường đa đám mây.

Thiết lập môi trường trên Windows và cấu hình AWS CLI§

Để bắt đầu thực hành, chúng ta cần cài đặt Terraform CLI, AWS CLI và cấu hình khóa truy cập (Credentials) một cách an toàn trên Windows.

Cảnh báo bảo mật cực kỳ quan trọng về Secret Key

Tuyệt đối không bao giờ được hard-code hoặc lưu trực tiếp AWS Access Key ID và AWS Secret Access Key vào trong các tệp cấu hình Terraform. Nếu bạn vô tình đẩy chúng lên GitHub, kẻ xấu có thể quét và chiếm quyền kiểm soát tài khoản AWS của bạn chỉ trong vài phút, gây ra các hóa đơn hàng ngàn USD.

PowerShell - Cài đặt và cấu hình 
# 1. Tải và cài đặt Terraform bằng Chocolatey trên Windows (Nếu có)
choco install terraform

# 2. Hoặc kiểm tra xem Terraform đã được nhận diện trong Path chưa
terraform -version

# 3. Cấu hình AWS Credentials thông qua AWS CLI
# Lệnh này sẽ nhắc bạn nhập Access Key, Secret Key, Region mặc định và Output Format.
aws configure --profile terraform-learning

# 4. Xác minh thông tin đăng nhập và quyền truy cập hiện tại
aws sts get-caller-identity --profile terraform-learning
What this means: Cách đọc phản hồi từ lệnh STS

Khi chạy lệnh aws sts get-caller-identity, nếu nhận được một phản hồi dạng JSON chứa các trường UserId, Account, và Arn của User IAM mà bạn vừa cấu hình, nghĩa là máy tính của bạn đã kết nối thành công tới AWS API.

Mô hình tư duy và chu trình hoạt động của Terraform§

Để không bị bối rối khi làm việc với Terraform, bạn cần ghi nhớ sâu sắc chu trình 4 bước hoạt động chính của nó.

Chu trình 4 bước cốt lõi

1. Viết cấu hình: Bạn định nghĩa các tài nguyên mong muốn trong các file .tf.
2. Khởi tạo (Init): Chạy terraform init để tải về các Provider (ví dụ: Provider AWS để giao tiếp với AWS API).
3. Lập kế hoạch (Plan): Chạy terraform plan để so sánh mã nguồn của bạn với hạ tầng thực tế đang chạy trên AWS. Nó sẽ liệt kê ra những gì sẽ được tạo mới (+), cập nhật (~), hoặc xóa bỏ (-).
4. Thực thi (Apply): Chạy terraform apply để thực hiện các thay đổi đó lên đám mây AWS.

Why this matters: Tầm quan trọng của tệp State

Làm thế nào Terraform biết hạ tầng thực tế đang có gì để so sánh? Đó là nhờ tệp tin terraform.tfstate. Đây là cơ sở dữ liệu nội bộ lưu giữ ánh xạ giữa mã nguồn của bạn và ID của các tài nguyên thực tế trên AWS. Tệp này phải được bảo vệ cực kỳ nghiêm ngặt.

Cấu trúc thư mục chuẩn của một dự án Terraform§

Mặc dù bạn có thể viết tất cả mã nguồn vào duy nhất một file main.tf, nhưng trong môi trường thực tế, việc phân tách các tệp tin theo chức năng là bắt buộc để dễ quản lý và review.

Cấu trúc thư mục khuyên dùng cho dự án
project-root/
├── versions.tf          # Định nghĩa phiên bản Terraform và Providers
├── provider.tf          # Khai báo cấu hình kết nối của Provider (AWS)
├── main.tf              # Nơi khai báo các tài nguyên chính
├── variables.tf         # Khai báo các tham số đầu vào (Input Variables)
├── terraform.tfvars     # Giá trị thực tế gán cho các Variables
├── outputs.tf           # Các thông tin đầu ra cần hiển thị sau khi apply
├── locals.tf            # Các biến tính toán nội bộ để tái sử dụng
├── backend.tf           # Khai báo nơi lưu trữ tệp State (S3 Remote State)
└── modules/             # Thư mục chứa các module con tự định nghĩa
    └── vpc/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
Why this matters: Tách cấu hình khỏi dữ liệu thực tế

Bằng việc tách variables.tf (khai báo khuôn mẫu) và terraform.tfvars (chứa giá trị cụ thể của từng môi trường), bạn có thể tái sử dụng toàn bộ mã nguồn của dự án để tạo ra môi trường Dev, Staging, và Prod mà không cần phải viết lại logic hạ tầng.

Cú pháp ngôn ngữ cấu hình HashiCorp (HCL)§

HCL (HashiCorp Configuration Language) được thiết kế để con người dễ đọc và máy tính dễ phân tích. Dưới đây là các kiểu dữ liệu và cú pháp lập trình chức năng cơ bản.

hcl_examples.tf - Các cú pháp cơ bản 
# 1. Các kiểu dữ liệu cơ bản
variable "project_name" {
  type    = string
  default = "ai-system"
}

variable "instance_count" {
  type    = number
  default = 3
}

variable "enable_public_ip" {
  type    = bool
  default = true
}

# 2. Kiểu dữ liệu phức hợp (List, Map, Object)
variable "availability_zones" {
  type    = list(string)
  default = ["ap-southeast-1a", "ap-southeast-1b"]
}

variable "tags" {
  type = map(string)
  default = {
    Environment = "dev"
    ManagedBy   = "terraform"
  }
}

# 3. Sử dụng biểu thức điều kiện (Ternary Operator)
locals {
  instance_type = var.enable_public_ip ? "t3.micro" : "t3.nano"
}

# 4. Vòng lặp for_each và count
resource "aws_s3_bucket" "buckets" {
  for_each = toset(["logs", "data", "artifacts"])
  bucket   = "${var.project_name}-${each.key}"
}
What this means: Khác biệt giữa Count và For_Each

count: Thích hợp khi bạn muốn tạo ra N tài nguyên hoàn toàn giống hệt nhau (chỉ khác số thứ tự 0, 1, 2). Tuy nhiên, nếu bạn xóa một phần tử ở giữa danh sách, Terraform sẽ phải hủy và tạo lại các phần tử phía sau.
for_each: Thích hợp khi các tài nguyên được phân biệt bằng các key duy nhất (như map hoặc set). Xóa một phần tử không ảnh hưởng đến các tài nguyên còn lại.

Dự án đầu tiên trên AWS: Tạo máy chủ EC2 đơn giản§

Hãy thực hành viết và chạy dự án Terraform đầu tiên của bạn để khởi tạo một máy chủ EC2 trên AWS.

versions.tf - Cấu hình phiên bản 
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
provider.tf - Cấu hình Provider AWS 
provider "aws" {
  region  = "ap-southeast-1"
  profile = "terraform-learning"
}
Beginner note: Sử dụng Data Source để tự động hóa việc lấy AMI ID

Thay vì hard-code một ID máy ảo (AMI) cố định (rất dễ bị lỗi hoặc lỗi thời khi chuyển đổi Region), chúng ta nên dùng khối data "aws_ami" để tự động truy vấn ID của phiên bản Ubuntu mới nhất từ nhà phát triển Canonical.

main.tf - Tài nguyên EC2 với Dynamic AMI 
# Truy vấn dynamic AMI ID của Ubuntu 22.04 LTS mới nhất từ AWS API
data "aws_ami" "ubuntu" {
  most_recent = true

  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }

  owners = ["099720109477"] # ID của hãng Canonical phát triển Ubuntu
}

resource "aws_instance" "my_first_server" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"

  tags = {
    Name = "my-first-terraform-ec2"
  }
}
outputs.tf - Hiển thị Public IP 
output "server_public_ip" {
  description = "Địa chỉ IP công cộng của EC2 instance"
  value       = aws_instance.my_first_server.public_ip
}
Lưu ý cho người mới: Quy trình thực thi lệnh

Hãy mở PowerShell tại thư mục chứa các file trên và chạy tuần tự các lệnh sau:

# 1. Định dạng code cho đẹp và chuẩn
terraform fmt

# 2. Khởi tạo thư mục và tải Provider AWS
terraform init

# 3. Xác thực cú pháp xem có lỗi chính tả nào không
terraform validate

# 4. Xem trước kế hoạch thay đổi hạ tầng

# 5. Áp dụng triển khai (Nhập 'yes' khi được hỏi)

Vòng đời tài nguyên và tư duy thay đổi hạ tầng§

Làm thế nào Terraform xử lý khi bạn cập nhật mã nguồn của một tài nguyên đang chạy?

What this means: 3 kịch bản cập nhật tài nguyên

1. Update In-Place (Cập nhật tại chỗ): AWS cho phép thay đổi tham số đó mà không cần tắt máy ảo (ví dụ: đổi tag tên).
2. Destroy and Recreate (Hủy và tạo mới): Một số tham số đặc thù của AWS bắt buộc phải tạo mới tài nguyên (ví dụ: thay đổi Subnet ID của EC2). Terraform sẽ hủy máy cũ trước, sau đó tạo máy mới.
3. Immutable Infrastructure (Hạ tầng bất biến): Thay vì sửa chữa máy chủ đang chạy, ta luôn tạo máy mới hoàn toàn để thay thế để đảm bảo độ tin cậy.

Why this matters: Khối Lifecycle Meta-Argument

Nếu bạn cập nhật tài nguyên bắt buộc phải Recreate, việc hủy máy cũ trước sẽ làm gián đoạn hệ thống (Downtime). Bạn có thể dùng tùy chọn create_before_destroy = true để Terraform tạo máy mới chạy ổn định trước, rồi mới hủy máy cũ.

lifecycle_example.tf - Cấu hình vòng đời 
resource "aws_instance" "lifecycle_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  lifecycle {
    # Tạo tài nguyên mới chạy ổn định rồi mới hủy tài nguyên cũ
    create_before_destroy = true
    
    # Ngăn chặn việc vô tình hủy tài nguyên quan trọng (như database)
    prevent_destroy = false
    
    # Bỏ qua các thay đổi đối với tag nếu hệ thống bên ngoài tự động thay đổi chúng
    ignore_changes = [
      tags,
    ]
  }
}

Biến, Dữ liệu nội bộ, Đầu ra và Chiến lược gắn nhãn AWS§

Quản lý mã nguồn sạch sẽ bằng cách sử dụng hiệu quả Variables, Locals, Outputs và thiết lập chiến lược phân loại hạ tầng.

What this means: Cách dùng từng loại biến

Variables (Tham số đầu vào): Dùng để truyền cấu hình từ ngoài vào (ví dụ: chọn Region, Instance Type). Có thể cấu hình giá trị mặc định, kiểm tra định dạng dữ liệu (validation).
Locals (Biến tính toán nội bộ): Dùng như hằng số trong code, gán các chuỗi tính toán lặp đi lặp lại để tránh lặp code (DRY - Don't Repeat Yourself).
Outputs (Dữ liệu đầu ra): Xuất thông tin ra màn hình sau khi chạy xong (ví dụ: DNS name của Load Balancer) hoặc truyền sang dự án Terraform khác.

variables_and_tags.tf - Ví dụ thực tế 
variable "environment" {
  type        = string
  description = "Môi trường triển khai hạ tầng"
  
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Môi trường triển khai bắt buộc phải thuộc một trong các giá trị: dev, staging, prod."
  }
}

locals {
  name_prefix = "my-company-${var.environment}"
  common_tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Project     = "data-analytics"
  }
}

# Sử dụng locals để tạo tên tài nguyên đồng nhất
resource "aws_s3_bucket" "analytics_data" {
  bucket = "${local.name_prefix}-analytics-bucket"
  tags   = local.common_tags
}

Các viên gạch nền tảng của AWS trong Terraform§

Để xây dựng một kiến trúc chuẩn trên AWS, bạn cần nắm vững cách viết mã nguồn Terraform để khởi dựng các tài nguyên cốt lõi dưới đây.

Why this matters: Mạng VPC là nền móng

Bất kỳ ứng dụng nào trên AWS đều cần một không gian mạng riêng biệt và bảo mật (VPC). Bạn phải thiết kế Subnet công cộng (Public Subnet) cho Load Balancer/Web Server hướng ra Internet, và Subnet riêng tư (Private Subnet) cho Database/AI model để tránh bị tấn công trực tiếp.

Nhấn vào đây để xem toàn bộ mã nguồn mô tả VPC, Subnets, SG, ALB và IAM
aws_infra_blocks.tf 
# 1. Tạo mạng VPC riêng biệt
resource "aws_vpc" "custom_vpc" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true
  tags                 = { Name = "custom-vpc" }
}

# 2. Tạo Subnet Public (Có Route hướng ra Internet Gateway)
resource "aws_subnet" "public_subnet" {
  vpc_id                  = aws_vpc.custom_vpc.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "ap-southeast-1a"
  map_public_ip_on_launch = true
  tags                 = { Name = "public-subnet-1a" }
}

# 3. Tạo Subnet Private (Không cho phép truy cập trực tiếp từ Internet)
resource "aws_subnet" "private_subnet" {
  vpc_id            = aws_vpc.custom_vpc.id
  cidr_block        = "10.0.2.0/24"
  availability_zone = "ap-southeast-1b"
  tags              = { Name = "private-subnet-1b" }
}

# 4. Thiết lập Internet Gateway kết nối VPC ra bên ngoài
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.custom_vpc.id
  tags   = { Name = "custom-igw" }
}

# 5. Tạo nhóm bảo mật (Security Group)
resource "aws_security_group" "web_sg" {
  name        = "allow-http-web"
  description = "Cho phep truy cap HTTP cong cong"
  vpc_id      = aws_vpc.custom_vpc.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

Xây dựng Modules để tái sử dụng cấu trúc hạ tầng§

Module trong Terraform tương tự như hàm (function) trong lập trình. Nó cho phép bạn gom các tài nguyên liên quan lại với nhau để tái sử dụng nhiều lần.

What this means: Root Module vs Child Module

Root Module: Là thư mục chính nơi bạn chạy các lệnh terraform apply.
Child Module: Là các module được gọi từ bên ngoài (hoặc từ thư mục con modules/) để khởi tạo các thành phần nhỏ hơn.

Sử dụng module trong main.tf ở thư mục gốc 
module "production_vpc" {
  source      = "./modules/vpc"
  vpc_cidr    = "172.16.0.0/16"
  vpc_name    = "prod-network"
  environment = "production"
}

# Lấy output từ module VPC để truyền vào EC2
resource "aws_instance" "app_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.medium"
  subnet_id     = module.production_vpc.private_subnet_ids[0]
}
Why this matters: Độc lập hóa và Đóng gói

Bằng cách đóng gói VPC thành một module chuẩn, đội phát triển ứng dụng chỉ cần khai báo CIDR block mà không cần quan tâm đến cách cấu hình chi tiết Internet Gateway, Route Table hay Subnet. Điều này tăng tốc độ phát triển và giảm thiểu sai sót đáng kể.

Quản lý State và cấu hình lưu trữ Remote State trên AWS S3§

Mặc định, Terraform lưu tệp state ngay tại máy tính của bạn (local state). Nhưng khi làm việc nhóm, bạn bắt buộc phải cấu hình Remote State.

Why this matters: Rủi ro của việc dùng chung local state

Nếu hai lập trình viên cùng chỉnh sửa mã nguồn và chạy terraform apply trên máy cá nhân của họ mà không đồng bộ file state, họ sẽ ghi đè và phá hủy hạ tầng của nhau. Hơn nữa, tệp state chứa các thông tin nhạy cảm ở dạng clear-text (như mật khẩu cơ sở dữ liệu), không bao giờ được commit lên Git.

Cấu hình S3 Backend hiện đại với native locking (Terraform 1.10+)

Trước đây, ta cần DynamoDB để khóa file state (State Locking). Từ các phiên bản mới của Terraform, S3 đã hỗ trợ cơ chế khóa bản ghi thông qua tham số use_lockfile = true cực kỳ tiện lợi.

backend.tf - Cấu hình Remote State 
terraform {
  backend "s3" {
    bucket       = "my-company-terraform-states"
    key          = "global/s3/terraform.tfstate"
    region       = "ap-southeast-1"
    
    # Kích hoạt State Locking trực tiếp bằng cơ chế mới của S3
    use_lockfile = true
  }
}

Xử lý sai lệch hạ tầng (Drift) và Tái cấu trúc mã nguồn§

Trong thực tế, bạn sẽ gặp tình huống hạ tầng trên AWS bị thay đổi thủ công trên Console (Drift), hoặc bạn muốn đổi tên tài nguyên trong code mà không muốn Terraform xóa đi tạo lại.

What this means: Drift là gì?

Drift là hiện tượng xảy ra khi hạ tầng thực tế bị thay đổi nằm ngoài tầm kiểm soát của Terraform (ví dụ: một ai đó đăng nhập AWS Console và xóa một Security Group). Chạy lệnh terraform plan sẽ ngay lập tức phát hiện sự sai lệch này và đề xuất phương án cập nhật lại.

refactoring_examples.tf - Khối import và moved mới 
# 1. Đưa tài nguyên có sẵn ngoài Console vào quản lý của Terraform (Không cần chạy lệnh CLI cũ)
import {
  to = aws_s3_bucket.legacy_bucket
  id = "my-legacy-bucket-name-on-aws"

  bucket = "my-legacy-bucket-name-on-aws"
}

}

Tích hợp CI/CD tự động hóa hạ tầng đám mây§

Mục tiêu tối thượng của IaC là đưa hạ tầng vào quy trình tích hợp và triển khai liên tục (CI/CD) thông qua Gitlab CI hoặc Jenkins.

  - validate
  - plan
  - apply

validate_code:
  stage: validate
  script:
    - terraform init -backend=false
    - terraform validate

# 2. Tạo bản nháp kế hoạch thay đổi (Lưu artifact)

  stage: plan

      - tfplan

# 3. Apply hạ tầng (Chỉ chạy khi merge vào main và cần phê duyệt thủ công)
apply_infrastructure:
  stage: apply
  script:
    - terraform apply -auto-approve tfplan
  when: manual
  only:
    - main

Chiến lược triển khai không gián đoạn (Zero-Downtime)§

Làm thế nào để cập nhật máy chủ web của bạn mà người dùng không gặp lỗi gián đoạn dịch vụ?

Why this matters: Blue/Green Deployment với Terraform

Blue Environment: Môi trường hiện tại đang chạy ứng dụng phiên bản cũ.
Green Environment: Môi trường mới chạy ứng dụng phiên bản mới.
Terraform sẽ tạo mới hoàn toàn cụm Green, chạy kiểm tra chất lượng (Health Check). Khi cụm Green chạy tốt, Load Balancer sẽ chuyển hướng traffic từ Blue sang Green một cách mượt mà.

Phối hợp Terraform và Ansible trong quản lý cấu hình hệ thống§

Đừng cố gắng biến Terraform thành công cụ cài đặt phần mềm. Hãy sử dụng đúng công cụ cho đúng mục đích.

Ví dụ về sự kết hợp lý tưởng

Bước 1 (Terraform): Provisioning ra mạng VPC, IAM Roles, Security Groups và các máy chủ ảo EC2 mới.
Bước 2 (Ansible): Nhận danh sách IP máy chủ vừa tạo, thực hiện SSH vào để cài đặt Docker, cấu hình Python, tải mã nguồn ứng dụng về chạy.

Bảo mật hạ tầng đám mây và quản lý khóa truy cập§

Các nguyên tắc bảo mật tối thượng khi vận hành Terraform trên AWS.

Nguyên tắc đặc quyền tối thiểu (Least Privilege)

Không bao giờ chạy Terraform bằng tài khoản AWS Root Account. Hãy tạo một IAM User riêng biệt chỉ có các quyền hạn cụ thể cần thiết để quản trị tài nguyên hạ tầng của dự án.

Tích hợp HashiCorp Vault

Đối với môi trường sản xuất lớn, hãy sử dụng HashiCorp Vault để cấp phát các khóa truy cập AWS tạm thời có thời hạn ngắn (Dynamic Credentials), thay vì sử dụng Access Key cố định dài hạn.

Kiến trúc hạ tầng dành riêng cho ứng dụng Trí tuệ nhân tạo (AI)§

Tại sao các kỹ sư AI cần biết cách viết code Terraform?

Phân định vai trò: Terraform làm gì trong dự án AI?

Terraform quản lý: Các tài nguyên chứa mô hình và chạy xử lý (như S3 Buckets chứa weights của mô hình, API Gateway định tuyến request, AWS Lambda chạy inference không máy chủ, ECR Registry chứa Docker images, IAM gán quyền).
Terraform KHÔNG quản lý: Code thuật toán Python của mô hình LLM, quá trình huấn luyện (Training) hoặc điều chỉnh tham số (Hyperparameter tuning).

Dịch vụ AWS cho AI Khi nào nên dùng Chi phí và Dọn dẹp
AWS Lambda Phù hợp cho API suy luận (Inference API) nhẹ, chạy không liên tục, tiết kiệm chi phí. Rẻ (Chỉ trả tiền khi chạy). Hầu như không tốn phí duy trì.
AWS ECS/Fargate Dành cho các mô hình LLM lớn, cần chạy liên tục trên Container Docker và cần GPU. Đắt hơn (Tính phí theo giờ chạy container). Cần giám sát và tắt khi không dùng.

Beginner note: Nguyên tắc thiết kế mạng cho ALB và Dynamic AZs

Một AWS Application Load Balancer (ALB) bắt buộc phải liên kết với ít nhất 2 Public Subnet nằm ở 2 Availability Zone (AZ) khác nhau để đảm bảo tính sẵn sàng cao (High Availability).
Thay vì hard-code các hậu tố như "a", "b", hoặc "c" (vốn dễ lỗi vì không phải vùng AWS nào cũng hỗ trợ các ký tự này giống nhau), chúng ta sử dụng data source aws_availability_zones để truy vấn danh sách AZ khả dụng thực tế của Region và trỏ chỉ mục động (ví dụ: names[0] và names[1]).

  enable_dns_hostnames = true

  state = "available"
}

resource "aws_subnet" "public_1" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = var.public_subnet_1_cidr

  map_public_ip_on_launch = true
  tags = { Name = "${var.project_name}-public-1" }

resource "aws_subnet" "public_2" {

Why this matters: Ingestion Pipeline cần Log để làm gì?

Trong các hệ thống xử lý dữ liệu nhạy cảm cho AI, việc ghi log đóng vai trò sống còn để:

  • Gỡ lỗi (Debug): Phát hiện xem các định dạng tệp tin đầu vào có bị lỗi cấu trúc hoặc kích thước vượt ngưỡng không.
  • Bảo mật (Security & Audit): Ghi vết lại lịch sử hoạt động để chứng minh chỉ có các tệp tin hợp lệ được xử lý, phát hiện hành vi cố tình đẩy mã độc hoặc tệp tin không hợp lệ.

Các lỗi phổ biến (Anti-Patterns) và Gỡ lỗi§

Tổng hợp các sai sót kinh điển và phương án khắc phục nhanh khi triển khai Terraform trên AWS.

Lỗi 1: AccessDenied khi chạy Terraform apply

Nguyên nhân: AWS credentials cấu hình trong profile thiếu quyền truy cập để tạo tài nguyên tương ứng.
Phương án sửa: Kiểm tra lại IAM Policy đính kèm với User IAM, đảm bảo đã cấp đủ quyền tạo tài nguyên tương đương.

Lỗi 2: State Lock Error (Không thể lấy bản khóa ghi state)

Nguyên nhân: Có người dùng khác đang đồng thời apply hạ tầng hoặc tiến trình apply trước đó của bạn bị ngắt đột ngột giữa chừng.
Phương án sửa: Nếu chắc chắn không có ai khác đang chạy, bạn có thể giải phóng bản khóa bằng lệnh terraform force-unlock <LOCK_ID>.

Lỗi 3: Xóa nhầm tài nguyên trên AWS Console gây lỗi lệch State

Nguyên nhân: Tự ý thao tác tay (sửa, xóa) trên Web Console thay vì dùng code.
Phương án sửa: Hãy chạy terraform plan, Terraform sẽ tự động so sánh và đưa ra phương án tạo lại hoặc cập nhật lại đúng như khai báo trong code của bạn.
