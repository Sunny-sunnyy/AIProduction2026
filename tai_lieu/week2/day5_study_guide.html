Section 1

CI/CD & GitHub Actions Basics - Tự động hóa tích hợp liên tục

Ý nghĩa cốt lõi của CI/CD

Tích hợp liên tục (Continuous Integration) và Triển khai liên tục (Continuous Deployment) là trụ cột DevOps giúp tự động hóa quá trình kiểm thử, đóng gói và đẩy mã nguồn mới lên AWS ngay khi có thay đổi trên repository GitHub.

Lợi ích: Loại bỏ hoàn toàn lỗi thao tác thủ công, tăng tốc phát hành sản phẩm, đảm bảo hệ thống luôn trong trạng thái ổn định và có thể tự động khôi phục (rollback) nhanh chóng.

Kiến trúc GitHub Actions

GitHub Actions là nền tảng tự động hóa trực tiếp trong repository GitHub:

  • Workflow: Quy trình tự động hóa được định nghĩa trong file YAML đặt tại thư mục .github/workflows/.
  • Event/Trigger: Sự kiện kích hoạt chạy pipeline (như git push nhánh main, pull request, hoặc trigger tay qua workflow_dispatch).
  • Job: Nhóm các bước (steps) thực thi tuần tự trên cùng một máy ảo độc lập gọi là Runner (ví dụ: máy ảo ubuntu-latest).
Section 2

Remote State & DynamoDB Locks - Quản lý trạng thái từ xa

Khi chạy Terraform ở máy local, file terraform.tfstate được lưu trực tiếp trên đĩa cứng. Tuy nhiên, trong môi trường CI/CD, máy ảo Runner chạy GitHub Actions là một máy ảo tạm thời (ephemeral VM) - nó sẽ bị xóa sạch hoàn toàn ngay sau khi kết thúc công việc. Nếu lưu state local, pipeline tiếp theo sẽ mất vết hạ tầng và cố gắng tạo lại toàn bộ tài nguyên, gây lỗi trùng lặp nghiêm trọng.

S3 Remote State & DynamoDB Lock Table

AWS S3 được dùng làm kho lưu trữ tệp tin .tfstate tập trung trực tuyến. S3 bucket này bắt buộc phải bật tính năng Versioning để lưu lịch sử và khôi phục khi state bị lỗi.

DynamoDB Lock Table (chứa partition key có tên chính xác là LockID) làm nhiệm vụ khóa trạng thái (State Locking). Khi một tiến trình apply đang chạy, một khóa tạm thời được ghi vào DynamoDB. Các tiến trình khác (ví dụ: kỹ sư khác hoặc một pipeline khác kích hoạt song song) cố chạy apply lúc này sẽ bị chặn lại, ngăn chặn lỗi ghi đè gây hỏng file state.

Bootstrapping & Target Apply

Để tạo S3 bucket lưu state và DynamoDB lock table bằng Terraform, ta gặp bài toán "con gà và quả trứng": Cần backend S3 để chạy init, nhưng backend S3 lại yêu cầu S3 bucket đó tồn tại trước.

Giải pháp (Bootstrapping): Viết file cấu hình phụ tạm thời backend-setup.tf định nghĩa S3 & DynamoDB. Chạy lệnh terraform apply khoanh vùng chọn lọc bằng cờ -target để AWS tạo trước các tài nguyên nền móng này. Sau khi tạo xong, cấu hình backend S3 chính thức trong backend.tf và xóa tệp tạm backend-setup.tf đi.

Section 3

OpenID Connect (OIDC) - Xác thực không khóa (Passwordless Auth)

Cách làm truyền thống để cấp quyền cho GitHub Actions truy cập vào AWS là tạo một IAM User với Access Key ID và Secret Access Key dài hạn, rồi lưu vào GitHub Secrets. Phương pháp này mang lại rủi ro bảo mật cực kỳ lớn: nếu tài khoản GitHub bị xâm nhập, hacker sẽ chiếm toàn quyền AWS; khóa dài hạn cũng không tự xoay vòng (rotation).

OpenID Connect (OIDC): Cơ chế xác thực liên kết định danh (Federated Identity) hiện đại. GitHub Actions Runner chứng minh danh tính với AWS Security Token Service (STS) bằng một Web Identity Token ngắn hạn do GitHub ký số. AWS xác thực token đó và cấp quyền tạm thời (credentials tự xoay vòng sau vài phút) cho Actions thực thi mà không cần lưu trữ bất kỳ mật khẩu dài hạn nào.

sequenceDiagram participant GitHub as GitHub Actions Runner participant STS as AWS STS (Security Token Service) participant IAM as AWS IAM Deploy Role participant AWS as AWS Resources (S3, Lambda, CloudFront) GitHub->>GitHub: Khởi tạo Job (Nhận OIDC Token từ GitHub OIDC Provider) GitHub->>STS: Gọi AssumeRoleWithWebIdentity (Gửi OIDC Token) Note over STS: Xác thực OIDC Provider & Thumbprint
Kiểm tra Repo trust policy condition STS-->>GitHub: Cấp AccessKey, SecretKey & SessionToken tạm thời (thời hạn ngắn) GitHub->>AWS: Thực thi Terraform Apply / CLI commands (Ký request qua Token tạm) Note over AWS: Thực thi tạo lập hoặc gỡ bỏ tài nguyên AWS-->>GitHub: Hoàn thành tác vụ

IAM Role Trust Policy

Để AWS cho phép giả lập vai trò (assume role) an toàn, ta khai báo điều kiện trust policy nghiêm ngặt:

  • Khóa aud (audience) bắt buộc phải khớp với sts.amazonaws.com.
  • Khóa sub (subject) bắt buộc phải chứa ràng buộc tên repository chính xác của bạn: repo:owner/repo-name:*. Điều này ngăn chặn các tài khoản GitHub của người khác giả mạo để gọi vào tài khoản AWS của bạn.

Quyền hạn Role (IAM Policy)

IAM Role dành cho GitHub Actions cần được cấp quyền tối thiểu để có thể tự động tạo hạ tầng:

  • Gắn các AWS Managed Policies: Lambda, S3, API Gateway, CloudFront, Bedrock, DynamoDB, ACM, Route 53.
  • Cấp quyền IAM quản trị: Cấp thêm các quyền tạo, cập nhật và gán IAM Role cho Lambda (iam:CreateRole, iam:PassRole, v.v.) thông qua inline policy.
Section 4

Nested Git Repositories Fix - Giải quyết lỗi thư mục Git lồng nhau

Khi sử dụng các công cụ khởi tạo khung dự án (scaffolding tools) tự động như create-next-app cho frontend Next.js hoặc uv init cho backend Python, các công cụ này thường tự động tạo sẵn một thư mục ẩn .git bên trong thư mục con đó.

Lỗi Git Submodule xám

Nếu có thư mục ẩn .git bên trong thư mục con frontend/ hoặc backend/, khi ta chạy lệnh khởi tạo Git và commit ở thư mục root cha, Git cha sẽ coi các thư mục con đó là một Git Submodule chưa được cấu hình.

Symptom: Trên giao diện web GitHub, thư mục frontend/ hiển thị dưới dạng một biểu tượng màu xám có dấu mũi tên và không thể nhấp mở ra xem file con bên trong. Toàn bộ code frontend thực tế bị bỏ qua không được push lên đám mây, khiến pipeline của GitHub Actions bị crash do thiếu tệp package.json.

Cách khắc phục triệt để

Thực hiện các bước dọn dẹp và nạp lại dữ liệu sau để loại bỏ hoàn toàn repo con:

  1. Xóa index tracking cũ của Git cha: 
    git rm --cached frontend backend -f
  2. Xóa thư mục ẩn .git của thư mục con:
    • Windows: Remove-Item -Path frontend/.git -Recurse -Force
    • Linux/Mac: rm -rf frontend/.git
  3. Thêm lại toàn bộ và commit: 
    git add .
git commit -m "Fix nested repositories"

React Input Focus Improvement - Tối ưu hóa trải nghiệm hội thoại

Một lỗi UI nhỏ nhưng gây ảnh hưởng lớn đến trải nghiệm người dùng (UX) trong giao diện chatbot cũ: Khi người dùng nhấn gửi tin nhắn (hoặc nhấn phím Enter), sau khi tin nhắn được gửi đi và API bắt đầu xử lý, ô nhập liệu text input bị mất focus (con trỏ chuột biến mất). Người dùng bắt buộc phải dùng chuột click lại vào ô input thì mới có thể gõ tiếp tin nhắn thứ hai.

Giải pháp sử dụng React useRef

Để kiểm soát trực tiếp phần tử DOM của ô input trong React mà không làm trigger re-render lại component, chúng ta sử dụng hook useRef: 

import { useRef } from 'react';

// Khai báo ref trong component
const inputRef = useRef<HTMLInputElement>(null);

// Liên kết ref vào thẻ input HTML
<input ref={inputRef} type="text" ... />

Tự động lấy lại Focus sau khi gửi

Trong hàm gửi tin nhắn sendMessage, chúng ta thực hiện lấy lại focus cho ô nhập liệu bên trong khối finally của tiến trình gọi API (đảm bảo luôn chạy bất kể API gọi thành công hay thất bại): 

} finally {
    setIsLoading(false);
    // Trì hoãn 100ms để trình duyệt cập nhật lại trạng thái DOM
    setTimeout(() => {
        inputRef.current?.focus();
    }, 100);
}

Bootstrap HCL Code Walkthrough - Khởi tạo OIDC & Backend

Mã nguồn chi tiết của các tệp tin cấu hình Terraform phục vụ cho việc khởi tạo hạ tầng nền móng Remote State và OIDC Role trên AWS:

1. Khởi tạo S3 State Bucket & DynamoDB Lock Table (terraform/backend-setup.tf)

File: terraform/backend-setup.tf • Lưu ý: Thực hiện target apply một lần duy nhất tại workspace default rồi xóa file 
# Bucket luu tru State file chu ky phien ban (Versioning) và ma hoa tĩnh
resource "aws_s3_bucket" "terraform_state" {
  bucket = "twin-terraform-state-${data.aws_caller_identity.current.account_id}"
  
  tags = {
    Name        = "Terraform State Store"
    Environment = "global"
    ManagedBy   = "terraform"
  }
}

resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "terraform_state" {
  bucket                  = aws_s3_bucket.terraform_state.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Bang DynamoDB khoa phien LockID de State Locking hoạt động
resource "aws_dynamodb_table" "terraform_locks" {
  name         = "twin-terraform-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  tags = {
    Name        = "Terraform State Locks"
    Environment = "global"
    ManagedBy   = "terraform"
  }

2. Khai báo OIDC Federated Provider & Trust Policy (terraform/github-oidc.tf)

File: terraform/github-oidc.tf • Trọng tâm: Ràng buộc repo owner/name trong condition block của trust policy 
variable "github_repository" {
  description = "GitHub repository in format 'owner/repo'"
  type        = string
}

# 1. Khởi tạo OIDC Identity Provider cho token.actions.githubusercontent.com
resource "aws_iam_openid_connect_provider" "github" {
  url            = "https://token.actions.githubusercontent.com"
  client_id_list = ["sts.amazonaws.com"]
  
  # Thumbprint ky so xac thuc tu GitHub Actions documentation
  thumbprint_list = ["1b511abead59c6ce207077c0bf0e0043b1382612"]
}

# 2. IAM Role cho phep gia lap danh tinh (Assume Role With Web Identity)
resource "aws_iam_role" "github_actions" {
  name = "github-actions-twin-deploy"
  
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Principal = {
          Federated = aws_iam_openid_connect_provider.github.arn
        }
        Action = "sts:AssumeRoleWithWebIdentity"
        Condition = {
          StringEquals = {
            "token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
          }
          StringLike = {
            # Chỉ cho phép duy nhất repository chỉ định của bạn được giả lập role này
            "token.actions.githubusercontent.com:sub" = "repo:${var.github_repository}:*"
          }
        }
      }
    ]
  })
}

# 3. Custom Inline Policy cap cac quyen IAM Administrator dac thu
resource "aws_iam_role_policy" "github_additional" {
  name = "github-actions-additional"

Section 7

GitHub Actions Workflows YAML - Tệp tin cấu hình tự động hóa

Mã nguồn tệp tin YAML cấu hình pipeline tự động hóa quy trình triển khai hạ tầng và ứng dụng khi có sự kiện đẩy code lên GitHub.

1. Đường ống triển khai tự động hóa full-stack (.github/workflows/deploy.yml)

File: .github/workflows/deploy.yml • Runner: ubuntu-latest • Mục đích: Cài đặt Node/Python/Terraform, assume role OIDC, chạy deploy.sh và invalidate cache CloudFront 
name: Deploy Digital Twin

on:
  push:
    branches: [main] # Trigger tu dong khi push len nhanh main
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy'
        required: true
        default: 'dev'
        type: choice
        options:
          - dev
          - test
          - prod

permissions:
  id-token: write  # Bắt buộc phải có để OIDC xác thực thành công lấy credentials tạm
  contents: read

jobs:
  deploy:
    name: Deploy to ${{ github.event.inputs.environment || 'dev' }}
    runs-on: ubuntu-latest
    environment: ${{ github.event.inputs.environment || 'dev' }}
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          role-session-name: github-actions-deploy
          aws-region: ${{ secrets.DEFAULT_AWS_REGION }}

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install uv
        run: |
          curl -LsSf https://astral.sh/uv/install.sh | sh
          echo "$HOME/.local/bin" >> $GITHUB_PATH

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_wrapper: false  # Tắt wrapper để lấy ra output thô dạng string

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: frontend/package-lock.json

      - name: Run Deployment Script
        run: |
          export AWS_ACCOUNT_ID=${{ secrets.AWS_ACCOUNT_ID }}
          export DEFAULT_AWS_REGION=${{ secrets.DEFAULT_AWS_REGION }}
          chmod +x scripts/deploy.sh
          ./scripts/deploy.sh ${{ github.event.inputs.environment || 'dev' }}
        env:
          AWS_ROLE_ARN: ${{ secrets.AWS_ROLE_ARN }}
          
      - name: Get Deployment Output URLs
        id: deploy_outputs
        working-directory: ./terraform
        run: |
          terraform workspace select ${{ github.event.inputs.environment || 'dev' }}
          echo "cloudfront_url=$(terraform output -raw cloudfront_url)" >> $GITHUB_OUTPUT
          echo "api_url=$(terraform output -raw api_gateway_url)" >> $GITHUB_OUTPUT
          echo "frontend_bucket=$(terraform output -raw s3_frontend_bucket)" >> $GITHUB_OUTPUT

      # Xóa cache CloudFront cũ để người dùng nhận ngay giao diện frontend mới
      - name: Invalidate CloudFront
        run: |
          DISTRIBUTION_ID=$(aws cloudfront list-distributions \
            --query "DistributionList.Items[?Origins.Items[?DomainName=='${{ steps.deploy_outputs.outputs.frontend_bucket }}.s3-website-${{ secrets.DEFAULT_AWS_REGION }}.amazonaws.com']].Id | [0]" \
            --output text)
          
          if [ "$DISTRIBUTION_ID" != "None" ] && [ -n "$DISTRIBUTION_ID" ]; then
            aws cloudfront create-invalidation \
              --distribution-id $DISTRIBUTION_ID \
              --paths "/*"
          fi

      - name: Deployment Summary
        run: |
          echo "✅ Deployment Complete!"
          echo "🌐 CloudFront URL: ${{ steps.deploy_outputs.outputs.cloudfront_url }}"
          echo "📡 API Gateway: ${{ steps.deploy_outputs.outputs.api_url }}"
          echo "🪣 Frontend Bucket: ${{ steps.deploy_outputs.outputs.frontend_bucket }}"
Section 8

Deploy/Destroy Script Upgrades - Cấu hình Backend từ xa

Để tương tác với S3 Remote State, các script deploy/destroy local của chúng ta phải được nâng cấp để truyền các tham số cấu hình backend động thông qua cờ -backend-config.

Nâng cấp lệnh init trong `deploy.sh`

Thay thế câu lệnh init tĩnh bằng việc truyền động tham số S3 Bucket chứa Account ID:

# Lấy Account ID của tài khoản chạy
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
AWS_REGION=${DEFAULT_AWS_REGION:-us-east-1}

# Init động kết nối S3 State Store
terraform init -input=false \
  -backend-config="bucket=twin-terraform-state-${AWS_ACCOUNT_ID}" \
  -backend-config="key=${ENVIRONMENT}/terraform.tfstate" \
  -backend-config="region=${AWS_REGION}" \
  -backend-config="dynamodb_table=twin-terraform-locks" \
  -backend-config="encrypt=true"

Bản vá Dummy package trong `destroy.sh`

Khi chạy destroy trên GitHub Actions runner sạch, code backend Python chưa được build nên không có file zip `lambda-deployment.zip`. Terraform sẽ báo lỗi compile và từ chối chạy destroy.

Giải pháp: Tạo một tệp nén dummy tạm thời trước khi ra lệnh destroy:

# Tạo zip giả lập nếu chưa có
if [ ! -f "../backend/lambda-deployment.zip" ]; then
    echo "Creating dummy lambda package for destroy..."
    echo "dummy" | zip ../backend/lambda-deployment.zip -
fi

Options / Trade-offs - Bản đồ lựa chọn thiết kế

Lựa chọn - Option Ưu điểm - Pros Nhược điểm - Cons Ngữ cảnh lựa chọn - When to choose
AWS Access Keys (Dài hạn) Triển khai nhanh, đơn giản, chỉ cần copy chuỗi key vào GitHub Secrets là chạy ngay. Rủi ro bảo mật cực cao nếu lộ khóa, khóa không tự xoay vòng, vi phạm tiêu chuẩn bảo mật đám mây chuyên nghiệp. Chỉ dùng cho các bài thử nghiệm lab siêu tốc cá nhân, không khuyên dùng cho dự án thực tế.
AWS OIDC (Passwordless) (Recommended) Bảo mật tối đa, không cần lưu trữ khóa dài hạn, tự động xoay vòng credentials tạm, kiểm soát chi tiết repo được gán quyền. Phức tạp trong khâu cấu hình ban đầu (phải tạo OIDC Identity Provider và viết trust policy điều kiện). Bắt buộc áp dụng cho mọi dự án doanh nghiệp thực tế chạy môi trường Production để tuân thủ quy tắc bảo mật.
Flat Monorepo (Gom phẳng) (Recommended) Quản lý tập trung toàn bộ code frontend, backend và cấu hình hạ tầng Terraform trong 1 repo duy nhất, deploy đồng bộ. Repo có kích thước nặng hơn, khó phân tách quyền truy cập mã nguồn đối với team lớn đông lập trình viên. Rất thích hợp cho các dự án serverless gọn nhẹ, các lập trình viên Fullstack AI hoặc dự án vừa và nhỏ.
Git Submodules (Nhánh con) Chia nhỏ repo con độc lập, giữ nguyên lịch sử phát triển riêng của frontend và backend. Thao tác kéo code và cập nhật phiên bản cực kỳ phức tạp, dễ gây lỗi lệch tag phiên bản. Chỉ nên dùng khi các thành phần con là các thư viện chia sẻ được phát triển song song bởi các team chuyên biệt khác.
Section 10

Pitfalls - Bẫy phát sinh lỗi & Cách khắc phục

Các trường hợp dễ gây lỗi (Failure Modes)

  • Lỗi submodule xám trên GitHub: Thư mục frontend/ hiển thị màu xám có dấu mũi tên trên GitHub và không thể click mở ra xem code bên trong.
  • Lỗi OIDC AssumeRoleWithWebIdentity AccessDenied: Pipeline chạy đến bước cấu hình AWS credentials thì báo lỗi từ chối truy cập 403.
  • Lỗi destroy bị crash do thiếu zip file: Lệnh destroy hạ tầng chạy trên máy ảo Actions bị dừng đột ngột và báo lỗi thiếu file lambda-deployment.zip.
  • Lỗi YAML Invalid syntax: GitHub Actions tab báo lỗi đỏ toàn bộ workflow và không cho phép chạy do tệp tin cấu hình YAML không hợp lệ.

Biện pháp ngăn ngừa & Xử lý (Fix / Prevention)

  • Xóa triệt để folder .git con: Nhớ chạy lệnh xóa thư mục quản lý Git ẩn của frontend trước khi add ở thư mục root (chạy lệnh: Remove-Item -Path frontend/.git -Recurse -Force).
  • Kiểm tra điều kiện Trust Policy: Đảm bảo trường github_repository được truyền đúng định dạng owner/repo-name (ví dụ: johndoe/digital-twin) và KHÔNG được chứa URL link https://... ở trước.
  • Bản vá Dummy package: Sử dụng đoạn code nén dummy zip giả lập trong file destroy.sh trước khi thực thi lệnh destroy để lừa bộ biên dịch của Terraform.
  • Không dùng ký tự Tab: Tuyệt đối không sử dụng Tab để thụt dòng trong file YAML. Hãy cài đặt các extension YAML Linter để kiểm tra cú pháp tự động trong IDE.
Section 11

Flashcards & Quick Quiz - Ôn tập tương tác

Q: Tại sao phải bật S3 Bucket Versioning cho bucket chứa file state của Terraform?

A: Để đảm bảo tính an toàn dữ liệu. File state lưu cấu trúc ánh xạ hạ tầng. Nếu quá trình apply bị mất điện hoặc mất mạng giữa chừng gây hỏng file state (corrupted), tính năng Versioning cho phép ta khôi phục lại (roll back) nhanh chóng về phiên bản state cũ ổn định trước đó.

Q: Tại sao OIDC lại được coi là phương thức bảo mật passwordless vượt trội?

A: Vì nó loại bỏ việc phải lưu trữ AWS Access Keys dài hạn trên GitHub. GitHub Actions lấy token ngắn hạn tự động, AWS xác thực token qua OIDC Provider và chỉ cấp quyền tạm thời trong vài phút. Không có rủi ro lộ key vĩnh viễn khi repo bị hack.

Q: Cờ -target trong lệnh terraform apply giải quyết bài toán gì?

A: Giải quyết bài toán bootstrapping (con gà và quả trứng). Cho phép ta chỉ định tạo trước S3 Bucket và DynamoDB Table dùng để lưu state trước khi cấu hình backend S3 được kích hoạt chính thức.

Section 12

Quick Memory - Ghi nhớ nhanh cuối bài

Must remember - Điều bắt buộc phải nhớ

  1. Chạy init động qua cờ -backend-config để nạp S3 bucket chứa Account ID động.
  2. Xóa thư mục ẩn .git ở các folder con để tránh lỗi submodule màu xám trên GitHub.
  3. Ghi đè biến môi trường API URL vào file .env.production của Next.js trước khi build.
  4. Sử dụng React useRef và hàm focus() để sửa lỗi mất trỏ chuột sau khi chat.
  5. Phân quyền OIDC IAM Role bắt buộc phải có quyền quản lý role iam:CreateRole và iam:PassRole.

  • Sự khác biệt giữa sự kiện push và workflow_dispatch trong file deploy.yml là gì?
