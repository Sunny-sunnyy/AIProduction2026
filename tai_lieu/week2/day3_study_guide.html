Section 1

Core Concepts - Khái niệm cốt lõi

Amazon Bedrock là gì?

Amazon Bedrock là dịch vụ đám mây được quản lý hoàn toàn (fully managed serverless service) của AWS. Nó cung cấp một API duy nhất để truy cập và sử dụng các mô hình nền tảng (Foundation Models - FMs) hiệu năng cao từ các công ty AI hàng đầu thế giới (Amazon, Anthropic, Meta, Cohere, AI21 Labs, v.v.).

Tầm quan trọng: Loại bỏ hoàn toàn gánh nặng vận hành hạ tầng phần cứng GPU phức tạp, hỗ trợ cơ chế thanh toán tập trung (unified billing) theo số lượng request thực tế, và cho phép tích hợp sâu sắc với hệ sinh thái IAM/VPC của AWS để bảo mật dữ liệu cấp doanh nghiệp.

Sự khác biệt chiến lược: AWS vs Azure

Trong khi Microsoft Azure đi theo hướng hợp tác độc quyền và tích hợp sâu với OpenAI (Azure OpenAI Service), AWS lại chọn chiến lược đa mô hình (multi-model hub) với Bedrock.

Khách hàng của AWS Bedrock có thể tự do thử nghiệm, so sánh và chuyển đổi giữa các mô hình hàng đầu (như Anthropic Claude, Meta Llama, Mistral, và dòng mô hình Amazon Nova chính chủ) thông qua một giao diện lập trình đồng nhất mà không cần thay đổi cấu trúc mã nguồn cốt lõi.

Section 3

IAM Permissions - Phân quyền hệ thống

Quy trình cấp quyền (Thực hiện bởi Root Account)

  1. Đăng nhập vào AWS Console bằng tài khoản Root User nhạy cảm.
  2. Truy cập dịch vụ IAM → Vào mục User groups → Chọn nhóm TwinAccess đã tạo ở bài trước.
  3. Nhấp vào tab Permissions → Chọn Add permissions → Attach policies.
  4. Tìm kiếm và gắn thêm 2 chính sách quản trị quan trọng:
    • AmazonBedrockFullAccess: Cho phép gọi mô hình, cấu hình và quản trị Bedrock.
    • CloudWatchFullAccess: Cấp quyền ghi log, đọc chỉ số và tạo dashboard giám sát.

Cấp quyền bổ sung & Kích hoạt Model Access

  • Cảnh báo thực tế (Andy C Tip): Hãy bổ sung thêm policy AmazonDynamoDBFullAccess vào nhóm TwinAccess ngay từ bây giờ để tránh phát sinh lỗi phân quyền ghi nhớ ở Day 5.
  • Model access kích hoạt: Sau khi phân quyền IAM, thoát Root account và đăng nhập lại bằng IAM User aiengineer. Đi đến Amazon Bedrock Console → Vào tab Model access ở sidebar trái → Nhấp Modify model access → Tích chọn các dòng model Nova mong muốn và nhấn Save changes để trạng thái chuyển sang Access granted.
Section 4

Converse API - Chuẩn hóa API hội thoại

Trước khi AWS giới thiệu Converse API, mỗi nhà cung cấp mô hình trên Bedrock yêu cầu một định dạng JSON body đầu vào hoàn toàn khác nhau khi gọi hàm invoke_model. Ví dụ: Anthropic yêu cầu khóa max_tokens và cấu hình prompt khác biệt so với Meta Llama hay Cohere.

Converse API (boto3: bedrock_client.converse()) ra đời để giải quyết triệt để sự phân mảnh này bằng cách cung cấp một giao thức API đồng nhất, chuẩn hóa hoàn toàn cấu trúc payload hội thoại tương thích với mô hình danh sách tin nhắn (messages list) tương tự như OpenAI API.

Cấu trúc tin nhắn chuẩn của Bedrock Converse

Mỗi tin nhắn trong danh sách gửi lên Bedrock phải tuân thủ schema:

{
  "role": "user" | "assistant",
  "content": [
    {
      "text": "Nội dung văn bản hội thoại..."
    }
  ]
}

Lưu ý: Khóa content bắt buộc phải là một danh sách (List) các block JSON. Đây là thiết kế mở rộng cho phép Bedrock xử lý Multimodal (hình ảnh, tài liệu đính kèm) sau này.

Xử lý System Prompt trong Converse API

Có hai cách để truyền chỉ thị hệ thống (System Prompt) nhằm định hình vai trò của Digital Twin:

  • Cách 1 (Tương thích đơn giản): Ghép trực tiếp System Prompt vào đầu tin nhắn của user với tiền tố "System: [nội dung prompt]".
  • Cách 2 (Khuyên dùng): Truyền trực tiếp qua tham số system của API: 
    bedrock_client.converse(
    modelId=BEDROCK_MODEL_ID,
    messages=messages,
    system=[{"text": prompt()}]
)

Cross-Region Inference Profile - Hồ sơ suy luận xuyên vùng

Tại môi trường production thực tế của AWS thế hệ mới (2026), việc gọi trực tiếp Model ID tĩnh của một vùng (ví dụ: amazon.nova-lite-v1:0 đặt tại us-east-1) không còn là phương án tối ưu do dễ bị chạm ngưỡng giới hạn băng thông (Rate limits / Quota limits) khi lượng request tăng đột biến.

Cơ chế tự động cân bằng tải xuyên vùng

Cross-Region Inference Profiles là cơ chế định tuyến thông minh của AWS Bedrock. Bằng cách sử dụng Model ID bắt đầu với tiền tố global. hoặc us. (ví dụ: global.amazon.nova-2-lite-v1:0), AWS sẽ tự động phân bổ request suy luận của bạn đến vùng địa lý có tài nguyên GPU rảnh rỗi gần nhất.

Cơ chế này mang lại hạn mức quota sử dụng mô hình cao hơn nhiều lần và giảm thiểu tối đa hiện tượng nghẽn mạng hoặc từ chối dịch vụ mà không cần lập trình viên tự viết code cân bằng tải.

Cấu hình Model ID thế hệ mới (2026)

Hãy lưu ý hai quy tắc cấu hình quan trọng sau để đảm bảo hệ thống hoạt động ổn định:

  • Khởi tạo boto3.client với region mặc định là us-east-1 (Bắc Virginia). Region của Bedrock client không bắt buộc phải trùng với vùng đặt Lambda function hay S3 của bạn.
  • Thiết lập biến môi trường BEDROCK_MODEL_ID trỏ tới hồ sơ suy luận xuyên vùng:
    • Toàn cầu: global.amazon.nova-2-lite-v1:0
    • Vùng Mỹ: us.amazon.nova-lite-v1:0
    • Vùng Âu: eu.amazon.nova-lite-v1:0
Section 6

Keyless Authentication - Xác thực không khóa

Trong mô hình OpenAI cũ, backend trên Lambda bắt buộc phải cấu hình biến môi trường OPENAI_API_KEY chứa chuỗi token bí mật. Việc lưu trữ khóa cứng này mang lại rủi ro bảo mật cực lớn nếu lập trình viên vô tình push file cấu hình lên kho chứa GitHub công khai.

Xác thực không khóa (Keyless Auth): Nhờ tích hợp sâu vào hệ sinh thái AWS, ứng dụng chạy trên Lambda gọi Bedrock mà không cần lưu trữ bất kỳ API key nào. Việc chứng thực được thực hiện hoàn toàn tự động thông qua IAM Execution Role được gán trực tiếp cho Lambda function.

sequenceDiagram participant Lambda as AWS Lambda Function participant IMDS as AWS Instance Metadata Service (IMDSv2) participant Boto3 as Boto3 SDK (Python) participant Bedrock as AWS Bedrock Service Lambda->>Boto3: Gọi bedrock_client.converse() (Không truyền API Key) Boto3->>IMDS: Gửi HTTP GET yêu cầu IAM Temporary Credentials Note over IMDS: Truy vấn địa chỉ IP nội bộ:
http://169.254.169.254/latest/meta-data/ IMDS-->>Boto3: Trả về access_key, secret_key & session_token (tạm thời) Boto3->>Boto3: Ký request API bằng thuật toán AWS Signature Version 4 (SigV4) Boto3->>Bedrock: Gửi HTTP Request kèm chữ ký SigV4 Note over Bedrock: Kiểm tra IAM Policy gắn trên Lambda Role Bedrock-->>Lambda: Phản hồi kết quả sinh chữ từ mô hình Nova

AWS Instance Metadata Service (IMDS): Boto3 client tự động nhận diện môi trường AWS và gửi truy vấn tới địa chỉ IP link-local đặc biệt http://169.254.169.254/latest/meta-data/iam/security-credentials/ để lấy token bảo mật tạm thời. Token này tự động xoay vòng (rotation) liên tục theo thời gian thực để ngăn chặn các cuộc tấn công đánh cắp phiên.

Monitoring & Budgets - Giám sát hệ thống & Kiểm soát tài chính

Vận hành một ứng dụng AI Agent trong sản xuất thực tế đòi hỏi kỹ sư phải nắm rõ trạng thái hoạt động của tài nguyên thông qua số liệu trực quan (Metrics) và kiểm soát chặt chẽ chi phí đám mây.

4 Chỉ số Vàng Lambda

  • Invocations: Số lượt gọi hàm Lambda xử lý tin nhắn chat của user.
  • Duration: Thời gian thực thi trung bình. Thời gian này phụ thuộc lớn vào tốc độ phản hồi của Bedrock.
  • Errors: Số lỗi runtime phát sinh (lỗi code, lỗi phân quyền).
  • Throttles: Số request bị từ chối do vượt quá giới hạn concurrency cho phép.

4 Chỉ số Hiệu năng Bedrock

  • InvocationLatency: Độ trễ thời gian phản hồi của mô hình nền tảng.
  • Invocations: Tổng số lượt gọi truy vấn mô hình AI.
  • InputTokenCount: Lượng token đầu vào gửi đi (ảnh hưởng trực tiếp bởi độ dài context window).
  • OutputTokenCount: Lượng token sinh ra từ câu trả lời của mô hình.

from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel
import os
from dotenv import load_dotenv
from typing import Optional, List, Dict
import json
import uuid
from datetime import datetime
import boto3
from botocore.exceptions import ClientError
from context import prompt

# Nap bien moi truong tu file .env
load_dotenv()

app = FastAPI()

# Cau hinh chia se tai nguyen nguon goc (CORS) cho Frontend
origins = os.getenv("CORS_ORIGINS", "http://localhost:3000").split(",")
app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=False,
    allow_methods=["GET", "POST", "OPTIONS"],
    allow_headers=["*"],
)

# Khoi tao Bedrock Client - Dung dịch vụ "bedrock-runtime"
# Dat default region ve us-east-1 de dam bao luon co san dong model Nova va co quota lon
bedrock_client = boto3.client(
    service_name="bedrock-runtime", 
    region_name=os.getenv("DEFAULT_AWS_REGION", "us-east-1")
)

# Lua chon Model ID - Mac dinh su dung Cross-Region Inference Profile cua Nova Lite v1:0
BEDROCK_MODEL_ID = os.getenv("BEDROCK_MODEL_ID", "global.amazon.nova-2-lite-v1:0")

# Cau hinh vung nho luu tru lich su chat
USE_S3 = os.getenv("USE_S3", "false").lower() == "true"
S3_BUCKET = os.getenv("S3_BUCKET", "")
MEMORY_DIR = os.getenv("MEMORY_DIR", "../memory")

if USE_S3:
    s3_client = boto3.client("s3")

# Dinh nghia cac Schema cho request/response bang Pydantic
class ChatRequest(BaseModel):
    message: str
    session_id: Optional[str] = None

class ChatResponse(BaseModel):
    response: str
    session_id: str

def load_conversation(session_id: str) -> List[Dict]:
    """Tai lich su chat tu S3 hoac local directory"""
    if USE_S3:
        try:
            response = s3_client.get_object(Bucket=S3_BUCKET, Key=f"{session_id}.json")
            return json.loads(response["Body"].read().decode("utf-8"))
        except ClientError as e:
            if e.response["Error"]["Code"] == "NoSuchKey":
                return []
            raise
    else:
        file_path = os.path.join(MEMORY_DIR, f"{session_id}.json")
        if os.path.exists(file_path):
            with open(file_path, "r") as f:
                return json.load(f)
        return []

def save_conversation(session_id: str, messages: List[Dict]):
    """Ghi lai lich su chat vao vung nho luu tru"""
    if USE_S3:
        s3_client.put_object(
            Bucket=S3_BUCKET,
            Key=f"{session_id}.json",
            Body=json.dumps(messages, indent=2),
            ContentType="application/json",
        )
    else:
        os.makedirs(MEMORY_DIR, exist_ok=True)
        file_path = os.path.join(MEMORY_DIR, f"{session_id}.json")
        with open(file_path, "w") as f:
            json.dump(messages, f, indent=2)

def call_bedrock(conversation: List[Dict], user_message: str) -> str:
    """Gui payload hoi thoai toi AWS Bedrock Converse API"""
    messages = []
    
    # Chen System Prompt dinh hinh Digital Twin duoi dang User Message dau tien
    messages.append({
        "role": "user", 
        "content": [{"text": f"System: {prompt()}"}]
    })
    
    # Nạp lịch sử (gioi han lay toi da 50 tin nhắn - 25 luot chat de tranh day context window)
    for msg in conversation[-50:]:
        messages.append({
            "role": msg["role"],
            "content": [{"text": msg["content"]}]
        })
    
    # Them tin nhan moi nhat cua nguoi dung
    messages.append({
        "role": "user",
        "content": [{"text": user_message}]
    })
    
    try:
        # Thuc hien goi Converse API dong nhat cua boto3 SDK
        response = bedrock_client.converse(
            modelId=BEDROCK_MODEL_ID,
            messages=messages,
            inferenceConfig={
                "maxTokens": 2000,
                "temperature": 0.7,
                "topP": 0.9
            }
        )
        
        # Trich xuat noi dung text tra ve tu dung cau truc phan cap JSON cua Bedrock
        return response["output"]["message"]["content"][0]["text"]
        
    except ClientError as e:
        error_code = e.response['Error']['Code']
        if error_code == 'ValidationException':
            # Gap loi neu gui 2 tin nhan lien tiep cung vai tro (vi du user - user) ma khong xen ke
            print(f"Bedrock validation error: {e}")
            raise HTTPException(status_code=400, detail="Invalid message format for Bedrock")
        elif error_code == 'AccessDeniedException':
            # Gap loi neu chua cap quyen IAM Policy hoac chua bat Model Access tren Console
            print(f"Bedrock access denied: {e}")
            raise HTTPException(status_code=403, detail="Access denied to Bedrock model")
        else:
            print(f"Bedrock error: {e}")
            raise HTTPException(status_code=500, detail=f"Bedrock error: {str(e)}")

@app.get("/health")
async def health_check():
    """Endpoint kiem tra suc khoe cho API Gateway / Route 53"""
    return {
        "status": "healthy", 
        "use_s3": USE_S3,
        "bedrock_model": BEDROCK_MODEL_ID
    }

@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest):
    try:
        session_id = request.session_id or str(uuid.uuid4())
        conversation = load_conversation(session_id)
        
        # Lay cau tra loi tu model Nova cua Bedrock
        assistant_response = call_bedrock(conversation, request.message)
        
        # Cap nhat lich su hoi thoai
        conversation.append(
            {"role": "user", "content": request.message, "timestamp": datetime.now().isoformat()}
        )
        conversation.append(
            {"role": "assistant", "content": assistant_response, "timestamp": datetime.now().isoformat()}
        )
        
        save_conversation(session_id, conversation)
        return ChatResponse(response=assistant_response, session_id=session_id)
    except HTTPException:
        raise
    except Exception as e:
        print(f"Error in chat endpoint: {str(e)}")
        raise HTTPException(status_code=500, detail=str(e))

# 3. Tao bucket moi tren S3 o khu vuc default region da dinh nghia
aws s3 mb s3://$deployBucket --region $env:DEFAULT_AWS_REGION

# 4. Upload file nén lambda-deployment.zip len S3 bucket tam thoi
aws s3 cp lambda-deployment.zip s3://$deployBucket/ --region $env:DEFAULT_AWS_REGION

# 5. Chi thi AWS Lambda keo ma nguon moi ve va giai nen cap nhat cho function twin-api
aws lambda update-function-code `
    --function-name twin-api `
    --s3-bucket $deployBucket `
    --s3-key lambda-deployment.zip `
    --region $env:DEFAULT_AWS_REGION

# 6. Don dep tai nguyen: Xoa tep nén va xoa bucket tam thoi de tranh phat sinh phi luu tru
aws s3 rm s3://$deployBucket/lambda-deployment.zip
aws s3 rb s3://$deployBucket

Options / Trade-offs - Bản đồ lựa chọn thiết kế

Đưa ra các đánh giá kỹ thuật và so sánh chi tiết giữa các phương án triển khai mô hình trí tuệ nhân tạo:

Phương án - Option Ưu điểm - Pros Nhược điểm - Cons Ngữ cảnh lựa chọn - When to choose
AWS Bedrock (Nova Lite/Micro) (Recommended) Request hoàn toàn chạy trong mạng nội bộ AWS (độ trễ cực thấp, bảo mật tuyệt đối), xác thực không khóa qua IAM, tích hợp CloudWatch giám sát chặt chẽ. Giới hạn trong danh sách các dòng mô hình mà AWS phân phối. Đòi hỏi thiết lập phân quyền IAM và kích hoạt Model Access phức tạp hơn OpenAI ban đầu. Bắt buộc cho các hệ thống ứng dụng AI chạy môi trường Production thực tế của doanh nghiệp trên đám mây AWS.
OpenAI API trực tiếp Khả năng lý luận của GPT-4o dẫn đầu thị trường, tài liệu hỗ trợ phong phú, tốc độ tích hợp ban đầu nhanh chóng (chỉ cần truyền API key). Dữ liệu phải truyền qua Internet công cộng tới máy chủ bên thứ ba (rủi ro bảo mật thông tin nhạy cảm), yêu cầu quản lý và bảo mật chuỗi API key, độ trễ mạng cao. Phù hợp cho giai đoạn thử nghiệm ý tưởng ban đầu (Proof of Concept - PoC) hoặc khi ứng dụng chưa chạy trên hạ tầng cloud AWS.
Direct Upload Lambda Code Đơn giản, nhanh gọn bằng 1 dòng lệnh CLI truyền cờ --zip-file. Dễ bị timeout và ngắt kết nối giữa chừng nếu file zip nặng hoặc đường truyền mạng chập chờn. Giới hạn dung lượng upload qua trình duyệt là 50MB. Chỉ dùng khi file nén deploy có kích thước siêu nhẹ (<5MB) và đường truyền mạng internet local cực kỳ ổn định.
S3 Deployment Bucket Pipeline (Recommended) Độ tin cậy cực cao, cho phép tiếp tục tải lên nếu bị gián đoạn (resume upload), an toàn cho các file nén dung lượng lớn. Tốn thêm công đoạn viết script tự động hóa tạo/xóa bucket tạm thời và chạy qua nhiều câu lệnh trung gian. Áp dụng khi gói nén zip chứa nhiều dependencies nặng (>10MB) hoặc tích hợp vào các đường ống CI/CD chuyên nghiệp.
Section 10

Pitfalls - Bẫy phát sinh lỗi & Cách khắc phục

Các trường hợp dễ gây lỗi (Failure Modes)

  • AccessDeniedException từ Bedrock: Backend Lambda trả về lỗi HTTP 403 hoặc 500 khi người dùng gửi tin nhắn chat, log ghi nhận lỗi phân quyền gọi mô hình.
  • ValidationException từ Bedrock: Hệ thống quăng lỗi cấu trúc dữ liệu không hợp lệ khi gửi mảng lịch sử hội thoại lên mô hình.
  • S3 BucketAlreadyExists Error: Kịch bản deploy bị crash ngay ở bước chạy lệnh aws s3 mb để khởi tạo deployment bucket tạm thời.
  • Bedrock ModelNotAvailable / Region Issues: Ứng dụng Lambda báo không tìm thấy mô hình hoặc báo lỗi vùng địa lý không tương thích.

Biện pháp ngăn ngừa & Xử lý (Fix / Prevention)

  • Kiểm tra 2 lớp phân quyền: Đảm bảo IAM Execution Role của Lambda có policy AmazonBedrockFullAccess VÀ trạng thái của mô hình Nova trên Bedrock Console (tab Model Access) đã hiển thị chữ Access granted màu xanh lá.
  • Xếp xen kẽ vai trò hội thoại: Đảm bảo mảng tin nhắn gửi lên Bedrock tuân thủ quy tắc xen kẽ tuần tự giữa user và assistant. Tuyệt đối không gửi 2 tin nhắn của cùng một vai trò liên tiếp nhau.
  • Cải tiến sinh tên bucket ngẫu nhiên: Thay đổi định dạng timestamp chi tiết hơn (đến tận giây yyyyMMddHHmmss) hoặc nối thêm chuỗi hash ngẫu nhiên vào tên bucket để triệt tiêu tỷ lệ trùng lặp.
  • Cố định Region cho Bedrock Client: Chỉ định rõ tham số region_name="us-east-1" khi khởi tạo client bedrock-runtime vì đây là vùng luôn được ưu tiên cập nhật model mới nhất và có quota rộng rãi nhất.
Section 11

Flashcards & Quick Quiz - Ôn tập tương tác

Q: Tại sao gọi Bedrock qua Lambda trong mạng nội bộ AWS lại không cần lưu cấu hình API Key?

A: Vì hệ thống sử dụng cơ chế Keyless Authentication. Khi Lambda khởi chạy, Boto3 SDK tự động truy vấn địa chỉ IP nội bộ của AWS Instance Metadata Service (IMDS) để lấy temporary credentials (access key, secret key, session token) được cấp theo IAM Execution Role gắn trên Lambda, sau đó tự động ký request bằng SigV4 để xác thực an toàn.

Q: Sự khác biệt bản chất giữa dịch vụ bedrock và bedrock-runtime trong Boto3 SDK là gì?

A: Client bedrock được sử dụng cho các tác vụ quản trị mặt bằng điều khiển (Control Plane) như: liệt kê mô hình có sẵn, yêu cầu quyền truy cập (model access) hoặc cấu hình fine-tuning. Client bedrock-runtime được sử dụng cho luồng dữ liệu thực thi (Data Plane) để chạy suy luận mô hình (sinh văn bản qua Converse API, sinh hình ảnh, nhúng vector embeddings).

Q: Cross-Region Inference Profiles giải quyết bài toán gì ở môi trường thực tế?

A: Nó giải quyết bài toán giới hạn băng thông (Rate limit / Quota exhaust) và tăng độ ổn định của hệ thống. Thay vì gửi request tới một model cố định ở 1 vùng (dễ bị nghẽn), AWS tự động phân tuyến request thông minh xuyên vùng tới vị trí có tài nguyên GPU rảnh gần nhất, giúp nâng cao đáng kể hạn mức sử dụng mô hình.

Quick Memory - Ghi nhớ nhanh cuối bài

Must remember - Điều bắt buộc phải nhớ

  1. Bắt buộc dùng client bedrock-runtime để thực hiện suy luận mô hình sinh văn bản.
  2. Họ mô hình Amazon Nova gồm: Nova Micro (nhẹ, rẻ), Nova Lite (cân bằng, đa phương tiện), Nova Pro (mạnh mẽ, lý luận phức tạp).
  3. Cross-region Inference Profiles sử dụng tiền tố global.* hoặc us.* để tăng quota và tự cân bằng tải.
  4. Ký tự ngắt dòng lệnh dài trong Windows PowerShell là dấu backtick (`` ` ``), còn trong Linux Bash Shell là dấu backslash (\).
  5. Luôn luôn dọn dẹp (xóa file nén và xóa bucket tạm thời trên S3) sau khi cập nhật Lambda thành công để tránh phát sinh chi phí lưu trữ rác.
