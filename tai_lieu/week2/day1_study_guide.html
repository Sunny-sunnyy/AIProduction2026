Section 4

Core Theory - Lý thuyết cốt lõi

Docx LLM Slide PDF

Key concepts - Khái niệm chính

  • Root user - Người dùng gốc: Tài khoản tối cao được tạo bằng email đăng ký AWS ban đầu, sở hữu quyền lực cao nhất và không thể bị giới hạn bởi bất kỳ chính sách nào. Chỉ dùng khi Billing, quản lý tài chính, dọn dẹp tài nguyên hoặc đóng tài khoản.
  • Infrastructure as Code (IaC) - Cơ sở hạ tầng dưới dạng mã: Phương pháp thiết lập và quản lý tài nguyên mạng, máy chủ bằng các file cấu hình declarative (khai báo) được kiểm soát phiên bản bằng Git thay vì thao tác tay trên web GUI.
  • Terraform: Công cụ IaC mã nguồn mở của HashiCorp, sử dụng ngôn ngữ cấu hình HCL (HashiCorp Configuration Language) để khai báo hạ tầng độc lập đám mây (cloud-agnostic).
  • Cloud Deployment Archetypes - Các mô hình triển khai đám mây: Phân loại hạ tầng gồm: IaaS (máy chủ EC2 ảo), PaaS (Elastic Beanstalk, Vercel tự quản lý runtime), CaaS (App Runner chạy container), Container Orchestration (ECS/EKS quản lý nhiều container), và Serverless FaaS (chạy code theo hàm Lambda).
  • FaaS (Function as a Service) - Hàm dưới dạng dịch vụ: Mô hình serverless chạy code event-driven (hướng sự kiện). Tự động co giãn theo tải thực tế và hỗ trợ scale-to-zero (co giãn về không khi rảnh).

Concept relationships - Quan hệ giữa khái niệm

Mối quan hệ giữa các mô hình triển khai đám mây nằm ở sự đánh đổi giữa control (quyền kiểm soát) và convenience (tính tiện lợi). Trong khi IaaS (EC2) cho phép cấu hình tối đa đến cấp hệ điều hành nhưng tốn công bảo trì lớn, thì Serverless FaaS (Lambda) giải phóng hoàn toàn sức lao động quản trị server nhưng bị giới hạn về thời gian thực thi (tối đa 15 phút) và gặp hiện tượng cold start. CDN (CloudFront) bổ trợ trực tiếp cho S3 bằng cách cache nội dung tĩnh tại các Edge Location để giảm tải trực tiếp cho S3 Origin.

Deep explanation - Giải thích sâu về scale-to-zero và cold start

Scale-to-zero (Co giãn về không): Trong kiến trúc Serverless (như AWS Lambda), khi ứng dụng không nhận được request nào, nhà cung cấp cloud sẽ giải phóng toàn bộ container đang chạy hàm đó. Người dùng sẽ không phải trả bất kỳ chi phí nào cho phần cứng nhàn rỗi. Đây là ưu điểm tài chính cực lớn cho startup.

Cold Start (Khởi động lạnh): Khi có một request mới gửi tới sau một thời gian dài không hoạt động, AWS sẽ phải khởi tạo một container siêu nhỏ mới, nạp runtime (Python, Node.js) và tải mã nguồn của hàm lên bộ nhớ. Quá trình này gây ra một độ trễ phản hồi (cold start latency) kéo dài từ vài trăm mili-giây đến vài giây cho người dùng đầu tiên. Các request tiếp theo sẽ được chuyển tới container ấm (warm container) có sẵn nên phản hồi cực kỳ nhanh.

Knowledge extension - Cơ chế Single Domain Routing bằng CloudFront để triệt tiêu lỗi CORS

Kiến thức mở rộng CORS (Cross-Origin Resource Sharing) block error xảy ra khi trình duyệt chặn request JavaScript từ một origin này (ví dụ frontend: http://localhost:3000) gửi tới origin khác (ví dụ backend API: http://localhost:8000).

Để triệt tiêu CORS tận gốc trên Production mà không cần mở header CORS wildcard (kém bảo mật), ta cấu hình Amazon CloudFront làm proxy trung tâm duy nhất dưới một domain chung (ví dụ: https://my-twin.cloudfront.net):

  • Cấu hình route mặc định /* trỏ về origin S3 Frontend Bucket để tải giao diện.
  • Cấu hình route phụ /api/* trỏ về origin Amazon API Gateway.

Do cả frontend và API cùng nằm dưới một domain duy nhất của CloudFront, trình duyệt sẽ coi đây là giao tiếp Same-Origin và hoàn toàn không kích hoạt cơ chế chặn CORS của trình duyệt.

3. Quy trình chi tiết đầu-cuối bằng văn bản (End-to-End Workflow Walkthrough)

Quy trình hoạt động xử lý hội thoại có lưu trữ bộ nhớ (Conversational Memory) cục bộ ở Day 1 được thực hiện qua các giai đoạn sau:

Giai đoạn 1: Frontend Client gửi Request

  1. Người dùng nhập tin nhắn vào ô input trên trình duyệt và bấm gửi (hoặc nhấn phím Enter).
  2. Hàm sendMessage() trong component [twin.tsx](file:///G:/AIProduction_t6_2026/production/week2/day1.md#L296) được kích hoạt.
  3. Tin nhắn mới của người dùng được đóng gói thành một đối tượng Message và cập nhật trực tiếp vào React state messages thông qua setMessages để hiển thị lập tức lên UI.
  4. Frontend gọi hàm fetch('http://localhost:8000/chat', ...) gửi HTTP POST request đến Backend API dưới định dạng JSON body chứa: message (nội dung tin nhắn) và session_id (lấy từ React state sessionId, nếu là tin nhắn đầu tiên thì giá trị này là rỗng).

Giai đoạn 2: Backend tiếp nhận và xử lý bộ nhớ

  1. FastAPI Server tiếp nhận request tại route [app.post("/chat")](file:///G:/AIProduction_t6_2026/production/week2/day1.md#L696) với schema ChatRequest.
  2. Hệ thống kiểm tra session_id từ request. Nếu trống (lượt chat đầu tiên), backend sử dụng thư viện uuid để tạo một chuỗi định danh ngẫu nhiên phiên bản 4 thông qua lệnh str(uuid.uuid4()).
  3. Backend gọi hàm load_conversation(session_id) để tìm kiếm tệp tin lịch sử:
    • Hàm này sử dụng thư viện pathlib.Path định vị tệp ../memory/{session_id}.json.
    • Nếu tệp tồn tại, nó mở tệp với mã hóa UTF-8 bằng open(..., encoding="utf-8") và sử dụng json.load(f) để giải tuần tự hóa (deserialize) nội dung tệp thành một danh sách (list) các tin nhắn cũ.
    • Nếu tệp chưa tồn tại, nó trả về một mảng rỗng [].

Giai đoạn 3: Gọi LLM suy luận với đầy đủ bối cảnh

  1. Backend khởi tạo danh sách tin nhắn gửi đi messages bắt đầu bằng System Prompt mô tả tính cách (Personality) được tải trước từ file me.txt.
  2. Backend lặp qua mảng lịch sử trò chuyện conversation vừa tải từ file JSON và đẩy toàn bộ các tin nhắn cũ vào danh sách messages.
  3. Backend đẩy tiếp tin nhắn hiện tại của người dùng (request.message) vào cuối danh sách messages. Điều này giúp LLM nhận được đầy đủ ngữ cảnh của cả cuộc hội thoại.
  4. Backend dùng SDK OpenAI gọi API suy luận: client.chat.completions.create(model="gpt-4o-mini", messages=messages).
  5. OpenAI trả về đối tượng phản hồi chứa nội dung câu trả lời của AI.

Giai đoạn 4: Cập nhật lịch sử và lưu trữ

  1. Backend thêm tin nhắn hiện tại của user và phản hồi của trợ lý AI vào danh sách conversation.
  2. Backend gọi hàm save_conversation(session_id, conversation):
    • Hàm mở tệp ../memory/{session_id}.json ở chế độ ghi đè ("w") với mã hóa UTF-8.
    • Nó gọi lệnh json.dump(messages, f, indent=2, ensure_ascii=False) để tuần tự hóa (serialize) dữ liệu xuống đĩa cứng. Tham số indent=2 thụt lề dòng giúp dễ đọc, và ensure_ascii=False đảm bảo tiếng Việt unicode được hiển thị nguyên bản thay vì bị mã hóa thành ký tự escape (như \u00fa).
  3. FastAPI trả về HTTP response JSON chứa câu trả lời của AI và session_id (mới sinh hoặc kế thừa) về cho client.

Giai đoạn 5: Frontend nhận kết quả và render UI

  1. Client Browser tiếp nhận response. Nếu state sessionId của React đang rỗng, nó sẽ cập nhật giá trị sessionId nhận được từ backend bằng setSessionId(data.session_id) để sử dụng cho các lượt chat kế tiếp.
  2. Frontend đóng gói câu trả lời của AI thành đối tượng Message và cập nhật vào React state messages để hiển thị lên màn hình.
Section 6

Techniques - Kỹ thuật sử dụng

Cloud Resource Cleanup

Purpose: Kiểm soát tối ưu hóa chi phí (cost optimization) tài nguyên đám mây, tránh việc phát sinh chi phí duy trì tài nguyên nhàn rỗi (idle cost) khi đã kết thúc thực hành.

When to use: Ngay sau khi hoàn thành các buổi học, bài lab thử nghiệm, kết thúc giai đoạn POC (Proof of Concept) hoặc dọn dẹp các tài nguyên cũ bị kẹt.

Trade-off: Xóa tài nguyên đồng nghĩa với việc mất môi trường chạy trực tiếp của ứng dụng, cần thiết lập IaC (Terraform) để có thể dựng lại nhanh chóng khi cần.

Edge Caching

Purpose: Giảm thiểu latency (độ trễ mạng) cho người dùng cuối bằng cách lưu trữ tạm thời nội dung tĩnh (static assets) tại các trạm biên CDN (Edge Location) gần người dùng nhất.

When to use: Dùng cho tất cả các static assets của ứng dụng như tệp frontend HTML/JS/CSS, hình ảnh, video và font chữ.

Trade-off: Phải kiểm soát cơ chế Invalidation (làm mới cache) khi deploy phiên bản mới để tránh người dùng tải phải tệp tin cũ bị lỗi đồng bộ.

JSON Session Serialization

Purpose: Thiết lập tính năng lưu giữ trạng thái (conversational memory) cho chatbot cục bộ một cách tối giản, gọn nhẹ mà không cần cài đặt và cấu hình cơ sở dữ liệu (database) phức tạp.

When to use: Phù hợp cho môi trường phát triển thử nghiệm cục bộ (local development) hoặc ứng dụng đơn người dùng (single-user application).

Trade-off: Tốc độ đọc/ghi file trên đĩa cứng sẽ chậm dần khi kích thước cuộc trò chuyện dài ra, dễ bị lỗi xung đột ghi file đồng thời khi có nhiều user nhắn tin cùng lúc.

backend/server.py (Stateful Version có Memory)

Phần mã nguồn backend hoàn chỉnh của máy chủ FastAPI, chịu trách nhiệm tiếp nhận request chat, tự động quản lý session và ghi nhớ lịch sử hội thoại của từng phiên chat độc lập ra các file JSON trên đĩa.

Purpose: API Server & File Memory Manager Key functions: load_conversation, save_conversation, chat, list_sessions 
from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel
from openai import OpenAI
import os
from dotenv import load_dotenv
from typing import Optional, List, Dict
import json
import uuid
from pathlib import Path

# Load bien moi truong tu file .env, override=True de ep tai lai bien moi nhat
load_dotenv(override=True)

app = FastAPI()

# Doc cau hinh CORS tu file .env, mac dinh cho phep frontend localhost:3000 truy cap
origins = os.getenv("CORS_ORIGINS", "http://localhost:3000").split(",")
app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# Khoi tao client OpenAI tu bien moi truong OPENAI_API_KEY
client = OpenAI()

# Khoi tao va tu dong tao thu muc luu tru memory o ngoai backend neu chua co
MEMORY_DIR = Path("../memory")
MEMORY_DIR.mkdir(exist_ok=True)

# Doc file persona de lam System Prompt cho AI chatbot
def load_personality():
    with open("me.txt", "r", encoding="utf-8") as f:
        return f.read().strip()

PERSONALITY = load_personality()

# Khai bao schema request va response cho API chat
class ChatRequest(BaseModel):
    message: str
    session_id: Optional[str] = None

class ChatResponse(BaseModel):
    response: str
    session_id: str

# Ham doc lich su chat tu file JSON cua session tuong ung
def load_conversation(session_id: str) -> List[Dict]:
    file_path = MEMORY_DIR / f"{session_id}.json"
    if file_path.exists():
        with open(file_path, "r", encoding="utf-8") as f:
            return json.load(f)
    return []

# Ham ghi de lich su chat moi nhat xuong file JSON dung dinh dang unicode tieng Viet
def save_conversation(session_id: str, messages: List[Dict]):
    file_path = MEMORY_DIR / f"{session_id}.json"
    with open(file_path, "w", encoding="utf-8") as f:
        # ensure_ascii=False giup hien thi dung ky tu tieng Viet co dau, indent=2 thut dong de doc
        json.dump(messages, f, indent=2, ensure_ascii=False)

@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest):
    try:
        # Sinh session ID ngau nhien bang UUID neu day la luot chat dau tien
        session_id = request.session_id or str(uuid.uuid4())
        
        # Load lai toan bo cuoc tro chuyen cu tu dia cung
        conversation = load_conversation(session_id)
        
        # Khoi tao mang tin nhan gui toi OpenAI voi System Personality prompt
        messages = [{"role": "system", "content": PERSONALITY}]
        
        # Nạp toan bo cac tin nhan lich su cu vao mang
        for msg in conversation:
            messages.append(msg)
        
        # Append tin nhan moi cua user vao cuoi mang tin nhan gui di
        messages.append({"role": "user", "content": request.message})
        
        # Goi OpenAI API hoan thanh logic hoi thoai
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages
        )
        
        assistant_response = response.choices[0].message.content
        
        # Cap nhat tin nhan moi cua user va bot vao danh sach lich su chat
        conversation.append({"role": "user", "content": request.message})
        conversation.append({"role": "assistant", "content": assistant_response})
        
        # Ghi lai lich su moi nhat vao file JSON session de tiep tuc nho o lan sau
        save_conversation(session_id, conversation)
        
        return ChatResponse(
            response=assistant_response,
            session_id=session_id
        )

    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

@app.get("/sessions")
async def list_sessions():
    # Quet va liet ke thong tin tat ca cac file session dang co trong thu muc memory
    sessions = []
    for file_path in MEMORY_DIR.glob("*.json"):
        session_id = file_path.stem # Lay ten file khong kem duoi extension
        with open(file_path, "r", encoding="utf-8") as f:
            conversation = json.load(f)
            sessions.append({
                "session_id": session_id,
                "message_count": len(conversation),
                "last_message": conversation[-1]["content"] if conversation else None
            })
    return {"sessions": sessions}

Section 8

Options / Trade-offs - Bản đồ lựa chọn

Lựa chọn - Option Phù hợp nhất - Best for Đánh đổi - Trade-off Rủi ro - Risk
Terraform (IaC) Dự án startup, doanh nghiệp có hạ tầng đa đám mây (multi-cloud), cần tính tự động hóa và ổn định cao. Phải học ngôn ngữ cấu hình mới (HCL) và cấu hình tệp tin state ban đầu khá phức tạp. Lỗi rách nát hạ tầng nếu tự ý chỉnh sửa thủ công tệp tin Terraform State file (`.tfstate`).
AWS CDK (IaC) Đội ngũ kỹ sư cam kết sử dụng 100% hạ tầng AWS và muốn lập trình hạ tầng bằng Python/TypeScript thuần. Bị khóa chặt vào hệ sinh thái của AWS (vendor lock-in), không thể chuyển đổi sang GCP hay Azure. Quá trình biên dịch CloudFormation gián tiếp tốn nhiều thời gian và khó debug cấu hình lỗi.
API Gateway + Lambda (Recommended) Hệ thống AI chatbot thực tế phục vụ khách hàng, cần kiểm soát rate-limiting, CORS, và bảo mật. Cấu hình phức tạp qua nhiều bước và phát sinh thêm chi phí nhỏ theo lưu lượng request qua Gateway. Độ trễ tăng nhẹ do phải đi qua proxy trung gian của API Gateway.
AWS Lambda Function URL Sản phẩm thử nghiệm nhỏ, webhook nội bộ hoặc các buổi lab thực hành nhanh để giảm chi phí. Thiếu hoàn toàn các tính năng bảo mật nâng cao và các công cụ rate-limiting, CORS handling. Dễ bị tấn công DDoS làm cạn kiệt tài nguyên hoặc tăng vọt chi phí gọi LLM API.
File-based Memory (Recommended local) Chatbot cục bộ, demo thử nghiệm hoặc chạy đơn người dùng giúp tiết kiệm thời gian. Tốc độ đọc/ghi file chậm dần khi lịch sử chat dài ra, khó mở rộng quy mô. Xung đột ghi đè đồng thời (concurrency lock) làm malformed định dạng JSON của file.
Database Memory (DynamoDB) (Recommended prod) Chatbot production phục vụ hàng nghìn người dùng đồng thời, cần tính nhất quán cao. Cần cấu hình kết nối database, thiết kế khóa phân vùng (partition key) và tốn thêm chi phí duy trì. Độ trễ tăng nhẹ nếu truy vấn database không được tối ưu hóa chỉ mục.
Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ lỗi

  • Bill Shock (Hóa đơn tăng vọt): Học viên quên không tắt các tài nguyên chạy ngầm của Week 1 (App Runner, ECR) dẫn đến bị AWS trừ tiền tài khoản lớn hàng tháng.
  • CORS Block Error: Frontend Next.js ở cổng 3000 không thể gửi tin nhắn chat đến FastAPI ở cổng 8000 do trình duyệt từ chối request chéo origin.
  • Concurrency File Lock: Nhiều request gửi lên backend cùng một lúc cho một session khiến các tiến trình đọc/ghi đè đè lên file JSON, gây lỗi malformed JSON syntax.
  • Cold Start Latency: Người dùng đầu tiên truy cập chatbot sau một khoảng thời gian dài gặp hiện tượng loading rất lâu (vài giây) trước khi nhận được phản hồi.

Fix / prevention - Cách tránh

  • Thiết lập cảnh báo Billing: Đăng nhập Root user, truy cập CloudWatch tạo Billing Alarm để gửi email cảnh báo tự động khi hóa đơn vượt ngưỡng nhỏ (ví dụ $5). Luôn dọn dẹp hạ tầng lab ngay sau khi học xong.
  • Cấu hình CORSMiddleware: Khai báo cụ thể origin http://localhost:3000 vào thuộc tính allow_origins của FastAPI. Đảm bảo ghi đủ cả giao thức http://.
  • Consistent DB Write: Khi chuyển giao lên cloud production, sử dụng database DynamoDB thay cho File-based Storage để tận dụng cơ chế khóa ghi giao dịch nhất quán.
  • Provisioned Concurrency: Trả thêm phí giữ ấm container Lambda, hoặc setup cron job tự động ping định kỳ để giữ ấm hàm Lambda không bị giải phóng.
Section 10

Flashcards / Q&A - Ôn tập tương tác

Q: Tại sao lưu trữ Docker image trên ECR vẫn phát sinh chi phí hàng tháng mặc dù service App Runner đã bị xóa?

A: Vì ECR tính phí lưu trữ dựa trên dung lượng đĩa cloud thực tế (Storage Cost) của Docker image (thường nặng từ vài trăm MB đến hàng GB). Xóa App Runner chỉ dừng runtime tính toán (CPU/RAM) chứ không giải phóng dung lượng đĩa ECR. Bạn bắt buộc phải xóa repository chứa image trên ECR để dừng phát sinh chi phí lưu trữ.

Q: Sự khác biệt cốt lõi về tính chất đám mây của Terraform và AWS CDK là gì?

A: Terraform là công cụ độc lập đám mây (cloud-agnostic), sử dụng cú pháp HCL để quản lý tài nguyên của nhiều cloud khác nhau (AWS, GCP, Azure) trên cùng một ngôn ngữ. AWS CDK là công cụ chính chủ của AWS, cho phép viết hạ tầng bằng ngôn ngữ lập trình phổ biến nhưng bị khóa chặt (vendor lock-in) chỉ dùng riêng cho dịch vụ AWS.

Q: Lợi ích chính của việc đặt API Gateway đứng trước AWS Lambda là gì?

A: API Gateway đóng vai trò như một proxy bảo mật trung tâm giúp định tuyến API, xử lý CORS, chứng thực API Key, kiểm soát rate-limiting (chống spam request) và theo dõi giám sát lưu lượng truy cập trước khi chuyển request kích hoạt Lambda.

Quick Memory - Ôn nhanh cuối bài

Must remember - Phải nhớ

  1. Luôn dọn dẹp ECR và App Runner sau bài thực hành để tối ưu hóa cost (chi phí).
  2. Root user là tài khoản tối cao dùng để quản lý Billing và cleanup triệt để tài nguyên nhàn rỗi.
  3. Terraform sử dụng cú pháp khai báo HCL độc lập đám mây (cloud-agnostic) để dựng cơ sở hạ tầng IaC.
  4. FaaS (Lambda) chỉ tính chi phí theo thời gian CPU chạy thực tế (tính bằng mili-giây) và số request, hỗ trợ scale-to-zero nhưng gặp hiện tượng cold start.
  5. Lưu trữ JSON theo Session ID (UUID) là kỹ thuật lưu bộ nhớ (conversational memory) đơn giản nhất cho phát triển cục bộ.
