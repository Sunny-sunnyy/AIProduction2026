Section 1

IaC & Terraform Philosophy - Triết lý và So sánh

Trong phát triển phần mềm hiện đại, việc cấu hình hạ tầng bằng cách click tay trên giao diện điều khiển (console) của Cloud Provider mang lại nhiều rủi ro lớn: không thể theo vết thay đổi, dễ sai sót do con người, không thể nhân bản môi trường đồng nhất và khó tích hợp tự động hóa.

Infrastructure as Code (IaC)

IaC là phương pháp quản lý và thiết lập tài nguyên đám mây (mạng, máy chủ ảo, cơ sở dữ liệu, bảo mật) thông qua các tệp cấu hình dạng văn bản được định nghĩa rõ ràng.

3 Lợi ích cốt lõi của IaC:

  • Kiểm soát phiên bản (Version Controlled): Lưu trữ code hạ tầng trên Git để theo dõi lịch sử chỉnh sửa dễ dàng.
  • Đánh giá ngang hàng (Reviewed): Kiểm duyệt các thay đổi hạ tầng thông qua Pull Requests (PRs).
  • Lặp lại & Tự động hóa (Automated/Repeatable): Tạo mới các môi trường phát triển (dev, test, prod) giống hệt nhau bằng 1 câu lệnh duy nhất.

So sánh: Terraform vs AWS CDK / CloudFormation

AWS CloudFormation và AWS CDK (Cloud Development Kit) là các công cụ chính chủ của AWS. Tuy nhiên, chúng bị khóa chặt vào hệ sinh thái AWS (vendor lock-in), không thể sử dụng cho Google Cloud hay Microsoft Azure.

Terraform (by HashiCorp) được chọn làm công cụ giảng dạy chính vì tính chất độc lập đám mây (cloud-agnostic). Một kỹ sư DevOps chỉ cần học cú pháp HCL của Terraform là có thể quản lý hạ tầng đa nền tảng (AWS, GCP, Azure, Kubernetes, Cloudflare, v.v.). Đây cũng là kỹ năng số 1 được săn đón trong các yêu cầu tuyển dụng kỹ sư Platform/DevOps.

Section 2

Clean Slate - Dọn dẹp tài nguyên thủ công

Trước khi chuyển giao toàn bộ quyền quản lý hạ tầng cho Terraform, chúng ta bắt buộc phải dọn dẹp sạch sẽ (Clean Slate) các tài nguyên đã tạo thủ công ở Day 2 & 3 để tránh xung đột định danh (như trùng tên S3 bucket toàn cầu) và tránh phát sinh chi phí thừa.

Bẫy Empty S3 Bucket

AWS chặn việc xóa trực tiếp một S3 bucket khi bên trong vẫn còn chứa file dữ liệu để tránh mất mát thông tin ngoài ý muốn.

Cách xử lý: Bắt buộc nhấp vào nút Empty trên console trước, gõ chuỗi "permanently delete" để xóa sạch files, sau đó mới có thể thực hiện lệnh Delete bucket (gõ tên bucket để xác nhận).

Bẫy Vô hiệu hóa CloudFront

Các phân phối CloudFront (distributions) được đồng bộ toàn cầu qua hàng trăm máy chủ Edge location của AWS. Do đó, nút Delete sẽ bị xám (không click được) nếu distribution đang ở trạng thái kích hoạt (Enabled).

Cách xử lý: Phải click nút Disable trước, kiên nhẫn chờ từ 5-10 phút cho đến khi cột Status hiển thị trạng thái Deployed và nút Enabled chuyển sang Disabled, lúc đó nút Delete mới sáng lên để thực hiện xóa vĩnh viễn.

Section 3

Core Terraform Concepts - Khái niệm cốt lõi

Hiểu rõ 6 khái niệm nền tảng trong cách thức hoạt động của Terraform:

1. Providers (Nhà cung cấp)

Các plugins làm nhiệm vụ kết nối và dịch lệnh từ Terraform sang API của nhà cung cấp đám mây cụ thể (như AWS, GCP, Azure, GitHub).

2. Variables (Biến đầu vào)

Tham số hóa cấu hình hạ tầng để code linh hoạt hơn, ví dụ: truyền tên môi trường (dev, prod) hoặc cấu hình giới hạn API throttling.

3. Resources (Tài nguyên)

Các viên gạch xây dựng cơ bản đại diện cho thực thể vật lý trên cloud (ví dụ: aws_s3_bucket, aws_lambda_function).

4. State File (Trạng thái)

Tệp tin terraform.tfstate là nguồn thông tin tin cậy duy nhất (source of truth) lưu trữ ánh xạ giữa code cấu hình logic và hạ tầng thực tế được tạo ra.

5. Outputs (Đầu ra)

Khai báo xuất thông tin hữu ích sau khi apply thành công (như CloudFront URL hoặc API Gateway Endpoint) phục vụ cho các script tự động hóa khác.

6. Workspaces (Môi trường cô lập)

Cho phép chạy cùng một bộ code cấu hình Terraform nhưng tạo ra nhiều tệp state độc lập (cô lập dev, test, prod).

Cực kỳ quan trọng: File State chứa các thông tin nhạy cảm của hệ thống và mã định danh tài nguyên thực tế. Tuyệt đối KHÔNG BAO GIỜ đưa file terraform.tfstate hoặc thư mục .terraform/ lên GitHub. Hãy cấu hình chúng trong file .gitignore.

Section 4

Workspace & Environment Isolation - Cô lập môi trường

Quy trình phát triển chuyên nghiệp yêu cầu phân tách hạ tầng thành nhiều môi trường độc lập. Terraform thực hiện việc này thông qua **Workspaces**. Mỗi workspace quản lý một file state riêng đặt trong thư mục terraform.tfstate.d/[workspace-name]/.

Môi trường Development (dev)

  • Model AI: Amazon Nova Micro (tối ưu hóa tốc độ và chi phí siêu rẻ).
  • API Throttling: Burst 10 / Rate 5 (thấp, đủ test).
  • Custom Domain: Không dùng (sử dụng domain mặc định của CloudFront/API Gateway).
  • Tên tài nguyên: Bắt đầu bằng twin-dev-.

Môi trường Testing (test)

  • Model AI: Amazon Nova Micro (tiết kiệm ngân sách chạy test tự động).
  • API Throttling: Burst 10 / Rate 5 (tiêu chuẩn).
  • Custom Domain: Không dùng.
  • Tên tài nguyên: Bắt đầu bằng twin-test-. Cô lập hoàn toàn với dev.

Môi trường Production (prod)

  • Model AI: Amazon Nova Lite (khả năng suy luận tốt hơn, chất lượng trò chuyện cao hơn).
  • API Throttling: Burst 20 / Rate 10 (nâng cao để phục vụ nhiều user).
  • Custom Domain: Có tích hợp Route 53 và ACM SSL Certificate.
  • Tên tài nguyên: Bắt đầu bằng twin-prod-.
Section 5

Next.js Dynamic Sync Workflow - Luồng đồng bộ động

Một thách thức lớn khi tự động hóa ứng dụng fullstack là: **Làm sao frontend static web chạy trên trình duyệt biết được URL API của backend Lambda?**

Do API Gateway tạo ra một Endpoint ngẫu nhiên (ví dụ: https://ab12cd34ef.execute-api.us-east-1.amazonaws.com) sau khi Terraform khởi tạo, URL này không cố định. Mặt khác, Next.js build tĩnh ra HTML/JS chạy client-side, tức là URL API bắt buộc phải được biên dịch thẳng vào file JS tĩnh **ngay tại thời điểm build**.

sequenceDiagram participant Script as Bash / PowerShell Script participant TF as Terraform Engine participant FE as Frontend Build (Next.js) participant S3 as S3 Frontend Bucket Script->>TF: Chạy terraform apply (tạo hạ tầng backend) TF-->>Script: Trả về outputs: api_gateway_url & s3_frontend_bucket Script->>Script: Trích xuất api_gateway_url từ output Script->>FE: Ghi đè vào file frontend/.env.production Note over FE: NEXT_PUBLIC_API_URL = [api_gateway_url] Script->>FE: Thực thi npm run build (Biên dịch tĩnh assets) Note over FE: Biên dịch mã tĩnh chứa URL API thật vào out/ folder Script->>S3: Đồng bộ (aws s3 sync) thư mục out/ lên S3 Bucket

Tiền tố NEXT_PUBLIC_

Next.js bảo vệ các biến cấu hình máy chủ nhạy cảm (như API keys, db password) bằng cách chặn không cho JavaScript phía client truy cập.

Chỉ những biến môi trường có tiền tố NEXT_PUBLIC_ (ví dụ: NEXT_PUBLIC_API_URL) mới được Next.js cho phép nhúng vào bản build client-side chạy trên trình duyệt của người dùng.

Kiến trúc Dependency chéo

Terraform tự động phân tích đồ thị phụ thuộc (Directed Acyclic Graph - DAG) để xác định thứ tự tạo tài nguyên.

Trong file HCL, ta truyền biến môi trường CORS cho Lambda trỏ tới CloudFront Domain Name. Để đảm bảo CloudFront đã có Domain Name thực tế nạp vào Lambda, ta sử dụng thuộc tính depends_on = [aws_cloudfront_distribution.main] trong định nghĩa Lambda.

Section 6

Custom Domain Configuration (Optional) - Cấu hình tên miền chuyên nghiệp

Để đưa Digital Twin lên một tên miền tùy chỉnh chuyên nghiệp (ví dụ: yourdomain.com), chúng ta tích hợp thêm Route 53 và ACM (AWS Certificate Manager) vào mã nguồn Terraform:

1. Đăng ký Domain

Đăng ký tên miền thông qua Route 53 Registered Domains (Root User) hoặc cấu hình Hosted Zone trỏ NS từ nhà đăng ký bên thứ ba.

2. Chứng chỉ SSL (ACM)

ACM Certificate bắt buộc phải được tạo và xác thực ở vùng us-east-1 (Bắc Virginia) thì CloudFront CDN mới có thể nhận diện và ánh xạ HTTPS thành công.

3. Route 53 Alias Record

Tạo các bản ghi DNS Alias (A & AAAA cho IPv6) trỏ trực tiếp Apex Domain (@) và subdomain (www) về địa chỉ phân phối của CloudFront.

Section 7

Code Walkthrough - Phân tích cấu hình Terraform HCL

Dưới đây là mã nguồn các tệp tin Terraform chính. Hãy phân tích cấu trúc cú pháp HCL để hiểu rõ cách thiết lập hạ tầng.

1. Khai báo Providers & Region Alias (terraform/versions.tf)

File: terraform/versions.tf • Trọng tâm: Cấu hình provider phụ us_east_1 chuyên cho ACM SSL validation 
terraform {
  required_version = ">= 1.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}

# Provider mặc định (sử dụng region cấu hình ở aws configure local)
provider "aws" {}

# Provider phụ với alias "us_east_1" bắt buộc định tuyến cho việc xác thực SSL ACM toàn cầu
provider "aws" {
  alias  = "us_east_1"
  region = "us-east-1"
}

2. Ràng buộc kiểm tra biến đầu vào (terraform/variables.tf)

File: terraform/variables.tf • Trọng tâm: Khối validation kiểm tra định dạng tên project và giới hạn môi trường 
variable "project_name" {
  description = "Name prefix for all resources"
  type        = string
  validation {
    # Chỉ cho phép chữ thường, số và dấu gạch ngang
    condition     = can(regex("^[a-z0-9-]+$", var.project_name))
    error_message = "Project name must contain only lowercase letters, numbers, and hyphens."
  }
}

variable "environment" {
  description = "Environment name (dev, test, prod)"
  type        = string
  validation {
    # Ngăn chặn các giá trị sai như staging, local, lab
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "Environment must be one of: dev, test, prod."
  }
}

variable "bedrock_model_id" {
  description = "Bedrock model ID"
  type        = string
  default     = "amazon.nova-micro-v1:0"
}

variable "lambda_timeout" {
  description = "Lambda function timeout in seconds"
  type        = number
  default     = 60
}

variable "api_throttle_burst_limit" {
  type        = number
  default     = 10
}

variable "api_throttle_rate_limit" {
  type        = number
  default     = 5
}

variable "use_custom_domain" {
  type        = bool
  default     = false
}

variable "root_domain" {
  type        = string
  default     = ""
}

3. Định nghĩa Tài nguyên & Dependency Link (terraform/main.tf)

File: terraform/main.tf • Trọng tâm: Gắn logic depends_on để truyền CORS từ CloudFront sang Lambda và phân quyền S3 public/private 
# Data source để lấy Account ID của tài khoản AWS đang chạy
data "aws_caller_identity" "current" {}

locals {
  # Cú pháp gán tên tiền tố phân biệt môi trường
  name_prefix = "${var.project_name}-${var.environment}"

  aliases = var.use_custom_domain && var.root_domain != "" ? [
    var.root_domain,
    "www.${var.root_domain}"
  ] : []

  common_tags = {
    Project     = var.project_name
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# 1. S3 Memory Bucket (Chặn public access tuyệt đối để bảo vệ lịch sử hội thoại)
resource "aws_s3_bucket" "memory" {
  bucket = "${local.name_prefix}-memory-${data.aws_caller_identity.current.account_id}"
  tags   = local.common_tags
}

resource "aws_s3_bucket_public_access_block" "memory" {
  bucket                  = aws_s3_bucket.memory.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# 2. S3 Frontend Bucket (Mở Policy cho phép đọc công khai để CloudFront lấy assets)
resource "aws_s3_bucket" "frontend" {
  bucket = "${local.name_prefix}-frontend-${data.aws_caller_identity.current.account_id}"
  tags   = local.common_tags
}

resource "aws_s3_bucket_public_access_block" "frontend" {
  bucket                  = aws_s3_bucket.frontend.id
  block_public_acls       = false
  block_public_policy     = false
  ignore_public_acls      = false
  restrict_public_buckets = false
}

resource "aws_s3_bucket_website_configuration" "frontend" {
  bucket = aws_s3_bucket.frontend.id
  index_document { suffix = "index.html" }
  error_document { key = "404.html" }
}

resource "aws_s3_bucket_policy" "frontend" {
  bucket = aws_s3_bucket.frontend.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "PublicReadGetObject"
        Effect    = "Allow"
        Principal = "*"
        Action    = "s3:GetObject"
        Resource  = "${aws_s3_bucket.frontend.arn}/*"
      }
    ]
  })
  depends_on = [aws_s3_bucket_public_access_block.frontend]
}

# 3. AWS Lambda Function
resource "aws_lambda_function" "api" {
  filename         = "${path.module}/../backend/lambda-deployment.zip"
  function_name    = "${local.name_prefix}-api"
  role             = aws_iam_role.lambda_role.arn
  handler          = "lambda_handler.handler"
  source_code_hash = filebase64sha256("${path.module}/../backend/lambda-deployment.zip")
  runtime          = "python3.12"
  timeout          = var.lambda_timeout
  tags             = local.common_tags

  environment {
    variables = {
      # Đọc URL CloudFront động hoặc tên miền tùy chỉnh để bảo mật CORS đầu vào
      CORS_ORIGINS     = var.use_custom_domain ? "https://${var.root_domain},https://www.${var.root_domain}" : "https://${aws_cloudfront_distribution.main.domain_name}"
      S3_BUCKET        = aws_s3_bucket.memory.id
      USE_S3           = "true"
      BEDROCK_MODEL_ID = var.bedrock_model_id
    }
  }

  # Bắt buộc đợi CloudFront Distribution được tạo trước để có domain_name cấu hình CORS_ORIGINS
  depends_on = [aws_cloudfront_distribution.main]
}
Section 8

Shell & PowerShell Scripts - Kịch bản tự động hóa đầu cuối

Để đơn giản hóa việc triển khai fullstack và dọn dẹp hạ tầng, chúng ta viết các script tự động hóa cho cả môi trường Linux/macOS (Bash) và Windows (PowerShell).

Script Triển khai (deploy.sh / deploy.ps1)

Thực hiện build backend, provision hạ tầng, cấu hình URL và build frontend:

  • Khởi tạo Workspace: Kiểm tra workspace môi trường (ví dụ: dev). Tạo mới nếu chưa tồn tại qua lệnh terraform workspace new dev, hoặc chọn workspace hiện có qua select.
  • Apply tự động: Thực thi terraform apply -auto-approve để tự động xác nhận không cần gõ "yes".
  • Ghi đè cấu hình: Trích xuất URL từ terraform output -raw, ghi thẳng vào file frontend/.env.production.
  • Build & Sync: Build Next.js ra thư mục out/, đẩy lên S3 frontend qua aws s3 sync --delete.

Script Hủy bỏ (destroy.sh / destroy.ps1)

Khắc phục lỗi S3 chứa file không cho xóa bằng cách empty bucket tự động trước khi hủy:

  • Empty Buckets: Script chạy lệnh CLI aws s3 rm s3://[bucket-name] --recursive cho cả bucket memory và bucket frontend.
  • Hủy hạ tầng: Chạy lệnh terraform destroy -auto-approve để gỡ bỏ toàn bộ tài nguyên đã tạo.
  • Giữ Workspace: State file trống được lưu lại. Để xóa hoàn toàn workspace rác: 
    terraform workspace select default
terraform workspace delete dev

Lưu ý quyền thực thi trên Unix (Linux/macOS): Sau khi tạo file shell script, bắt buộc phải cấp quyền thực thi bằng lệnh chmod +x scripts/deploy.sh scripts/destroy.sh trước khi chạy.

Section 9

Options / Trade-offs - Bản đồ lựa chọn thiết kế

Phương án - Option Ưu điểm - Pros Nhược điểm - Cons Ngữ cảnh lựa chọn - When to choose
Local State File (Mặc định) Rất đơn giản, cấu hình mặc định chạy ngay không cần thiết lập hạ tầng lưu trữ phụ trợ, tốc độ chạy nhanh. Không hỗ trợ làm việc nhóm (team collaboration), dễ mất dữ liệu nếu hỏng ổ cứng local, có nguy cơ lộ thông tin nhạy cảm. Phù hợp cho các bài thực hành cá nhân, môi trường lab học tập nghiên cứu.
Remote State (S3 + DynamoDB Lock) (Recommended) Hỗ trợ làm việc nhóm hoàn hảo, DynamoDB giúp khóa trạng thái (state locking) tránh xung đột đồng thời, S3 hỗ trợ versioning khôi phục lịch sử state. Phức tạp, phải viết thêm khối cấu hình backend và phải tạo trước S3 bucket + DynamoDB table thủ công từ trước. Bắt buộc cho mọi ứng dụng production thực tế của doanh nghiệp có từ 2 kỹ sư vận hành trở lên.
DNS Cố định cho API Gateway Frontend có thể hardcode URL API cố định, đơn giản hóa quy trình build frontend mà không cần script ghi đè .env.production động. Bắt buộc phải sở hữu custom domain, tốn kém chi phí mua tên miền và Route 53 hosting, không linh hoạt cho môi trường thử nghiệm nhanh. Khuyên dùng cho môi trường Production chính thức của sản phẩm thương mại.
Ghi đè biến động qua Script (Recommended) Vô cùng linh hoạt, hoạt động ổn định trên mọi môi trường dev/test/prod với bất kỳ URL ngẫu nhiên nào do AWS sinh ra mà không tốn chi phí. Làm tăng độ phức tạp của script deploy và tăng thời gian build frontend của đường ống CI/CD. Giải pháp tối ưu cho mọi đường ống phát triển/kiểm thử tự động hóa (dev/test env) của dự án.
Section 10

Pitfalls - Bẫy phát sinh lỗi & Cách khắc phục

Các trường hợp dễ gây lỗi (Failure Modes)

  • Lỗi S3 BucketNotEmpty / BucketAlreadyExists: Chạy lệnh terraform destroy bị crash giữa chừng và từ chối xóa S3 bucket, hoặc terraform apply báo trùng tên bucket toàn cầu.
  • Trang web báo undefined API URL hoặc fetch failed: Frontend CloudFront tải thành công giao diện chatbot nhưng khi gửi tin nhắn báo lỗi kết nối mạng (500/undefined).
  • Lỗi AccessDenied Credentials: Chạy lệnh terraform init hoặc plan báo lỗi thiếu thông tin tài khoản AWS credentials.
  • Route 53 ACM Certificate Validation Timeout: Quá trình deploy môi trường `prod` bị treo rất lâu (hơn 30 phút) ở bước tạo SSL certificate.

Biện pháp ngăn ngừa & Xử lý (Fix / Prevention)

  • Empty trước khi destroy & Gắn Account ID: Luôn sử dụng script destroy.sh/.ps1 để tự động chạy lệnh CLI xóa sạch file trước. Đồng thời nạp data.aws_caller_identity để sinh tên bucket chứa Account ID tránh trùng lặp toàn cầu.
  • Kiểm tra NEXT_PUBLIC_ prefix: Đảm bảo biến môi trường trong code React React được gọi đúng cú pháp process.env.NEXT_PUBLIC_API_URL và script deploy ghi đè chính xác vào tệp .env.production.
  • Cấu hình AWS CLI mặc định: Đảm bảo đã chạy lệnh aws configure trên máy local với thông tin tài khoản IAM User aiengineer có đầy đủ 6 policies trong group TwinAccess.
  • Kiểm tra bản ghi xác thực DNS: Đảm bảo Route 53 hosted zone đã tạo các bản ghi validation CNAME chính xác. ACM Certificate bắt buộc phải được deploy tại region us-east-1.
Section 11

Flashcards & Quick Quiz - Ôn tập tương tác

Q: Tại sao file terraform.tfstate tuyệt đối không được đưa lên Git?

A: Vì file state chứa toàn bộ thông tin cấu hình nhạy cảm dưới dạng bản rõ (plaintext), bao gồm cả biến môi trường, API keys, cơ sở dữ liệu credentials và mã định danh hạ tầng. Rò rỉ file state lên GitHub mang lại rủi ro bảo mật hệ thống nghiêm trọng.

Q: Tại sao ta phải tạo Workspace mới trong Terraform trước khi apply một môi trường?

A: Để cô lập file state. Nếu không tạo workspace, tất cả các môi trường (dev, test, prod) sẽ ghi chung vào file state mặc định default, dẫn đến việc môi trường này ghi đè và phá hủy hạ tầng của môi trường kia khi chạy apply.

Q: Tại sao chứng chỉ SSL (ACM Certificate) sử dụng với CloudFront bắt buộc phải deploy ở us-east-1?

A: Đây là yêu cầu kiến trúc của AWS: CloudFront là dịch vụ phân phối toàn cầu (Global Edge Network) và mặt bằng điều khiển trung tâm quản lý chứng chỉ SSL tùy chỉnh của CloudFront chỉ tích hợp với kho lưu trữ ACM đặt tại vùng Bắc Virginia (us-east-1).

Section 12

Quick Memory - Ghi nhớ nhanh cuối bài

Must remember - Điều bắt buộc phải nhớ

  1. Empty S3 bucket trước khi tiến hành xóa bucket để tránh lỗi AWS chặn.
  2. Chuyển CloudFront distribution sang Disabled và chờ 5-10 phút trước khi nhấn Delete.
  3. Tiền tố NEXT_PUBLIC_ cho phép React client-side truy cập biến môi trường Next.js.
  4. Mọi file state .tfstate và folder .terraform/ phải được đưa vào .gitignore.
  5. Tạo alias provider us_east_1 để cấu hình ACM SSL ở Bắc Virginia cho CloudFront.
