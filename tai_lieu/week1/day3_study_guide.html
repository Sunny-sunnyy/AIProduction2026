Cheat Sheet - Tóm tắt một trang

Core Idea - Ý chính

Day 3 tập trung vào việc biến ứng dụng generative AI mở thành một sản phẩm SaaS thương mại bằng cách bọc giao diện Next.js trong một lớp bảo vệ người dùng (Clerk Auth) và tích hợp cổng thanh toán (Clerk Billing/Stripe). Điều này đảm bảo mỗi request API gửi lên FastAPI đều được ký số, xác thực và giới hạn quyền truy cập theo gói cước Premium.

Must Know - Cần nhớ

  • Clerk SDK bản v6 (`@clerk/nextjs@6.39.0`) được sử dụng để duy trì tính tương thích với cấu trúc của Pages Router.
  • Client gửi JWT token qua HTTP Header Authorization: Bearer <token> cho backend FastAPI.
  • Backend FastAPI giải mã token bằng chữ ký công khai lấy từ CLERK_JWKS_URL giúp xác thực không trạng thái cực nhanh.
  • Token mặc định hết hạn sau 60 giây, gây lỗi 403 trên luồng kết nối dài. Cần cơ chế tự động refresh token và tái kết nối SSE.

Use in Practice - Thực hành

Áp dụng khi cần thương mại hóa ứng dụng AI nhanh chóng dưới dạng MVP (Minimum Viable Product). Giúp giải quyết nhanh bài toán quản lý người dùng, đăng nhập social auth, tích hợp cổng thanh toán mô phỏng (Clerk Gateway) hoặc thật (Stripe) mà không tốn hàng tuần xây dựng cơ sở hạ tầng auth/billing từ đầu.

Section 4

Core Theory - Lý thuyết cốt lõi

Slide Transcript

Key Concepts - Khái niệm chính

  • JSON Web Token (JWT) - mã thông báo web JSON: Một định dạng mã bảo mật dạng chuỗi nén dùng để truyền tải thông tin danh tính giữa client và server. JWT được ký số bằng khóa riêng tư của Clerk để đảm bảo không bị giả mạo.
  • JSON Web Key Set (JWKS) - bộ khóa web JSON: Tập hợp các khóa mật mã công khai do Clerk cung cấp qua một URL công khai. Backend sử dụng các khóa này để giải mã và kiểm tra chữ ký của JWT độc lập mà không cần thực hiện một request truy vấn mạng ngược lại Clerk server.
  • Stateless Authentication - xác thực phi trạng thái: Cơ chế xác thực không lưu phiên (session) trong bộ nhớ RAM hoặc database của backend. Backend chỉ cần giải mã token JWT thành công là đủ để phê duyệt request, giúp hệ thống mở rộng quy mô (scaling) rất tốt.
  • Zero-config Payment Gateway - cổng thanh toán không cấu hình: Cổng giả lập tích hợp sẵn của Clerk cho phép lập trình viên kiểm thử toàn bộ luồng thanh toán bằng thẻ test ngay lập tức mà không cần tạo tài khoản Stripe thật.
  • PCI-DSS Compliance - tuân thủ chuẩn bảo mật thẻ thanh toán: Tiêu chuẩn bảo mật bắt buộc đối với hệ thống lưu trữ/xử lý thông tin thẻ tín dụng. Việc tích hợp Stripe/Clerk Billing giúp startup đạt chuẩn này tự động vì thông tin thẻ truyền thẳng đến cổng thanh toán, không đi qua server của ứng dụng.

Concept Relationships - Quan hệ giữa các khái niệm

Mối quan hệ chặt chẽ giữa các thành phần được thể hiện qua sơ đồ kiến trúc truyền nhận JWT. Client (Next.js) đóng vai trò lấy token từ nhà cung cấp định danh Clerk, sau đó đính kèm token này vào mọi API request gửi tới Backend (FastAPI). Backend dựa vào cấu hình JWKS URL để tải khóa công khai và xác minh tính hợp lệ của token mà không phát sinh thêm tải (overhead) cho database.

Đồng thời, Billing System của Clerk liên kết chặt chẽ với trạng thái user session. Khi người dùng thanh toán qua cổng giả lập, Clerk cập nhật metadata của session token, giúp frontend Next.js tự động cập nhật giao diện thông qua component <Protect>.

Deep Explanation - Giải thích sâu về JWKS và Bảo mật phi trạng thái

Trong cơ chế xác thực truyền thống (Stateful Session), mỗi khi client gửi request, backend phải thực hiện truy vấn cơ sở dữ liệu hoặc Redis session store để kiểm tra xem session ID có hợp lệ hay không. Điều này tạo ra nút thắt cổ chai về hiệu năng (I/O bottleneck) khi số lượng người dùng tăng cao.

Với Stateless JWT, token bản thân nó đã chứa đầy đủ thông tin định danh (như User ID sub, thời gian hết hạn exp, và các quyền hạn org hay subscription) được mã hóa. Chữ ký số đi kèm token chứng minh rằng thông tin này do Clerk tạo ra và không bị thay đổi trên đường truyền.

Để xác minh chữ ký này mà không phải gọi một request mạng (HTTP request) đến Clerk (điều này sẽ làm tăng độ trễ mạng lên hàng trăm mili-giây), backend tải trước danh sách các khóa công khai từ CLERK_JWKS_URL và lưu vào bộ nhớ cache. Khi nhận được một JWT mới, thuật toán mã hóa bất đối xứng sử dụng khóa công khai này để xác nhận chữ ký của token trong vài mili-giây ở cấp CPU. Đây chính là bí quyết giúp các hệ thống API có hiệu năng cao và khả năng mở rộng không giới hạn.

Workflow / Pipeline - Quy trình hoạt động đầu-cuối

Code Slide

High-level Flow - Luồng tổng quát

  1. Khởi tạo: Ứng dụng Next.js bọc trong <ClerkProvider> tại pages/_app.tsx. Người dùng truy cập route được bảo vệ pages/product.tsx.
  2. Kiểm tra quyền: Component <Protect plan="premium_subscription"> đánh giá trạng thái người dùng. Nếu chưa thanh toán, hiển thị fallback <PricingTable />.
  3. Thanh toán: Người dùng mua gói qua Clerk Billing -> token session được cập nhật tự động phía Client.
  4. Lấy Token & Khởi tạo SSE: Hàm connectWithFreshToken() lấy token JWT mới và tạo kết nối fetchEventSource('/api') đính kèm token trong header Authorization.
  5. Xác thực API Backend: FastAPI chặn request tại middleware clerk_guard, lấy JWT verify bằng các khóa công khai của JWKS.
  6. Stream Ý Tưởng: Backend gọi OpenAI API với tham số stream và sinh dữ liệu định dạng SSE trả về client.

Data & Control Flow - Luồng dữ liệu / điều khiển

Luồng điều khiển được định hướng hoàn toàn bởi cơ chế khai báo (declarative check) ở frontend và dependency injection ở backend. Dữ liệu là luồng văn bản (text stream) truyền tải bất đồng bộ thông qua giao thức Server-Sent Events (SSE). Client sử dụng AbortController để chủ động điều phối luồng dữ liệu, cho phép hủy kết nối cũ và tái lập kết nối mới khi phát hiện token JWT hết hạn 60 giây.

End-to-End Detailed Workflow Walkthrough (Chi tiết cấp hàm)

Dưới đây là chi tiết quy trình khởi chạy, gọi hàm và trao đổi dữ liệu giữa các file trong hệ thống:

Giai đoạn 1: Khởi chạy và Kiểm tra quyền (Frontend)

  • Bước 1.1: pages/_app.tsx là điểm đầu vào của Next.js Pages Router. Hàm MyApp bọc toàn bộ component trong thẻ <ClerkProvider> để khởi tạo context xác thực toàn cục, cho phép các page con truy cập trạng thái auth.
  • Bước 1.2: Khi người dùng truy cập trang /product, file pages/product.tsx được thực thi. Hàm default export Product chạy đầu tiên.
  • Bước 1.3: Product sử dụng component <Protect> của Clerk SDK. Component này thực hiện kiểm tra thuộc tính plan="premium_subscription":
    • Nếu user chưa đăng nhập hoặc chưa đăng ký gói Premium, <Protect> sẽ ngăn chặn việc render component con và thay bằng giao diện khai báo trong thuộc tính fallback (chứa component <PricingTable />).
    • Nếu user đã sở hữu gói Premium, <Protect> tiến hành render component con <IdeaGenerator />.

Giai đoạn 2: Khởi tạo kết nối SSE (Frontend)

  • Bước 2.1: Component IdeaGenerator được mount, kích hoạt hook useEffect chạy duy nhất một lần (do mảng dependency rỗng).
  • Bước 2.2: Hook useEffect định nghĩa và gọi ngay lập tức hàm bất đồng bộ connectWithFreshToken().
  • Bước 2.3: Hàm connectWithFreshToken thực hiện:
    • Kiểm tra cờ isConnecting để tránh kết nối trùng lặp. Nếu đang kết nối, thoát hàm.
    • Hủy kết nối cũ đang chạy (nếu có) thông qua hàm controller.abort(), sau đó gán controller = new AbortController().
    • Gọi hàm getToken() (lấy từ hook useAuth()) để yêu cầu Clerk Client cấp một mã JWT mới nhất.
    • Nếu lấy thành công JWT, gọi hàm thư viện fetchEventSource('/api', { headers: { Authorization: `Bearer ${jwt}` }, signal: controller.signal, ... }).

Giai đoạn 3: Xác thực và Xử lý nghiệp vụ (Backend FastAPI)

  • Bước 3.1: Request SSE được truyền đến Backend FastAPI tại file api/index.py ở endpoint /api.
  • Bước 3.2: Endpoint idea khai báo dependency Depends(clerk_guard). Trước khi chạy code của hàm, FastAPI chuyển quyền kiểm soát sang đối tượng clerk_guard (kiểu ClerkHTTPBearer).
  • Bước 3.3: Hàm verify của clerk_guard đọc Authorization Header, bóc tách chuỗi token JWT sau chữ Bearer :
    • Tải công khai JWKS từ địa chỉ cấu hình trong biến môi trường CLERK_JWKS_URL.
    • Giải mã chữ ký số của JWT và kiểm tra tính toàn vẹn, đảm bảo token chưa hết hạn (nhỏ hơn 60 giây).
    • Nếu không hợp lệ, ném ngoại lệ trả về HTTP 403 Forbidden về phía Client.
    • Nếu hợp lệ, trả về đối tượng credentials và tiếp tục chạy hàm idea.
  • Bước 3.4: Hàm idea trích xuất User ID từ thuộc tính creds.decoded["sub"].
  • Bước 3.5: Hàm khởi tạo OpenAI client và gọi phương thức client.chat.completions.create(..., stream=True) để lấy một generator stream từ OpenAI.
  • Bước 3.6: Hàm định nghĩa generator nội bộ event_stream(), duyệt qua các chunk kết quả trả về của OpenAI. Định dạng mỗi chunk text thành chuẩn SSE: yield f"data: {line}\n\n".
  • Bước 3.7: Trả về đối tượng StreamingResponse(event_stream(), media_type="text/event-stream") duy trì kết nối HTTP dài để stream dữ liệu về trình duyệt.

Giai đoạn 4: Nhận Dữ liệu & Xoay vòng Token (Frontend Client)

  • Bước 4.1: Phía client Next.js tiếp nhận stream tại callback onmessage(ev) của hàm fetchEventSource. Hàm đọc ev.data, nối vào biến buffer và cập nhật state setIdea(buffer) để hiển thị văn bản ra màn hình thông qua ReactMarkdown.
  • Bước 4.2: Nếu kết nối đang chạy mà token JWT hết hạn (quá 60 giây) và backend ngắt kết nối bằng mã lỗi 403:
    • Sự kiện lỗi được bắt tại callback onerror(err) hoặc onopen(response) của client.
    • Code kiểm tra xem mã lỗi có phải là 403 hay không.
    • Nếu đúng, client chủ động cập nhật UI setIdea('Refreshing connection...'), reset buffer, đặt cờ isConnecting = false, hủy controller kết nối cũ, chờ 1 giây rồi gọi đệ quy hàm connectWithFreshToken() để lấy token mới và tái kết nối tự động.
  • Bước 4.3: Khi stream kết thúc bình thường, callback onclose() của fetchEventSource được kích hoạt để giải phóng cờ isConnecting = false và đóng kết nối an toàn.
Section 6

Techniques - Kỹ thuật sử dụng

1. Bearer JWT Authorization Header

Purpose - Mục đích: Truyền tải an toàn thông tin định danh người dùng đã được ký mã hóa từ Client lên Backend API.

When to use - Dùng khi nào: Dùng cho mọi API request cần bảo vệ và định danh người dùng trong các ứng dụng decoupled (frontend và backend tách biệt).

Trade-off - Đánh đổi: Phải quản lý việc lấy token bất đồng bộ ở frontend và đính kèm thủ công vào header của request. Cần lưu ý bảo mật token trong RAM, tránh lưu trữ lâu dài ở LocalStorage để giảm thiểu rủi ro XSS (Cross-Site Scripting).

2. Client-side Auth & Billing Guards

Purpose - Mục đích: Sử dụng các component khai báo (declarative components) của Clerk như <SignedIn>, <SignedOut>, và <Protect> để ẩn/hiển thị giao diện phù hợp với trạng thái đăng nhập và thanh toán.

When to use - Dùng khi nào: Xây dựng thanh điều hướng (navbar), trang Landing Page và bảo vệ định tuyến client-side nhanh chóng.

Trade-off - Đánh đổi: Giao diện ứng dụng Next.js bị phụ thuộc sâu vào cấu trúc và phiên bản của Clerk SDK. Nếu cập nhật thư viện lên bản mới hơn (như v7), một số component có thể bị thay đổi hành vi hoặc bị loại bỏ.

3. Auto Token Refresh & Reconnect (SSE)

Purpose - Mục đích: Bắt lỗi HTTP 403 Forbidden phát sinh do token JWT của Clerk hết hạn sau 60 giây, tự động lấy token mới và tái kết nối SSE không gây ngắt quãng trải nghiệm.

When to use - Dùng khi nào: Khi kết nối Server-Sent Events hoặc WebSocket kéo dài hơn thời gian hết hạn mặc định của token xác thực.

Trade-off - Đánh đổi: Phải viết code kết nối phức tạp hơn bằng cách tự quản lý trạng thái qua AbortController và cờ hiệu isConnecting, thay vì chỉ sử dụng hàm fetch mặc định đơn giản.

4. Hide My Email Testing Method

Purpose - Mục đích: Đăng ký nhiều tài khoản độc lập để kiểm thử luồng mua gói cước Premium nhiều lần mà không bị Clerk tự động gộp (Identity Consolidation) vào tài khoản Google gốc.

When to use - Dùng khi nào: Trong giai đoạn chạy thử nghiệm luồng thanh toán đầu-cuối (End-to-End Testing) trên môi trường Staging/Production.

Trade-off - Đánh đổi: Cần sử dụng các tài khoản Apple hoặc cấu hình email phụ (như Gmail alias `email+1@gmail.com`), tốn thời gian thao tác đăng ký thủ công.

"use client"

import { useEffect, useState } from 'react';
import ReactMarkdown from 'react-markdown';
import remarkGfm from 'remark-gfm';
import remarkBreaks from 'remark-breaks';
import { useAuth } from '@clerk/nextjs';
import { fetchEventSource } from '@microsoft/fetch-event-source';
import { Protect, PricingTable, UserButton } from '@clerk/nextjs';

function IdeaGenerator() {
    const { getToken } = useAuth();
    const [idea, setIdea] = useState<string>('…loading');

    useEffect(() => {
        let buffer = '';
        let controller: AbortController | null = null;
        let isConnecting = false; // Cờ chống gửi trùng yêu cầu kết nối song song

        const connectWithFreshToken = async () => {
            if (isConnecting) return;
            isConnecting = true;

            try {
                // Bước 1: Huỷ kết nối cũ trước khi tạo kết nối mới
                if (controller) {
                    controller.abort();
                }
                controller = new AbortController();

                // Bước 2: Lấy token JWT mới nhất từ Clerk Client
                const jwt = await getToken();
                if (!jwt) {
                    setIdea('Authentication required');
                    isConnecting = false;
                    return;
                }

                console.log('Connecting with fresh token...');

                // Bước 3: Khởi chạy kết nối Event Source dài
                await fetchEventSource('/api', {
                    headers: { Authorization: `Bearer ${jwt}` }, // Gửi kèm token xác thực
                    signal: controller.signal, // Gắn tín hiệu huỷ kết nối
                    onmessage(ev) {
                        console.log('Received message:', ev.data);
                        buffer += ev.data;
                        setIdea(buffer);
                    },
                    onerror(err) {
                        console.error('SSE onerror called:', err);
                        isConnecting = false;

                        // Bước 4: Nhận diện lỗi 403 (Token Clerk hết hạn mặc định sau 60s)
                        if (err instanceof Response && err.status === 403) {
                            console.log('Token expired in onerror, reconnecting...');
                            buffer = '';
                            setIdea('Refreshing connection...');
                            // Chờ 1 giây để Clerk làm mới phiên rồi gọi lại hàm đệ quy
                            setTimeout(() => connectWithFreshToken(), 1000);
                            return;
                        }
                        console.log('Non-403 error, letting fetchEventSource handle it');
                    },
                    onopen: async (response) => {
                        console.log('SSE onopen called, status:', response.status);

                        if (response.ok) {
                            console.log('SSE connection opened successfully');
                            isConnecting = false;
                        } else if (response.status === 403) {
                            // Bước 5: Chủ động bắt lỗi 403 ngay khi bắt đầu mở kết nối
                            console.log('403 detected in onopen, triggering reconnect');
                            isConnecting = false;
                            buffer = '';
                            setIdea('Refreshing connection...');
                            setTimeout(() => connectWithFreshToken(), 1000);
                            throw new Error('Authentication failed - triggering reconnect');
                        } else {
                            console.log('Non-403 error in onopen:', response.status);
                            isConnecting = false;
                            throw new Error(`HTTP ${response.status}`);
                        }
                    },
                    onclose() {
                        console.log('SSE connection closed');
                        isConnecting = false;
                    }
                });
            } catch (error) {
                console.error('Failed to connect:', error);
                isConnecting = false;

                // Tự động thử kết nối lại nếu gặp lỗi mạng thuần túy
                if (!controller?.signal.aborted) {
                    if (error instanceof TypeError || (error instanceof Error && error.message?.includes('fetch'))) {
                        console.log('Network error, retrying...');
                        setTimeout(() => connectWithFreshToken(), 2000);
                    } else {
                        setIdea('Connection failed. Please refresh the page.');
                    }
                }
            }
        };

        connectWithFreshToken();

        // Cleanup: Huỷ kết nối SSE khi component bị unmount khỏi màn hình
        return () => {
            if (controller) {
                controller.abort();
            }
        };
    }, []);

    return (
        <div className="container mx-auto px-4 py-12">
            <header className="text-center mb-12">
                <h1 className="text-5xl font-bold bg-gradient-to-r from-blue-600 to-indigo-600 bg-clip-text text-transparent mb-4">
                    Business Idea Generator
                </h1>
                <p className="text-gray-600 dark:text-gray-400 text-lg">
                    AI-powered innovation at your fingertips
                </p>
            </header>

            <div className="max-w-3xl mx-auto">
                <div className="bg-white dark:bg-gray-800 rounded-2xl shadow-xl p-8 backdrop-blur-lg bg-opacity-95">
                    {idea === '…loading' ? (
                        <div className="flex items-center justify-center py-12">
                            <div className="animate-pulse text-gray-400">
                                Generating your business idea...
                            </div>
                        </div>
                    ) : (
                        <div className="markdown-content text-gray-700 dark:text-gray-300">
                            <ReactMarkdown remarkPlugins={[remarkGfm, remarkBreaks]}>
                                {idea}
                            </ReactMarkdown>
                        </div>
                    )}
                </div>
            </div>
        </div>
    );
}

Backend FastAPI tiếp nhận request, verify chữ ký JWT không trạng thái bằng bộ khóa JWKS URL và thực hiện streaming kết quả.

FastAPI Backend clerk_guard Dependency OpenAI Stream Response 
import os
from fastapi import FastAPI, Depends  # type: ignore
from fastapi.responses import StreamingResponse  # type: ignore
from fastapi_clerk_auth import ClerkConfig, ClerkHTTPBearer, HTTPAuthorizationCredentials  # type: ignore
from openai import OpenAI  # type: ignore

app = FastAPI()

# Cấu hình đối tượng giải mã bằng JWKS URL lấy từ biến môi trường Clerk
clerk_config = ClerkConfig(jwks_url=os.getenv("CLERK_JWKS_URL"))
clerk_guard = ClerkHTTPBearer(clerk_config)

@app.get("/api")
# Đính kèm Depends(clerk_guard) để ép buộc FastAPI xác thực JWT trước khi chạy hàm
def idea(creds: HTTPAuthorizationCredentials = Depends(clerk_guard)):
    # Trích xuất User ID từ thuộc tính 'sub' của token đã được giải mã
    user_id = creds.decoded["sub"]  
    
    # Có thể sử dụng user_id này để:
    # 1. Ghi log lịch sử request của người dùng vào database
    # 2. Giới hạn số lượng request (rate limit) trong ngày
    
    client = OpenAI()
    prompt = [{"role": "user", "content": "Reply with a new business idea for AI Agents, formatted with headings, sub-headings and bullet points"}]
    stream = client.chat.completions.create(model="gpt-5-nano", messages=prompt, stream=True)

    # Hàm generator định dạng luồng SSE theo chuẩn
    def event_stream():
        for chunk in stream:
            text = chunk.choices[0].delta.content
            if text:
                lines = text.split("\n")
                # Đảm bảo mỗi dòng dữ liệu được bao bởi 'data: ' và kết thúc bằng hai ký tự xuống dòng
                for line in lines[:-1]:
                    yield f"data: {line}\n\n"
                    yield "data:  \n"
                yield f"data: {lines[-1]}\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")

Options / Trade-offs - Bản đồ lựa chọn

Lựa chọn thiết kế Phù hợp nhất khi Đánh đổi (Trade-off) Rủi ro chính (Risk)
Tự viết hệ thống Auth (Custom Auth) Dự án lớn, doanh nghiệp yêu cầu lưu trữ dữ liệu tại chỗ (On-Premise), đội ngũ kỹ sư bảo mật đông đảo. Tốn hàng tuần/tháng để phát triển; tự quản lý hạ tầng lưu trữ mật khẩu, session, hashing. Dễ gặp lỗ hổng bảo mật nghiêm trọng (SQL Injection, rò rỉ cơ sở dữ liệu mật khẩu) nếu không có kinh nghiệm sâu.
Sử dụng dịch vụ Clerk Auth Các dự án khởi nghiệp, sản phẩm MVP cần đưa ra thị trường nhanh để test tính năng (Go-to-market). Phụ thuộc hoàn toàn vào hạ tầng của Clerk; phát sinh chi phí hàng tháng khi lượng người dùng hoạt động (MAU) vượt hạn mức free. Ứng dụng có thể bị gián đoạn nếu Clerk Server gặp sự cố ngừng hoạt động (downtime).
Bảo vệ route ở Client-side (ẩn hiện UI) Ứng dụng đơn giản, UX mượt mà, không yêu cầu bảo mật quá cao về cấu trúc trang HTML/JS. Chỉ bảo mật ở mức giao diện trình duyệt; dữ liệu mã nguồn HTML vẫn được tải về. Nếu quên bảo mật ở API Endpoint backend, kẻ xấu vẫn có thể dùng công cụ để gọi trực tiếp API lấy dữ liệu thô.
Bảo vệ route ở Next.js Middleware Ứng dụng chứa thông tin cực kỳ nhạy cảm; muốn chặn request ngay tại Edge Server trước khi trả về HTML/JS. Cấu hình định tuyến phức tạp hơn; có thể làm tăng nhẹ độ trễ tải trang do phải xác thực trước ở Edge. Cấu hình sai có thể chặn luôn cả các file tĩnh (CSS, JS, hình ảnh) khiến trang web không tải được tài nguyên.
Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure Modes - Các lỗi dễ gặp

  • Lỗi 403 Forbidden sau 60 giây stream: Kết nối SSE bị ngắt giữa chừng khi AI đang viết dở ý tưởng. Lý do: Token JWT của Clerk chỉ có hiệu lực mặc định 60 giây để đảm bảo an toàn. Sau 60s, request backend từ chối token cũ.
  • Lỗi Màn hình xác thực kép (Double Auth): Người dùng truy cập trang chủ nhưng bị chặn bởi màn hình đăng nhập của Vercel thay vì giao diện của Clerk. Lý do: Quên tắt tính năng Deployment Protection trong cài đặt dự án của Vercel.
  • Lỗi Backend từ chối mọi token (401/403 liên tục): Client đã đăng nhập thành công và gửi request bình thường nhưng backend FastAPI luôn phản hồi không hợp lệ. Lý do: Cấu hình sai URL trong biến CLERK_JWKS_URL ở backend.

Fix / Prevention - Cách khắc phục

  • Khắc phục lỗi 403 SSE: Thực hiện bắt mã trạng thái 403 trong hàm onopen và onerror của fetchEventSource. Sử dụng AbortController để ngắt kết nối cũ một cách an toàn, sau đó lấy token mới bằng getToken() và thiết lập kết nối lại tự động.
  • Khắc phục lỗi xác thực kép Vercel: Vào Vercel Settings -> Project Settings -> Deployment Protection và chuyển trạng thái từ ON sang OFF, sau đó lưu lại.
  • Khắc phục cấu hình JWKS URL: Truy cập Clerk Dashboard -> Configure -> API Keys, sao chép chính xác dòng JWKS URL và cập nhật vào biến môi trường Vercel bằng câu lệnh vercel env add CLERK_JWKS_URL.

Flashcards & Quiz - Thẻ ôn tập tương tác

Q: Tại sao Clerk thiết lập thời hạn sống (TTL) của token JWT cực kỳ ngắn (chỉ 60 giây)?

A: Để giảm thiểu rủi ro bảo mật trong trường hợp token bị kẻ xấu đánh cắp (man-in-the-middle attack). Do token chỉ sống được 60 giây, kẻ xấu không thể tái sử dụng lâu dài. Token ngắn buộc Client phải liên tục xoay vòng lấy token mới (silent refresh) để bảo vệ người dùng.

Q: Vai trò của biến môi trường CLERK_JWKS_URL ở backend là gì?

A: Cung cấp địa chỉ để backend tải về bộ khóa công khai của Clerk. Khóa công khai này dùng để giải mã chữ ký số của JWT ngay lập tức ở CPU backend, thực hiện xác thực không trạng thái (stateless auth) mà không cần tạo request truy vấn mạng ngược lại server của Clerk.
