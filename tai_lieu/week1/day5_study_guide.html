Section 3

Core Theory - Lý thuyết chuyên sâu

Root User vs IAM User trên AWS

Root User (Tài khoản gốc): Là tài khoản được khởi tạo bằng email đăng ký đầu tiên. Tài khoản này có quyền tối cao và toàn phần đối với mọi tài nguyên, cấu hình mạng và thông tin thanh toán của hệ thống. Nếu rò rỉ Access Key của Root User lên GitHub, tin tặc có thể quét và chiếm quyền kiểm soát toàn bộ tài khoản, khởi chạy hàng trăm máy chủ ảo cấu hình cao để đào tiền ảo, dẫn đến những hóa đơn khổng lồ lên tới hàng chục nghìn USD.

IAM User (Người dùng danh tính): Được tạo ra dưới sự quản lý của Root User thông qua dịch vụ AWS Identity and Access Management. IAM User tuân thủ nghiêm ngặt Nguyên tắc đặc quyền tối thiểu (Least Privilege Access Control): chỉ được cấp đúng những quyền tối thiểu cần thiết để làm việc. Chúng ta sử dụng IAM User aiengineer cho công việc lập trình và deploy hàng ngày nhằm khoanh vùng thiệt hại tối đa nếu xảy ra sự cố bảo mật.

Triết lý AWS Budgets & Lưới an toàn tài chính

Một điểm khác biệt căn bản giữa AWS và các nhà cung cấp PaaS nhỏ hơn: AWS không hỗ trợ cơ chế giới hạn cứng (hard cap) tự động ngắt dịch vụ khi vượt quá ngân sách đã đặt. AWS ưu tiên tính liên tục của hệ thống sản xuất dành cho khách hàng doanh nghiệp lớn (ví dụ: không thể vì lỗi thanh toán hoặc vượt ngân sách nhỏ mà AWS tự động tắt các server đang vận hành của Netflix).

Do đó, trách nhiệm kiểm soát chi phí hoàn toàn thuộc về lập trình viên. Việc thiết lập AWS Budgets để gửi cảnh báo tự động về email cá nhân dựa trên cả chi phí thực tế (Actual Spend) lẫn chi phí dự báo (Forecasted Spend) của thuật toán AWS là bắt buộc. Thuật toán của AWS sẽ dự báo chi phí cả tháng dựa trên tốc độ tiêu thụ hiện tại của tài nguyên đang chạy, đóng vai trò cảnh báo sớm (early warning) giúp lập trình viên phát hiện sớm các tài nguyên quên tắt trước khi chúng thực sự tiêu tốn nhiều tiền.

Docker Containers vs Virtual Machines (VM)

Virtual Machines (Máy ảo - VM): Hoạt động dựa trên lớp phần cứng ảo hóa được quản lý bởi Hypervisor. Mỗi VM phải chạy một hệ điều hành khách (Guest OS) riêng biệt bên trong nó, dẫn đến dung lượng rất nặng (hàng chục GB), tiêu tốn lượng RAM lớn để duy trì hệ điều hành giả lập và mất vài phút để khởi động hoàn toàn.

Docker Containers: Là công nghệ ảo hóa ở cấp độ hệ điều hành (OS-level virtualization). Container không chứa hệ điều hành khách riêng biệt mà chia sẻ chung nhân hệ điều hành (OS Kernel) của máy chủ vật lý thông qua Docker Engine. Container chỉ đóng gói mã nguồn và các thư viện phụ thuộc của ứng dụng. Do đó, Docker container cực kỳ nhẹ (chỉ vài trăm MB), khởi động tức thì trong vòng vài giây và tiêu hao rất ít tài nguyên bộ nhớ.

Cơ chế hoạt động của AWS Lambda Web Adapter

AWS Lambda hoạt động theo cơ chế dựa trên sự kiện (Event-driven). Khi có request gửi tới, Lambda nhận một sự kiện JSON chứa thông tin HTTP request. Thông thường, để chạy một ứng dụng web như FastAPI trên Lambda, lập trình viên phải sửa đổi mã nguồn để sử dụng các thư viện trung gian như mangum dịch sự kiện JSON này.

AWS Lambda Web Adapter (LWA) là một công cụ mã nguồn mở của AWS Labs giúp giải quyết triệt để vấn đề này. Bằng cách copy binary của LWA vào thư mục /opt/extensions/lambda-adapter bên trong container, LWA hoạt động như một tiến trình extension ngầm. Khi Lambda khởi chạy container, LWA tự động bắt các sự kiện JSON từ Lambda runtime, dịch chúng thành HTTP Request chuẩn rồi gửi đến cổng 8000 của FastAPI, đồng thời nhận phản hồi HTTP từ FastAPI dịch ngược lại cho Lambda. Công nghệ này giúp chúng ta chạy mã nguồn FastAPI nguyên bản trên Lambda mà không cần chỉnh sửa bất kỳ dòng code backend nào.

Lý thuyết Docker Layers & Cache Optimization

Mỗi chỉ thị cấu hình trong Dockerfile (như RUN, COPY) tạo ra một lớp dữ liệu chỉ đọc (layer) xếp chồng lên nhau trong Docker Image. Khi build lại image, Docker engine sẽ kiểm tra xem nội dung của chỉ thị và các file liên quan có thay đổi không. Nếu không đổi, Docker sẽ tái sử dụng cache của layer cũ để tăng tốc độ build (Docker Cache).

Quy tắc vàng: Sắp xếp các chỉ thị ít thay đổi (như cài đặt thư viện hệ thống, cài đặt dependencies npm ci hoặc pip install) lên phía trước, và đặt các chỉ thị hay thay đổi (copy mã nguồn dự án) ở cuối cùng của Dockerfile. Điều này giúp giảm thời gian build image từ vài phút xuống còn vài giây khi cập nhật code ứng dụng.

Cân nhắc về Serverless Cold Start & RAM Allocation

Khi ứng dụng serverless không nhận request nào trong khoảng 15 phút, AWS Lambda sẽ tự động hủy container instance để giải phóng tài nguyên (Scale-to-Zero). Khi có request mới đến, Lambda phải thực hiện Cold Start (Khởi động lạnh): cấp phát máy chủ ảo, kéo Docker image từ ECR, dựng container, và khởi động FastAPI server. Quá trình này mất khoảng 10-30 giây.

Để giảm thiểu độ trễ khởi động lạnh, chúng ta áp dụng các kỹ thuật: tối giản hóa kích thước Docker image (sử dụng image nền python-slim), giảm thiểu import thư viện không cần thiết và đặc biệt là tăng dung lượng RAM cấp phát cho Lambda lên 1024 MB. Trên AWS Lambda, việc tăng RAM sẽ tỷ lệ thuận với việc AWS cấp thêm sức mạnh CPU ảo (vCPU), giúp container khởi động nhanh hơn đáng kể so với cấu hình RAM mặc định thấp.

Docker Multi-Stage Build & Xây dựng ảnh đa nền tảng

Multi-Stage Build: Tách biệt giai đoạn build (sử dụng môi trường Node.js đầy đủ để biên dịch Next.js tĩnh) và giai đoạn run (chỉ chạy FastAPI trên môi trường Python gọn nhẹ). Điều này giúp loại bỏ Node.js SDK và thư mục node_modules khổng lồ ra khỏi sản phẩm cuối cùng.

Multi-Platform Build: Các dòng máy Mac Apple Silicon chạy kiến trúc ARM64, trong khi máy chủ AWS Lambda mặc định chạy kiến trúc x86_64 (amd64). Do đó, bạn bắt buộc phải build Docker bằng cờ chỉ định nền tảng để tránh lỗi Exec format error khi Lambda khởi chạy container.

# Giai đoạn 1: Build Next.js
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Giai đoạn 2: Tạo Python container nhẹ
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY api/server.py .
COPY --from=builder /app/out ./static

Giải thích: Từ khóa AS builder đặt tên cho stage đầu tiên. Chỉ thị COPY --from=builder ở stage 2 chỉ copy thư mục tĩnh đã biên dịch mà không mang theo các dependencies của Node.js, tối ưu hóa dung lượng image chỉ còn khoảng 200MB.

Amazon ECR (Elastic Container Registry) - Kho ảnh Container

Amazon ECR là kho chứa Docker image private bảo mật tích hợp sâu với hệ thống phân quyền IAM của AWS. Để đẩy image lên ECR, Docker CLI cần xác thực thông qua một token tạm thời được cấp bởi AWS CLI.

# 1. Lấy token đăng nhập tạm thời (hiệu lực 12 giờ)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com

# 2. Tag image chuẩn cấu trúc ECR
docker tag app:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest

# 3. Đẩy image lên repository
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest

Giải thích: Lệnh get-login-password lấy token mã hóa ngắn hạn và pipe trực tiếp vào docker login qua toán tử |. Khi docker push chạy, ECR chỉ truyền tải những layers có thay đổi (layer caching) giúp quá trình deploy sau đó diễn ra cực kỳ nhanh chóng.

Triển khai Container Serverless với AWS Lambda & Function URLs

AWS Lambda hỗ trợ chạy các container image dung lượng lên tới 10GB, cho phép tích hợp các thư viện AI cồng kềnh (như PyTorch, OpenCV). Để chạy các framework như FastAPI mà không sửa đổi mã nguồn, chúng ta dùng Lambda Web Adapter.

Function URLs: Cung cấp HTTPS endpoint trực tiếp miễn phí đi kèm Lambda. Chế độ RESPONSE_STREAM cho phép stream dữ liệu trực tiếp về client mà không bị buffer, thích hợp với các tác vụ stream text từ OpenAI.

# FastAPI StreamingResponse cho Server-Sent Events (SSE)
from fastapi.responses import StreamingResponse

@app.post("/api/consultation")
def consultation_summary(visit: Visit):
    def event_stream():
        for chunk in stream:
            text = chunk.choices[0].delta.content
            if text:
                yield f"data: {text}\n\n"
    return StreamingResponse(event_stream(), media_type="text/event-stream")

Giải thích: Việc chọn Invoke Mode là RESPONSE_STREAM trên Function URL đảm bảo các khối dữ liệu từ StreamingResponse được gửi về trình duyệt lập tức thay vì bị giữ lại (buffered) cho đến khi toàn bộ request hoàn tất.

Lambda Execution Role & Cơ chế uỷ quyền trong AWS

Trong AWS, một IAM Role (vai trò) khác với IAM User (người dùng). IAM User dành cho con người hoặc công cụ lập trình ngoài AWS, còn IAM Role dành cho các tiến trình chạy bên trong hạ tầng AWS. Lambda cần một Execution Role (vai trò thực thi) để có quyền ghi logs vào CloudWatch hoặc gọi các dịch vụ khác.

// Trust Policy cho phép dịch vụ Lambda đảm nhận IAM Role này
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Giải thích: Trust Policy cấu hình quyền cho phép thực thể dịch vụ lambda.amazonaws.com được phép đảm nhận vai trò thực thi này. Việc gán IAMFullAccess cho IAM User aiengineer khi deploy là cần thiết để Lambda có thể tự động tạo ra role thực thi này mà không bị lỗi Access Denied.

import os
from pathlib import Path
from fastapi import FastAPI, Depends
from fastapi.responses import StreamingResponse, FileResponse
from fastapi.staticfiles import StaticFiles
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel
from fastapi_clerk_auth import ClerkConfig, ClerkHTTPBearer, HTTPAuthorizationCredentials
from openai import OpenAI

app = FastAPI()

# Gắn cấu hình CORS middleware để cho phép trình duyệt gọi API chéo domain trong lúc dev cục bộ
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# Khởi tạo bộ bảo vệ xác thực Clerk Auth từ khóa JWKS URL nạp từ biến môi trường
clerk_config = ClerkConfig(jwks_url=os.getenv("CLERK_JWKS_URL"))
clerk_guard = ClerkHTTPBearer(clerk_config)

# Khai báo cấu trúc dữ liệu Pydantic BaseModel để tự động validate đầu vào của request body
class Visit(BaseModel):
    patient_name: str
    date_of_visit: str
    notes: str

# System Prompt cố định phân chia chính xác cấu trúc đầu ra của bệnh án y khoa
system_prompt = """
You are provided with notes written by a doctor from a patient's visit.
Your job is to summarize the visit for the doctor and provide an email.
Reply with exactly three sections with the headings:
### Summary of visit for the doctor's records
### Next steps for the doctor
### Draft of email to patient in patient-friendly language
"""

def user_prompt_for(visit: Visit) -> str:
    return f"""Create the summary, next steps and draft email for:
Patient Name: {visit.patient_name}
Date of Visit: {visit.date_of_visit}
Notes:
{visit.notes}"""

@app.post("/api/consultation")
def consultation_summary(
    visit: Visit,
    creds: HTTPAuthorizationCredentials = Depends(clerk_guard), # Chốt chặn xác thực Clerk JWT Token
):
    # Trích xuất user ID của Clerk từ token đã được giải mã và kiểm tra chữ ký số thành công
    user_id = creds.decoded["sub"]
    client = OpenAI() # Khởi tạo OpenAI client (lấy API Key tự động từ biến môi trường)

    user_prompt = user_prompt_for(visit)
    prompt = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ]

    # Gửi request lên OpenAI API kích hoạt chế độ stream=True
    stream = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=prompt,
        stream=True,
    )

    # Hàm generator sử dụng cơ chế yield để sinh và đẩy dữ liệu chunk-by-chunk chuẩn định dạng SSE
    def event_stream():
        for chunk in stream:
            text = chunk.choices[0].delta.content
            if text:
                # Xử lý phân tách xuống dòng của Markdown để trình duyệt client hiển thị chuẩn xác
                lines = text.split("\n")
                for line in lines[:-1]:
                    yield f"data: {line}\n\n"
                    yield "data:  \n"
                yield f"data: {lines[-1]}\n\n"

    # Trả về StreamingResponse với kiểu MIME text/event-stream hỗ trợ Server-Sent Events
    return StreamingResponse(event_stream(), media_type="text/event-stream")

@app.get("/health")
def health_check():
    """Endpoint giám sát tình trạng hoạt động của container"""

Options & Trade-offs - Bản đồ lựa chọn kiến trúc

Bảng so sánh kỹ thuật chi tiết giúp Solopreneur và Startup đánh giá tối ưu các giải pháp tính toán compute container trên môi trường đám mây AWS trong năm 2026:

Tham số so sánh AWS Lambda Container (Lựa chọn tối ưu Day 5) AWS App Runner (Cũ - Chế độ bảo trì) Amazon ECS Express Mode (Fargate)
Chi phí tối thiểu hàng tháng Gần như $0/tháng (Tận dụng Free Tier vĩnh viễn của Lambda gồm 1 triệu request và 400.000 GB-giây miễn phí hàng tháng. Phí lưu trữ ECR chỉ ~$0.05 - $0.10/tháng). Từ $5 - $6/tháng (Bắt buộc phải trả phí duy trì tối thiểu 1 container instance luôn hoạt động). Từ $16 - $20/tháng (Bắt buộc trả phí duy trì cố định cho Application Load Balancer chạy không tải).
Cơ chế co giãn (Scaling) Scale to zero (Tự động tắt sạch container khi không có tải, co giãn tối đa tức thì khi có request). Tự động co giãn theo số lượng request đồng thời cấu hình sẵn (giới hạn dưới tối thiểu là 1). Co giãn theo ngưỡng sử dụng tài nguyên CPU/RAM (không hỗ trợ cơ chế scale-to-zero).
Độ trễ Cold Start Có hiện tượng trễ 10-30 giây ở request đầu tiên sau thời gian dài rảnh rỗi (~15 phút). Không có độ trễ khởi động lạnh do luôn duy trì ít nhất một instance chạy ấm. Không có độ trễ khởi động lạnh do container chạy liên tục 24/7.
Phản hồi Streaming (SSE) Hỗ trợ tuyệt vời qua Function URL cấu hình ở chế độ RESPONSE_STREAM. Hỗ trợ trực tiếp qua kết nối HTTP streaming thông thường. Đòi hỏi cấu hình Load Balancer hỗ trợ WebSocket hoặc thiết lập mạng phức tạp hơn.
Độ phức tạp mạng lưới Thấp nhất. Nhận HTTPS URL trực tiếp qua Function URL mà không cần tạo VPC hay API Gateway. Thấp. Tự động cấp phát tên miền HTTPS ngẫu nhiên có chứng chỉ bảo mật. Cao. Phải tự cấu hình VPC, Subnets, Route Tables, Security Groups và Load Balancers.
Đánh giá chung Tối ưu nhất cho MVP, Solopreneurs, dự án mới học tập cần chi phí vận hành bằng $0 và co giãn tự động tốt. Đang duy trì cho khách hàng cũ. Không nên sử dụng cho các dự án mới từ tháng 5/2026. Phù hợp cho các doanh nghiệp lớn, có lưu lượng truy cập lớn liên tục và ngân sách hạ tầng dồi dào.
Section 8

Pitfalls - Các lỗi thường gặp & Cách xử lý phòng ngừa

Các trường hợp lỗi (Failure Modes)

  • Lỗi Exec format error trong CloudWatch Logs: Khi truy cập Function URL, trình duyệt báo lỗi và kiểm tra log ghi nhận lỗi định dạng thực thi nhị phân. Lý do: Image được build mặc định trên máy Mac M1/M2/M3 dùng kiến trúc ARM64, trong khi máy chủ ảo của AWS Lambda mặc định chạy kiến trúc CPU x86_64 (amd64).
  • Giao diện chatbot bị đơ, chữ hiển thị toàn bộ cùng lúc: Kết quả summary của bác sĩ đứng im khoảng 15 giây rồi xuất hiện một lượt, làm mất hoàn toàn hiệu ứng chạy chữ streaming thời gian thực. Lý do: Invoke Mode của Lambda Function URL bị đặt nhầm thành BUFFERED (mặc định khi tạo trên console), làm AWS tích lũy toàn bộ dữ liệu phản hồi mới trả về một lượt.
  • Lỗi HTTP 502/503 Bad Gateway khi gọi API: Trang web hiển thị được giao diện tĩnh nhưng các yêu cầu sinh summary trả về lỗi 502 hoặc 503. Lý do: Cấu hình RAM của Lambda quá thấp (mặc định 128MB) dẫn đến Out of Memory (OOM) làm crash tiến trình uvicorn khi chạy FastAPI, hoặc điền thiếu các khóa biến môi trường trong cấu hình Lambda.

Cách khắc phục & Phòng ngừa (Fix/Prevention)

  • Khắc phục Exec format error: Luôn thêm flag --platform linux/amd64 vào câu lệnh docker build để Docker Desktop thực hiện biên dịch chéo sang tập lệnh amd64 tương thích đám mây.
  • Khắc phục lỗi đơ streaming: Vào Lambda Console → Tab Configuration → Mục Function URL → Click Edit → Sửa cấu hình Invoke Mode từ BUFFERED thành RESPONSE_STREAM và bấm lưu lại.
  • Khắc phục lỗi 502/503 Bad Gateway: Truy cập Lambda General configuration, tăng cấu hình bộ nhớ RAM lên tối thiểu 1024 MB và thời gian chờ Timeout lên 5 phút. Kiểm tra kỹ 3 biến môi trường CLERK_SECRET_KEY, CLERK_JWKS_URL và OPENAI_API_KEY xem có khoảng trắng thừa hay ký tự sai lệch không.
Section 9

Knowledge Extension - Kiến thức mở rộng thực tế

Quản lý vòng đời lưu trữ ảnh Docker với ECR Lifecycle Policies

Mỗi khi chạy script cập nhật ứng dụng, Docker build sẽ đẩy một image mới lên ECR với tag latest. Image cũ hơn sẽ bị mất tag và trở thành các ảnh orphan (untagged image).

Mặc dù không có nhãn định dạng, các image mồ côi này vẫn nằm trên ECR và tiêu tốn phí lưu trữ của AWS ECR (khoảng $0.10/GB/tháng). Bằng cách thiết lập ECR Lifecycle Policy cho repository, chúng ta có thể cấu hình quy tắc tự động hóa: **Tự động xóa các ảnh untagged cũ hơn 14 ngày**, giúp tối ưu hóa chi phí lưu trữ trên AWS hoàn toàn tự động mà không cần dọn dẹp thủ công.

Xây dựng CI/CD tự động bằng GitHub Actions

Trong quy trình vận hành công nghiệp, các lập trình viên không chạy lệnh build và push Docker thủ công từ terminal máy cá nhân vì dễ gây lỗi cấu hình. Thay vào đó, họ sử dụng **GitHub Actions** để tự động hóa hoàn toàn quy trình CI/CD:

Study Pack - Thẻ ôn tập tương tác & Quick Quiz

Q: Tại sao AWS không cung cấp giới hạn cứng (hard cap) tự động tắt dịch vụ khi vượt quá ngân sách đặt ra?

A: Nhằm bảo vệ tính liên tục hoạt động của doanh nghiệp sản xuất lớn (ví dụ: Không thể vì một sự cố lỗi billing nhỏ mà AWS tự động đánh sập hệ thống máy chủ của Netflix hay một ngân hàng). Trách nhiệm kiểm soát chi phí thuộc về lập trình viên thông qua thiết lập Budget Alerts gửi về email và thắt trần Reserved Concurrency ở mức thấp.

Q: Sự khác nhau về mặt tính phí giữa Reserved Concurrency và Provisioned Concurrency là gì?

A: **Reserved Concurrency** là hoàn toàn miễn phí, chỉ đóng vai trò chốt chặn giới hạn trần số lượng container chạy đồng thời của Lambda để tránh rò rỉ chi phí (ví dụ giới hạn bằng 2). **Provisioned Concurrency** là tính năng trả phí theo thời gian duy trì một số container ở trạng thái ấm liên tục trong RAM để loại bỏ cold start, phí này tính liên tục ngay cả khi không có request nào gửi tới ứng dụng.
