Chương 1: Cam kết của Sổ tay & Giới thiệu Tổng quan

Chào mừng bạn đến với cẩm nang Docker chuyên sâu dành riêng cho kỹ sư AI và lập trình viên hoạt động trên môi trường Windows kết hợp với WSL2 Ubuntu. Trong kỷ nguyên của trí tuệ nhân tạo và học máy, việc thiết lập một `development environment - môi trường phát triển` nhất quán, hiệu quả và có khả năng truy cập trực tiếp vào phần cứng như card đồ họa (GPU) là vô cùng quan trọng.

Sổ tay này được thiết kế để dẫn dắt bạn qua từng bước từ các khái niệm nền tảng đến các kỹ thuật thực chiến nâng cao, giúp bạn làm chủ Docker một cách nhanh chóng và chuyên nghiệp.

Tại sao lại tập trung vào Windows + WSL2 Ubuntu?

Windows cung cấp một giao diện người dùng thân thiện và hệ sinh thái ứng dụng văn phòng phong phú. Tuy nhiên, phần lớn các công cụ và thư viện AI/ML mạnh mẽ nhất (như PyTorch, TensorFlow) lại được tối ưu hóa tốt nhất trên Linux. WSL2 (Windows Subsystem for Linux 2) chính là chiếc cầu nối hoàn hảo, cho phép chạy một hệ phân phối Linux thực thụ trực tiếp trong Windows mà không cần sử dụng một giải pháp cồng kềnh như `virtual machine - máy ảo` truyền thống.

Nhờ có WSL2, Docker có thể chạy ở chế độ native với hiệu năng gần như tương đương với Linux thực tế, đồng thời hỗ trợ GPU acceleration thông qua NVIDIA CUDA để huấn luyện các mô hình học sâu.

Cam kết của Sổ tay (Handbook Promise):

Sau khi hoàn thành cuốn sổ tay này, bạn sẽ:

  • Hiểu rõ nguyên lý hoạt động sâu bên trong của Docker.
  • Thiết lập thành công và tối ưu hóa Docker Desktop tích hợp WSL2 Ubuntu với `input/output speed - tốc độ nhập/xuất` dữ liệu cao nhất.
  • Thành thạo các câu lệnh quản lý `container (vùng chạy cô lập)`, `image (ảnh đóng gói)`, `volume (vùng lưu dữ liệu)` và `network (mạng)`.
  • Triển khai các ứng dụng đa container phức tạp bằng Docker Compose.

Chương 2: Docker là gì? Định nghĩa & Sức mạnh của Container hóa

Docker là một nền tảng mã nguồn mở giúp tự động hóa quá trình đóng gói, phân phối và khởi chạy ứng dụng dưới dạng các `container (vùng chạy cô lập)` độc lập. Nó giải quyết triệt để lỗi kinh điển "chạy trên máy tôi thì được nhưng lên server lại lỗi".

Hãy tưởng tượng Docker giống như các container vận chuyển tiêu chuẩn hóa trong ngành hàng hải. Trước khi có container chuẩn hóa, việc xếp dỡ các loại hàng hóa khác nhau (gạo, ôtô, chất lỏng) lên tàu rất phức tạp và dễ hư hỏng. Container tiêu chuẩn hóa đã giải quyết vấn đề đó bằng cách đóng gói mọi thứ vào các hộp thép có cùng kích thước, dễ dàng xếp chồng và di chuyển bởi bất kỳ con tàu hay bến cảng nào. Tương tự, Docker đóng gói toàn bộ mã nguồn của ứng dụng cùng với các dependencies (thư viện phụ thuộc), thư viện hệ thống và cấu hình vào một thực thể duy nhất gọi là `image (ảnh đóng gói)`.

Docker Engine là gì?

Docker Engine là trái tim của hệ thống Docker. Đây là một ứng dụng client-server (mô hình khách - chủ) gồm ba thành phần chính:

  1. Docker Daemon (dockerd): Một `daemon (tiến trình nền)` chạy liên tục trên hệ điều hành máy chủ, chịu trách nhiệm quản lý các đối tượng của Docker như image, container, network và volume. Nó lắng nghe các yêu cầu thông qua Docker Engine API.
  2. REST API: Một giao diện lập trình ứng dụng chỉ định các điểm cuối (endpoints) mà các chương trình khác có thể sử dụng để giao tiếp và ra lệnh cho Docker daemon.
  3. Docker CLI (docker): `command line interface - giao diện dòng lệnh` cho phép người dùng gõ lệnh để tương tác với Docker daemon. CLI sử dụng REST API để gửi lệnh đến daemon.
Tại sao các đội ngũ phát triển lại lựa chọn Container hóa (Containerization)?
  • Tính nhất quán giữa các môi trường (Environment Consistency): Một container chạy tốt trên máy của lập trình viên chắc chắn sẽ hoạt động y hệt trên máy chủ thử nghiệm và `production environment - môi trường vận hành`.
  • Tiết kiệm tài nguyên (Resource Efficiency): Không giống như máy ảo, các container chia sẻ chung nhân hệ điều hành của máy chủ nên khởi động trong vài mili giây và tiêu tốn cực kỳ ít bộ nhớ RAM và CPU.
  • Kiến trúc hướng dịch vụ nhỏ (Microservices Architecture): Docker cho phép phân rã một ứng dụng cồng kềnh thành các dịch vụ độc lập, giúp dễ dàng nâng cấp, bảo trì và mở rộng quy mô.

Chương 3: So sánh Container và Máy ảo (Container vs Virtual Machine)

Để hiểu rõ tại sao Docker lại nhẹ và nhanh đến vậy, chúng ta cần so sánh nó với `virtual machine - máy ảo` (VM) truyền thống. Sự khác biệt cốt lõi nằm ở cách thức quản lý tài nguyên và chia sẻ nhân hệ điều hành.

Virtual Machine (Máy ảo): Chạy trên nền tảng của một `hypervisor - trình quản lý máy ảo` (ví dụ: VMware, VirtualBox, Hyper-V). Mỗi VM yêu cầu một hệ điều hành khách (Guest OS) riêng biệt, hoàn chỉnh với nhân hệ điều hành, trình điều khiển thiết bị (drivers), thư viện và toàn bộ các tiến trình hệ thống đi kèm. Điều này khiến VM rất nặng (thường từ vài GB đến vài chục GB) và khởi động rất chậm.

Docker Container (Vùng chạy cô lập): Chạy trực tiếp trên `host operating system - hệ điều hành máy chủ` và chia sẻ chung `kernel - nhân` của hệ điều hành máy chủ. Các container được cô lập với nhau thông qua các tính năng có sẵn trong nhân Linux như namespaces (để cô lập không gian tên tiến trình, mạng, người dùng) và control groups (cgroups - để giới hạn tài nguyên CPU, RAM). Nhờ vậy, container cực kỳ nhẹ (chỉ vài MB đến vài trăm MB) và khởi động gần như ngay lập tức.

Bảng so sánh chi tiết

Đặc tính Virtual Machine (Máy ảo) Docker Container (Vùng chạy cô lập)
Kiến trúc (Architecture) Mỗi VM có hệ điều hành khách (Guest OS) đầy đủ và chạy trên Hypervisor. Chia sẻ chung nhân hệ điều hành máy chủ (Host OS kernel). Không cần Guest OS.
Hiệu năng (Performance) Hiệu năng thấp hơn do hao phí tài nguyên chạy hệ điều hành khách. Hiệu năng cao, gần như tương đương với ứng dụng chạy native trên máy chủ.
Dung lượng (Storage) Nặng (vài GB đến hàng chục GB) cho mỗi máy ảo. Rất nhẹ (vài MB đến vài trăm MB) do chỉ chứa ứng dụng và thư viện cần thiết.
Thời gian khởi động Chậm (vài chục giây đến vài phút) để nạp xong hệ điều hành. Nhanh (vài phần mười giây hoặc vài mili giây).
Khả năng cô lập Cô lập hoàn toàn ở cấp độ phần cứng thông qua Hypervisor (rất an toàn). Cô lập ở cấp độ tiến trình thông qua Kernel Namespaces & Cgroups (an toàn cao).

Chương 4: Hệ sinh thái Docker (Docker Ecosystem)

Docker không chỉ là một công cụ đơn lẻ mà là cả một hệ sinh thái gồm nhiều thành phần phối hợp chặt chẽ với nhau để tạo nên một quy trình quản lý container hoàn chỉnh:

1. Docker Engine (Runtime Cốt Lõi) ▼

Bộ công cụ runtime cốt lõi chạy trực tiếp trên hệ điều hành máy chủ, chịu trách nhiệm xây dựng (build), chạy (run) và phân phối các container.

2. Docker Desktop (Ứng Dụng GUI) ▼

Ứng dụng giao diện đồ họa thân thiện, dễ cài đặt trên Windows và macOS. Nó tích hợp sẵn Docker Engine, Docker CLI, Docker Compose, Kubernetes, và một giao diện trực quan để quản lý container, image, volume và network mà không cần gõ lệnh.

3. Docker CLI (Giao Diện Dòng Lệnh) ▼

Công cụ dòng lệnh giúp lập trình viên giao tiếp và gửi yêu cầu điều khiển đến Docker Daemon thông qua REST API.

4. Docker Hub & Registry (Kho Lưu Trữ Image) ▼

`registry (kho lưu trữ)` công cộng được lưu trữ trên đám mây bởi chính Docker. Đây là nơi chứa hàng triệu image có sẵn từ hệ điều hành (Ubuntu, Alpine), cơ sở dữ liệu (PostgreSQL, MongoDB, Redis) đến các môi trường lập trình (Python, Node.js, Go). Ngoài ra, các tổ chức có thể tự dựng các private registry riêng tư để bảo mật thông tin.

5. Docker Compose (Quản Lý Đa Container) ▼

Công cụ định nghĩa và chạy các ứng dụng Docker đa container. Bằng cách sử dụng một tệp tin cấu hình YAML duy nhất với tên hiện đại (modern name) là compose.yaml (hoặc tên cũ legacy name là docker-compose.yml), bạn có thể khởi chạy toàn bộ hệ thống gồm web server (máy chủ web), database (cơ sở dữ liệu), cache (bộ nhớ đệm) và hàng loạt dịch vụ khác chỉ bằng một câu lệnh duy nhất.

Chương 5: Cài đặt Docker trên Windows 11 & Tích hợp WSL2 Ubuntu

Để tối ưu hóa hiệu năng phát triển trên Windows, chúng ta sử dụng WSL 2 làm engine vận hành chính cho Docker thay vì giải pháp ảo hóa Hyper-V truyền thống. Dưới đây là các bước thiết lập và tối ưu hệ thống chi tiết.

Các bước thiết lập và kích hoạt WSL2 Integration

  1. Kích hoạt WSL 2 trên Windows: Mở PowerShell với quyền Administrator và chạy câu lệnh wsl --install. Khởi động lại máy tính khi được yêu cầu. Cài đặt hệ phân phối Ubuntu từ Microsoft Store.
  2. Cài đặt Docker Desktop: Tải bộ cài đặt từ trang chủ Docker. Trong quá trình cài đặt, hãy tích chọn mục Use WSL 2 instead of Hyper-V (recommended).
  3. Kích hoạt WSL 2 Integration:
    • Mở ứng dụng Docker Desktop trên Windows.
    • Vào Settings (biểu tượng bánh răng) > General > đảm bảo đã tích chọn Use the WSL 2 based engine.
    • Vào Resources > WSL integration > Kích hoạt công tắc cho bản phân phối Ubuntu của bạn (ví dụ Ubuntu). Nhấp Apply & Restart.

Cấu hình giới hạn tài nguyên hệ thống thông qua .wslconfig

Theo mặc định, WSL2 có thể tiêu tốn tới 75% lượng bộ nhớ RAM của máy chủ Windows để làm bộ đệm, dẫn đến tình trạng máy chủ bị đơ, giật lag khi Docker chạy các tác vụ nặng. Để khắc phục điều này, ta cần tạo tệp cấu hình .wslconfig trong thư mục người dùng Windows (ví dụ: C:\Users\Username\.wslconfig).

ini 
[wsl2]
memory=8GB
processors=4
swap=2GB
localhostForwarding=true

Đoạn mã trên đại diện cho cấu hình hệ thống giới hạn WSL2 chỉ sử dụng tối đa 8GB bộ nhớ RAM và 4 nhân CPU từ máy chủ vật lý, giúp ngăn ngừa tình trạng cạn kiệt tài nguyên hệ thống trên hệ điều hành Windows.

Nguyên tắc vàng về tốc độ nhập/xuất (Crucial I/O Best Practice)

Tuyệt đối KHÔNG lưu trữ mã nguồn dự án của bạn trên hệ thống tệp Windows (ví dụ ở /mnt/c/Users/...) rồi mount vào Docker container chạy trên WSL2. Do sự khác biệt về kiến trúc hệ thống tệp giữa NTFS (Windows) và ext4 (Linux), việc đọc/ghi dữ liệu xuyên biên giới thông qua giao thức 9p sẽ khiến `input/output speed - tốc độ nhập/xuất` dữ liệu giảm tới 10-20 lần!

Giải pháp: Luôn lưu dự án trong hệ thống tệp gốc của Linux trong WSL2 (ví dụ tại thư mục cá nhân ~/projects/). Bạn vẫn có thể chỉnh sửa mã nguồn này từ Windows bằng cách mở VS Code và cài đặt tiện ích mở rộng WSL, sau đó gõ code . ngay trong terminal của Ubuntu.

Chương 6: Quy trình làm việc với Container đầu tiên (First Container Workflow)

Chúng ta sẽ thực hành chạy container đầu tiên của mình thông qua CLI của WSL2 Ubuntu và tìm hiểu chi tiết những gì diễn ra bên dưới hệ thống.

Các câu lệnh thao tác cơ bản

Hãy mở terminal Ubuntu trong WSL2 và thực thi các câu lệnh sau:

bash 
# 1. Chạy container hello-world để kiểm tra cài đặt
docker run hello-world

# 2. Xem danh sách các container đang hoạt động
docker ps

# 3. Xem toàn bộ danh sách các container (bao gồm cả các container đã dừng)
docker ps -a

# 4. Dừng một container đang chạy
docker stop [container_id_or_name]

# 5. Xóa một container đã dừng khỏi hệ thống
docker rm [container_id_or_name]

Các câu lệnh trên lần lượt thực hiện việc tải và chạy container thử nghiệm, liệt kê danh sách container đang chạy và đã dừng, dừng hoạt động của một container và xóa container đó khỏi bộ nhớ đĩa.

Cơ chế hoạt động bên trong (Internal Mechanics) của docker run

Khi bạn gõ lệnh docker run hello-world trong terminal, một chuỗi các sự kiện phức tạp sẽ diễn ra ở hậu trường:

  1. Yêu cầu từ Client: Docker CLI dịch câu lệnh của bạn và gửi một yêu cầu API REST đến Docker Daemon (dockerd) đang chạy ngầm trên máy chủ.
  2. Kiểm tra cục bộ: Docker Daemon nhận yêu cầu tạo container từ image hello-world. Nó sẽ tìm kiếm trong local cache (kho lưu trữ cục bộ) xem đã có image nào tên là hello-world:latest chưa.
  3. Tải về từ Registry: Nếu không tìm thấy ở local, Docker Daemon sẽ tự động kết nối tới Docker Hub để tải (pull) các lớp tệp tin của image hello-world về máy.
  4. Tạo và Cấp phát: Sau khi tải xong image, Daemon sẽ tạo một `container (vùng chạy cô lập)` mới từ image này. Nó cấp phát một hệ thống tệp tin ghi được (read-write layer) phía trên các lớp read-only của image, đồng thiết lập môi trường mạng cô lập và cấp phát IP nội bộ cho container.
  5. Khởi chạy tiến trình: Daemon khởi chạy tiến trình chính của container. Trong trường hợp của hello-world, tiến trình này chỉ đơn giản là in ra một đoạn văn bản chào mừng lên màn hình stdout.
  6. Chuyển tiếp kết quả: Docker Daemon thu thập luồng dữ liệu đầu ra từ container và chuyển tiếp ngược lại cho Docker CLI để hiển thị lên màn hình terminal của bạn.
  7. Kết thúc vòng đời: Khi tiến trình in văn bản hoàn thành, tiến trình chính kết thúc. Container tự động chuyển sang trạng thái Stopped (Đã dừng) nhưng dữ liệu của container vẫn tồn tại trên ổ đĩa cho đến khi bạn chạy lệnh docker rm.

Chương 7: Kiến thức cơ bản về Image và Container (Image & Container Fundamentals)

Để làm việc hiệu quả với Docker, bạn cần nắm vững các thao tác quản lý image và container hàng ngày, đồng thời phân biệt được những nhóm lệnh thường dùng và ít dùng.

Các câu lệnh quản lý hàng ngày

Dưới đây là tập hợp các lệnh cơ bản giúp bạn làm việc trực tiếp với image và container:

bash 
# Tìm kiếm image trên Docker Hub
docker search python

# Tải image về máy cục bộ
docker pull python:3.10-slim

# Chạy container với các cờ (flags) quan trọng
docker run -it -d --name my-python-app --rm python:3.10-slim bash

# Thực thi lệnh mới bên trong container đang chạy
docker exec -it my-python-app pip install numpy

# Xem nhật ký (logs) xuất ra của container
docker logs -f my-python-app

# Kiểm tra siêu dữ liệu (metadata) chi tiết của container
docker inspect my-python-app

# Đặt nhãn mới (tag) cho image
docker tag python:3.10-slim my-registry.com/my-python:v1

Các lệnh trên thể hiện đầy đủ quy trình từ tìm kiếm, tải image, khởi chạy container ở `detached mode - chế độ chạy ngầm` với các cờ cấu hình terminal, thực thi lệnh bổ sung ở `interactive mode - chế độ tương tác`, theo dõi nhật ký, kiểm tra cấu hình và đặt thẻ phân bản cho image.

Giải thích chi tiết các cờ (flags) của lệnh docker run:

  • -i (--interactive): Giữ luồng nhập chuẩn (stdin) mở để bạn có thể gửi dữ liệu nhập vào container.
  • -t (--tty): Cấp phát một terminal ảo (pseudo-TTY) giúp giao diện terminal hiển thị đẹp mắt, hỗ trợ màu sắc và các phím tắt.
  • -d (--detach): Chạy container ở chế độ chạy ngầm, trả lại quyền điều khiển terminal cho bạn ngay lập tức.
  • --name: Đặt tên dễ nhớ cho container thay vì để Docker tự sinh tên ngẫu nhiên.
  • -h (--hostname): Thiết lập tên máy chủ (hostname) bên trong container.
  • --rm: Tự động xóa sạch container (bao gồm cả phân vùng ghi được) ngay sau khi container dừng hoạt động, tránh lãng phí ổ cứng.

Các lệnh ít khi sử dụng (Rarely-used Commands)

Có một số lệnh mặc dù tồn tại trong Docker nhưng rất ít khi được dùng trong quy trình làm việc hiện đại:

  • docker commit [container_id] [new_image_name]: Tạo một image mới từ những thay đổi hiện tại của một container đang chạy.
  • docker save -o [file_name].tar [image_name]: Xuất một image ra thành tệp lưu trữ nén định dạng .tar.
  • docker load -i [file_name].tar: Nhập một image vào local từ một tệp .tar có sẵn.
Tại sao Dockerfile luôn được ưu tiên hơn docker commit?

Trong thực tế phát triển chuyên nghiệp, việc dùng docker commit để tạo image bị coi là một anti-pattern vì những lý do sau:

  1. Không thể tái lập (No Reproducibility): Image tạo ra từ commit là một "hộp đen". Không ai biết chính xác bạn đã cài đặt những gì, cấu hình ra sao bên trong container trước khi commit.
  2. Dung lượng khổng lồ (Image Bloat): Việc sửa đổi và commit thủ công thường tạo ra nhiều lớp tệp tin rác không cần thiết, làm tăng kích thước image lên gấp nhiều lần.
  3. Thiếu quản lý phiên bản (No Version Control): Bạn không thể đưa quy trình tạo image thủ công vào Git để theo dõi lịch sử thay đổi hoặc tích hợp vào hệ thống CI/CD tự động.

Thay vào đó, lập trình viên sử dụng Dockerfile - một tệp cấu hình dạng văn bản thuần chứa danh sách các câu lệnh tuần tự để xây dựng image. Dockerfile giúp quy trình xây dựng image trở nên minh bạch, có thể đưa vào Git, tự động hóa 100% và tối ưu hóa kích thước nhờ cơ chế cache từng lớp.

Chương 8: Cổng, Tiến trình và Vòng đời Container (Ports, Processes, and Lifecycle)

Để vận hành container trong môi trường mạng và quản lý chúng hiệu quả, bạn cần hiểu sâu về cách ánh xạ cổng, cơ chế cô lập tiến trình của hệ điều hành và các trạng thái vòng đời của một container.

Port Mapping (Ánh xạ cổng) chi tiết

Mặc định, các container chạy trong một mạng ảo cô lập và không thể truy cập trực tiếp từ máy chủ Windows hoặc mạng bên ngoài. Để cho phép lưu lượng mạng đi vào container, chúng ta phải sử dụng tính năng `port mapping - ánh xạ cổng` thông qua tham số -p host_port:container_port.

bash 
# Ánh xạ cổng 8080 của máy chủ vào cổng 80 bên trong container Nginx
docker run -d -p 8080:80 --name my-web-server nginx

Khi người dùng truy cập địa chỉ http://localhost:8080 trên trình duyệt web của Windows, Docker Engine sẽ nhận tín hiệu tại cổng 8080 của máy chủ và sử dụng cơ chế định tuyến mạng để chuyển tiếp toàn bộ lưu lượng này tới cổng 80 bên trong container Nginx đang chạy.

Cô lập tiến trình (Process Isolation)

Mỗi container Docker thực chất chỉ là một nhóm các tiến trình chạy trực tiếp trên `kernel - nhân` Linux của WSL2, nhưng bị cô lập thông qua các cơ chế cốt lõi của nhân Linux:

  • Namespaces: Giúp container sở hữu không gian tên riêng độc lập. Nó có bảng tiến trình riêng (PID namespace bắt đầu từ 1), card mạng riêng (NET namespace), hệ thống tệp tin riêng (MNT namespace).
  • Control Groups (cgroups): Giới hạn phần cứng. Nếu container chạy vượt quá giới hạn tài nguyên bộ nhớ RAM được cấp phép, nhân Linux sẽ kích hoạt bộ giết tiến trình (OOM Killer) để chấm dứt container đó ngay lập tức nhằm bảo vệ sự ổn định của hệ thống máy chủ.

Vòng đời của Container (Container Lifecycle Transitions)

Một container di chuyển qua nhiều trạng thái khác nhau trong suốt `container lifecycle - vòng đời của vùng chạy` của nó:

Các trạng thái chính trong vòng đời:
  • Created (Đã tạo): Container đã được khởi tạo từ image (cấp phát hệ thống tệp tin ghi được, cấu hình mạng) nhưng tiến trình chính chưa chạy. Lệnh kích hoạt: docker create.
  • Running (Đang hoạt động): Tiến trình chính của container đang chạy tích cực. Lệnh kích hoạt: docker start hoặc docker run.
  • Stopped (Đã dừng): Tiến trình chính nhận được tín hiệu kết thúc (SIGTERM/SIGKILL) và dừng lại. RAM được giải phóng nhưng dữ liệu trên đĩa vẫn được giữ lại. Lệnh kích hoạt: docker stop.
  • Exited (Đã thoát): Container đã dừng do tiến trình chính tự kết thúc (hoàn thành công việc hoặc gặp lỗi).
  • Paused (Tạm ngưng): Toàn bộ tiến trình bị đóng băng trạng thái CPU. Container không tiêu thụ CPU nhưng vẫn giữ nguyên dung lượng RAM. Lệnh kích hoạt: docker pause / docker unpause.
  • Deleted (Đã xóa): Container bị hủy bỏ hoàn toàn khỏi ổ đĩa. Lệnh kích hoạt: docker rm.

Chương 9: Dockerfile và Image Layers (Dockerfile & Image Layers)

Để tự động hóa quy trình đóng gói ứng dụng, Docker sử dụng Dockerfile - tệp công thức build image. Đây là một tệp văn bản chứa các chỉ thị tuần tự để xây dựng một image (ảnh đóng gói) hoàn chỉnh.

Khi Docker build một image, mỗi chỉ thị trong Dockerfile sẽ tạo ra một image layers - các lớp của image. Các lớp này được xếp chồng lên nhau và có tính chất bất biến (layer immutability). Khi một lớp đã được tạo ra, nó không bao giờ thay đổi. Nếu bạn sửa đổi mã nguồn hoặc cấu hình và build lại, Docker sẽ tái sử dụng các lớp cũ không thay đổi từ bộ nhớ đệm xây dựng (building cache), giúp tăng tốc độ build đáng kể.

Cơ chế Layer Immutability & Building Cache

Docker sử dụng hệ thống tệp tin dạng Union File System (UnionFS). Các image layers - các lớp của image là các lớp chỉ đọc (read-only layers). Khi chạy một container (vùng chạy cô lập), Docker chỉ thêm một lớp mỏng cho phép ghi (read-write layer) ở trên cùng.

Trong quá trình build, nếu một chỉ thị không thay đổi và các tệp tin liên quan (được copy) không có sự biến động, Docker sẽ dùng lại lớp cache. Tuy nhiên, nếu một lớp bị thay đổi (ví dụ: code thay đổi làm chỉ thị COPY phát hiện sự khác biệt), toàn bộ các lớp phía sau nó sẽ bị mất cache và phải chạy lại từ đầu. Do đó, quy tắc vàng là đặt các chỉ thị ít thay đổi (như cài đặt thư viện) ở phía trên và các chỉ thị hay thay đổi (như copy mã nguồn) ở dưới cùng.

Ví dụ Dockerfile hoàn chỉnh cho Python FastAPI

Dưới đây là một cấu trúc Dockerfile tối ưu cho ứng dụng FastAPI sử dụng python:3.10-slim làm base image:

dockerfile 
# Use an official Python runtime as a parent image
FROM python:3.10-slim

# Set the working directory in the container
WORKDIR /app

# Set environment variables to prevent Python from writing pyc files and buffering stdout/stderr
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# Define build-time argument for app version
ARG APP_VERSION=1.0.0

# Expose the port the app runs on
EXPOSE 8000

# Install system dependencies (build-essential for compiling C extensions if any)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

# Copy only the requirements file first to leverage build cache
COPY requirements.txt .

# Install Python packages
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application code
COPY . .

# Set a mount point for app logs
VOLUME /app/logs

# Define default command to run the application using Uvicorn
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Đoạn mã trên mô tả quy trình xây dựng image cho ứng dụng Python FastAPI, bắt đầu từ việc khai báo base image siêu nhẹ, thiết lập thư mục làm việc, cài đặt thư viện hệ thống và thư viện Python, sau đó sao chép toàn bộ mã nguồn và thiết lập câu lệnh khởi động ứng dụng web.

Giải thích chi tiết các chỉ thị Dockerfile chính:

  • FROM: Chỉ định base image (ảnh đóng gói cơ sở) khởi đầu cho quá trình build. Mọi Dockerfile đều phải bắt đầu bằng lệnh này.
  • WORKDIR: Thiết lập thư mục làm việc mặc định bên trong container (vùng chạy cô lập). Nếu thư mục chưa tồn tại, Docker sẽ tự động tạo ra.
  • COPY: Sao chép tệp tin hoặc thư mục từ hệ điều hành máy chủ vào trong hệ thống tệp tin của container (vùng chạy cô lập).
  • RUN: Thực thi một câu lệnh trong quá trình build (ví dụ cài đặt gói phần mềm). Lệnh này sẽ tạo ra một lớp image mới chứa kết quả thực thi.
  • CMD: Chỉ định câu lệnh mặc định sẽ chạy khi container (vùng chạy cô lập) được khởi động. Chỉ có thể có một lệnh CMD trong Dockerfile, nếu khai báo nhiều thì lệnh cuối cùng sẽ đè lên các lệnh trước.
  • ENTRYPOINT: Cấu hình container chạy như một file thực thi. Các tham số truyền vào từ CLI sau lệnh chạy container sẽ được nối tiếp vào sau chỉ thị ENTRYPOINT.
  • EXPOSE: Khai báo cổng mà container sẽ lắng nghe khi chạy. Đây chỉ là một chỉ dẫn mang tính tài liệu (metadata), không tự động mở cổng ra máy host nếu không dùng cờ ánh xạ cổng.
  • ENV: Thiết lập các biến môi trường bên trong container. Các biến này có giá trị tồn tại cả trong lúc build lẫn lúc container đang chạy.
  • ARG: Định nghĩa các biến số chỉ được sử dụng trong quá trình build image. Người dùng có thể truyền giá trị cho biến này thông qua cờ --build-arg khi chạy lệnh build.
  • VOLUME: Tạo ra một điểm gắn kết (mount point) với hệ thống tệp tin bên ngoài để lưu trữ dữ liệu bền vững, độc lập với vòng đời của container.

Chương 10: Lưu trữ Dữ liệu Bền vững: Volumes và Bind Mounts

Theo mặc định, hệ thống lưu trữ bên trong một container (vùng chạy cô lập) là hệ thống lưu trữ tạm thời (ephemeral storage). Khi container bị xóa, toàn bộ dữ liệu được ghi trên lớp ghi (read-write layer) của nó sẽ biến mất vĩnh viễn. Để lưu trữ dữ liệu bền vững, Docker cung cấp hai giải pháp chính: named volume - vùng lưu dữ liệu do Docker quản lý và bind mount - ánh xạ thư mục thật từ host.

Cực kỳ hữu ích trong development environment - môi trường phát triển để mount mã nguồn từ máy host vào container, giúp áp dụng cơ chế cập nhật nóng (hot-reloading).

Các câu lệnh quản lý Volume

Mở terminal WSL2 và sử dụng các lệnh sau để quản lý các volume trong hệ thống:

bash 
# 1. Liệt kê toàn bộ các volume đang có trên hệ thống
docker volume ls

# 2. Tạo một volume mới với tên tự chọn
docker volume create my-db-data

# 3. Xem thông tin cấu hình và đường dẫn chi tiết của volume
docker volume inspect my-db-data

# 4. Xóa một volume cụ thể (volume phải không được sử dụng bởi container nào)
docker volume rm my-db-data

# 5. Dọn dẹp và xóa sạch tất cả các volume không sử dụng để giải phóng đĩa
docker volume prune

Hệ lệnh trên cho phép kiểm tra danh sách vùng dữ liệu, khởi tạo vùng dữ liệu mới, truy xuất cấu hình đĩa chi tiết và dọn dẹp các phân vùng thừa để tối ưu bộ nhớ.

Cú pháp Mount dữ liệu: -v so với --mount

Để gắn kết dữ liệu vào container khi khởi chạy, Docker hỗ trợ hai cú pháp: -v (hoặc --volume) và --mount.

  • Cú pháp -v (viết tắt): Kết hợp tất cả cấu hình trong một chuỗi ký tự phân cách bằng dấu hai chấm (ví dụ: -v my-vol:/app/data). Cú pháp này ngắn gọn nhưng dễ gây nhầm lẫn nếu đường dẫn chứa ký tự đặc biệt, và nếu volume chưa tồn tại Docker sẽ tự động tạo một thư mục trống cho bạn.
  • Cú pháp --mount (khuyên dùng): Sử dụng các cặp khóa-giá trị (key-value) phân tách bằng dấu phẩy, giúp cú pháp trở nên tường minh, dễ đọc và dễ gỡ lỗi hơn. Nếu volume chỉ định không tồn tại, Docker sẽ báo lỗi ngay lập tức thay vì âm thầm tạo thư mục rác.
bash 
# Khởi chạy container PostgreSQL sử dụng cú pháp --mount (Khuyên dùng)
docker run -d \
  --name production-postgres \
  --mount type=volume,source=my-db-data,target=/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=my_secure_password \
  postgres:15-alpine

# Khởi chạy container sử dụng cú pháp -v truyền thống
docker run -d \
  --name development-nginx \
  -v /home/user/my-project:/usr/share/nginx/html \
  nginx:alpine

Khối mã trên thể hiện hai cách thức mount dữ liệu: sử dụng --mount tường minh để liên kết volume hệ thống vào container PostgreSQL và sử dụng -v để ánh xạ trực tiếp thư mục mã nguồn từ máy host vào container web server Nginx.

Chương 11: Mạng trong Docker (Docker Networking)

Docker cung cấp cơ chế ảo hóa network (mạng) để các container có thể giao tiếp an toàn với nhau hoặc kết nối ra internet bên ngoài. Hiểu rõ kiến trúc mạng giúp bạn cấu hình chính xác cách kết nối giữa các thành phần ứng dụng.

Các Driver Mạng mặc định (Network Drivers)

  • Bridge Network (Mạng cầu): Đây là driver mạng mặc định. Khi bạn khởi chạy container mà không chỉ định mạng, nó sẽ kết nối vào mạng bridge mặc định. Các container trong mạng bridge này có thể kết nối internet nhưng chỉ có thể giao tiếp trực tiếp với nhau thông qua địa chỉ IP ảo nội bộ (thường có dạng 172.17.x.x).
  • Host Network (Mạng máy chủ): Gỡ bỏ toàn bộ sự cô lập về mạng giữa container và máy host. Container sẽ sử dụng trực tiếp card mạng và các cổng của máy host (ví dụ: container chạy ứng dụng lắng nghe cổng 80 sẽ xuất trực tiếp trên cổng 80 của máy host mà không cần cấu hình ánh xạ cổng).
  • None Network (Không dùng mạng): Cô lập hoàn toàn container khỏi mọi kết nối mạng. Container sẽ không có card mạng ngoài loopback (localhost), thích hợp cho các tác vụ tính toán offline cần độ bảo mật cực cao.
Sức mạnh của Custom Bridge Network & DNS Server tích hợp

Khi bạn tự tạo một mạng cầu tùy chỉnh (custom bridge network), Docker sẽ kích hoạt một DNS server nội bộ. DNS server này cho phép các container tự động nhận diện và phân giải tên miền (DNS resolution) dựa trên tên của container (container name) thay vì địa chỉ IP ảo.

Ví dụ: Bạn có container ứng dụng tên là web-app và container cơ sở dữ liệu tên là db-server cùng tham gia mạng app-net. Từ bên trong container web-app, bạn có thể kết nối trực tiếp đến database thông qua chuỗi kết nối postgresql://db-server:5432 mà không cần biết IP của database là gì.

Các câu lệnh quản lý Mạng

Sử dụng các câu lệnh sau để giám sát và thiết lập cấu hình mạng ảo:

bash 
# 1. Liệt kê tất cả các mạng ảo đang tồn tại
docker network ls

# 2. Tạo một mạng cầu ảo tùy chỉnh mới
docker network create my-custom-network

# 3. Xem chi tiết thông tin mạng (bao gồm danh sách container đang kết nối)
docker network inspect my-custom-network

# 4. Xóa một mạng ảo tùy chỉnh khỏi hệ thống
docker network rm my-custom-network

Các lệnh trên giúp lập trình viên kiểm tra danh sách kết nối mạng, tạo phân vùng mạng độc lập, giám sát các container trong mạng và xóa bỏ các cấu hình mạng không còn sử dụng.

Ví dụ thực tế: Kết nối 2 container trên một Custom Network

Chúng ta sẽ thực hiện khởi chạy một container cơ sở dữ liệu PostgreSQL và một container ứng dụng Python kết nối với nhau thông qua mạng cầu tùy chỉnh:

bash 
# Bước 1: Tạo mạng cầu ảo tùy chỉnh tên là 'app-bridge'
docker network create app-bridge

# Bước 2: Chạy container cơ sở dữ liệu PostgreSQL kết nối vào mạng 'app-bridge'
docker run -d \
  --name db-container \
  --network app-bridge \
  -e POSTGRES_USER=my_user \
  -e POSTGRES_PASSWORD=my_secret_password \
  -e POSTGRES_DB=app_db \
  postgres:15-alpine

# Bước 3: Chạy container ứng dụng Python kết nối vào cùng mạng 'app-bridge'
# Lưu ý: Ứng dụng kết nối tới db qua host là 'db-container'
docker run -it --rm \
  --name app-container \
  --network app-bridge \
  python:3.10-slim \
  python -c "import socket; print('Connected successfully!' if socket.create_connection(('db-container', 5432), timeout=5) else 'Failed')"

Đoạn script trên tạo ra mạng cầu ảo tùy chỉnh, khởi chạy database Postgres trong mạng này, và sau đó chạy một container Python tạm thời để kiểm tra kết nối mạng đến Postgres bằng cách phân giải trực tiếp hostname db-container.

Chương 12: Ứng dụng đa Container (Multi-container Applications)

Trong kiến trúc phần mềm hiện đại, việc đóng gói toàn bộ hệ thống (gồm mã nguồn ứng dụng, cơ sở dữ liệu, bộ nhớ đệm, hàng đợi công việc) vào trong một container (vùng chạy cô lập) duy nhất là một sai lầm nghiêm trọng (anti-pattern). Thay vào đó, chúng ta áp dụng mô hình thiết kế dịch vụ nhỏ tách biệt (decoupled microservices pattern).

Nguyên tắc: Mỗi container chỉ chạy một tiến trình chính duy nhất

Triết lý cốt lõi của Docker là "một container chịu trách nhiệm cho một tiến trình chính duy nhất" (one main process per container). Điều này mang lại những lợi ích vượt trội:

  • Cô lập lỗi (Fault Isolation): Nếu tiến trình ứng dụng web gặp lỗi rò rỉ bộ nhớ hoặc bị sập, tiến trình cơ sở dữ liệu chạy ở container khác vẫn hoàn toàn hoạt động bình thường, bảo toàn dữ liệu tránh bị hỏng hóc.
  • Khả năng co giãn độc lập (Horizontal Scaling): Khi lượng truy cập tăng cao, bạn có thể dễ dàng nhân bản (scale) ứng dụng web thành 5 hay 10 container chạy song song mà không cần phải nhân bản cơ sở dữ liệu - thành phần vốn tiêu tốn rất nhiều tài nguyên phần cứng.
  • Tái sử dụng Image linh hoạt: Bạn có thể sử dụng các image cơ sở dữ liệu chuẩn hóa do nhà phát hành cung cấp trực tiếp trên Docker Hub (như PostgreSQL, Redis, Nginx) thay vì phải tự viết script cài đặt thủ công các dịch vụ này chồng chéo lên nhau.
  • Quản lý nhật ký rõ ràng (Clean Logging): Khi mỗi container chỉ có một tiến trình chính (PID 1), luồng stdout/stderr của container đó sẽ chỉ phản ánh duy nhất hoạt động của tiến trình này, giúp công tác theo dõi và xử lý lỗi (troubleshooting) cực kỳ dễ dàng.

Mặc dù việc phân tách các thành phần thành các container độc lập là tối ưu, thách thức đặt ra là làm thế nào để quản lý vòng đời, cấu hình mạng và lưu trữ của hàng loạt container này một cách đồng bộ. Việc gõ hàng chục dòng lệnh docker run thủ công cho mỗi lần khởi động hệ thống là không khả thi. Để giải quyết bài toán này, Docker cung cấp công cụ Docker Compose.

Chương 13: Docker Compose - Quản lý Phối hợp Đa Container

Docker Compose là công cụ giúp bạn định nghĩa và khởi chạy hệ thống ứng dụng gồm nhiều container (vùng chạy cô lập) thông qua một tệp cấu hình duy nhất viết bằng ngôn ngữ YAML. Phiên bản hiện đại của Docker khuyến nghị sử dụng tên tệp là compose.yaml thay vì định dạng cũ docker-compose.yml.

Các thành phần cốt lõi của compose.yaml

  • services (dịch vụ): Định nghĩa các container mà bạn muốn khởi chạy. Mỗi dịch vụ sẽ chứa các thông tin cấu hình như build image, ports, environments, volumes và networks.
  • ports (cổng): Thiết lập cấu hình port mapping - ánh xạ cổng từ máy host vào container.
  • environment (biến môi trường): Khai báo các biến môi trường sẽ được truyền vào container khi khởi chạy.
  • volumes (vùng dữ liệu): Khai báo các điểm gắn kết lưu trữ dữ liệu bền vững.
  • networks (mạng ảo): Khấu hình các mạng cầu ảo chung để các dịch vụ có thể liên lạc với nhau.

Tệp compose.yaml hoàn chỉnh liên kết FastAPI và PostgreSQL

Dưới đây là một tệp cấu hình compose.yaml thực chiến, tích hợp ứng dụng FastAPI (từ Chương 9) và cơ sở dữ liệu PostgreSQL sử dụng volume bền vững và mạng cầu ảo:

yaml 
services:
  # Dịch vụ ứng dụng FastAPI (FastAPI Web Application service)
  web-app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: fastapi-web-app
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://postgres_user:secure_pwd@db-service:5432/app_db
    depends_on:
      db-service:
        condition: service_healthy
    networks:
      - app-network
    volumes:
      - .:/app

  # Dịch vụ cơ sở dữ liệu PostgreSQL (PostgreSQL Database service)
  db-service:
    image: postgres:15-alpine
    container_name: postgres-database
    environment:
      - POSTGRES_USER=postgres_user
      - POSTGRES_PASSWORD=secure_pwd
      - POSTGRES_DB=app_db
    ports:
      - "5432:5432"
    volumes:
      - db-volume:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres_user -d app_db"]
      interval: 5s
      timeout: 5s
      retries: 5
    networks:
      - app-network

# Định nghĩa volume lưu trữ bền vững ngoài container (Persistent volume definition)
volumes:
  db-volume:
    name: pg_app_data

# Định nghĩa mạng cầu ảo nội bộ (Internal bridge network definition)
networks:
  app-network:
    driver: bridge

Tệp tin YAML trên định nghĩa hệ thống gồm hai dịch vụ: dịch vụ web-app được tự động build từ mã nguồn local và lắng nghe cổng 8000, dịch vụ db-service sử dụng ảnh đóng gói PostgreSQL chính thức, cấu hình phân vùng dữ liệu bền vững và kết nối chung qua một mạng cầu ảo.

Giải thích chi tiết từng dòng cấu hình (Line-by-line Annotations)

  • services:: Bắt đầu khai báo danh sách các container thành phần trong hệ thống.
  • web-app:: Tên của dịch vụ thứ nhất (ứng dụng FastAPI). Tên này cũng đóng vai trò là một hostname nội bộ.
  • build:: Cấu hình build image từ Dockerfile cục bộ. context: . nghĩa là thư mục hiện tại, và dockerfile: Dockerfile là tên file công thức.
  • ports: - "8000:8000": Ánh xạ cổng 8000 trên máy host vào cổng 8000 của container web-app.
  • environment:: Khai báo biến môi trường. Biến DATABASE_URL sử dụng hostname db-service để kết nối tới database.
  • depends_on: db-service: condition: service_healthy:

    Cơ chế quan trọng: Mặc định, depends_on chỉ đảm bảo Docker Engine sẽ khởi chạy container db-service trước container web-app về mặt tiến trình, chứ không đảm bảo cơ sở dữ liệu bên trong đã hoàn tất khởi tạo và sẵn sàng (ready) nhận kết nối. Điều này thường dẫn đến việc ứng dụng web bị lỗi kết nối và sập ngay khi khởi động.

    Bằng cách sử dụng cú pháp mở rộng với điều kiện condition: service_healthy kết hợp với cấu hình healthcheck trong db-service, Docker Compose sẽ đảm bảo container web-app chỉ thực sự được khởi chạy khi cơ sở dữ liệu đã báo cáo trạng thái hoàn toàn sẵn sàng nhận kết nối.

  • volumes: - .:/app: Thực hiện bind mount - ánh xạ thư mục thật từ host từ thư mục hiện tại trên host vào thư mục /app trong container, giúp đồng bộ mã nguồn để phát triển hot-reloading.
  • db-service:: Tên dịch vụ thứ hai (PostgreSQL).
  • image: postgres:15-alpine: Thay vì build từ Dockerfile, Docker sẽ tải trực tiếp image Postgres siêu nhẹ từ Docker Hub.
  • volumes: - db-volume:/var/lib/postgresql/data: Mount named volume - vùng lưu dữ liệu do Docker quản lý tên là db-volume vào thư mục lưu trữ dữ liệu mặc định của Postgres bên trong container để bảo toàn dữ liệu database.
  • volumes: db-volume: name: pg_app_data: Khai báo với Docker Engine tạo và quản lý một volume vật lý bền vững có tên là pg_app_data.
  • networks: app-network: driver: bridge: Khởi tạo mạng cầu ảo tùy chỉnh tên là app-network sử dụng driver bridge mặc định.
Các câu lệnh điều khiển Docker Compose cơ bản

Mở terminal tại thư mục chứa file compose.yaml và thực thi:

  • docker compose up -d: Khởi tạo và chạy toàn bộ hệ thống ở chế độ chạy ngầm (detached mode).
  • docker compose ps: Xem trạng thái hoạt động của các container do compose quản lý.
  • docker compose logs -f: Theo dõi luồng logs tổng hợp từ tất cả các container trong thời gian thực.
  • docker compose down: Dừng và xóa sạch các container, mạng ảo được tạo bởi compose (nhưng giữ nguyên dữ liệu trong volume).
  • docker compose down -v: Dừng hệ thống đồng thời xóa sạch toàn bộ các container và cả các volume dữ liệu đi kèm.

Chương 14: Quy trình làm việc cho người mới bắt đầu (Workflow for Beginners)

Để giúp những người mới bắt đầu có một quy trình phát triển phần mềm mượt mà, hiệu quả và không gặp lỗi trên môi trường Windows + WSL2, dưới đây là luồng làm việc thực tế được tối ưu hóa:

  1. Lập trình Cục bộ (Local Coding) trên WSL2:

    Tạo thư mục dự án của bạn nằm hoàn toàn trong hệ thống tệp tin gốc của Linux WSL2 (ví dụ: ~/projects/my-web-app/). Mở dự án bằng VS Code từ Windows thông qua tích hợp WSL. Viết mã nguồn ứng dụng của bạn (ví dụ: file main.py của FastAPI).

  2. Thiết lập Dockerfile (Dockerfile Setup):

    Viết một Dockerfile - tệp công thức build image tối ưu nằm ở gốc dự án. Định nghĩa cách thức cài đặt môi trường chạy ứng dụng, khai báo cổng, và lệnh khởi chạy máy chủ phát triển (ví dụ: chạy Uvicorn với flag --reload).

  3. Định nghĩa Compose (Compose Definition):

    Tạo tệp compose.yaml ở gốc dự án để mô tả mối liên kết giữa ứng dụng web và cơ sở dữ liệu. Sử dụng named volume - vùng lưu dữ liệu do Docker quản lý cho dữ liệu PostgreSQL để đảm bảo hiệu năng I/O tốt nhất.

  4. Cấu hình cập nhật nóng bằng Bind Mounts (Hot-reloading with Bind Mounts):

    Trong tệp compose.yaml, tại cấu hình của dịch vụ ứng dụng web, hãy ánh xạ thư mục hiện tại của máy host vào thư mục làm việc của container (ví dụ: - .:/app). Vì tệp tin được liên kết trực tiếp, khi bạn chỉnh sửa và lưu tệp code trong VS Code trên Windows, các thay đổi sẽ ngay lập tức xuất hiện bên trong container. Máy chủ Uvicorn phát hiện thay đổi và reload ứng dụng trong tích tắc mà không cần bạn phải build lại image.

Quy trình vận hành hàng ngày (Daily Run Loop)
  1. Bật Docker Desktop trên Windows.
  2. Mở terminal Ubuntu WSL2, di chuyển tới thư mục dự án: cd ~/projects/my-web-app/.
  3. Chạy hệ thống lên: docker compose up -d.
  4. Lập trình mã nguồn thoải mái trên VS Code, lưu file và kiểm tra kết quả ngay lập tức trên trình duyệt tại địa chỉ http://localhost:8000.

Chương 15: Các thực hành tốt nhất (Best Practices)

Khi phát triển ứng dụng bằng Docker, việc viết một Dockerfile - tệp công thức build image tối ưu là tối quan trọng. Điều này giúp tăng tốc độ build, giảm thiểu băng thông mạng khi pull/push và nâng cao tính bảo mật cho hệ thống. Dưới đây là 5 quy tắc thực tế vàng để tối ưu hóa:

  1. Giảm thiểu số lượng lớp layer (minimizing layers - giảm thiểu số lượng lớp):

    Mỗi chỉ lệnh trong Dockerfile như RUN, COPY, và ADD sẽ tạo ra một layer (lớp) vật lý mới. Càng nhiều lớp, dung lượng và thời gian tải của image (ảnh đóng gói) càng tăng. Hãy gộp các câu lệnh cài đặt hệ thống có liên quan lại với nhau bằng các ký tự nối lệnh && và ngắt dòng \.

  2. Sử dụng tệp .dockerignore (dockerignore file - tệp tin loại trừ khi build):

    Tương tự như .gitignore, tệp tin này định nghĩa các đường dẫn thư mục và tệp tin cục bộ mà bạn không muốn gửi vào build context - ngữ cảnh xây dựng của Docker Engine. Việc loại bỏ các thư mục nặng như node_modules, .git, hoặc .venv sẽ giúp quá trình gửi dữ liệu lên Docker Daemon diễn ra trong nháy mắt.

  3. Tách biệt bí mật/khóa API khỏi image (separating secrets/API keys from images - tách biệt thông tin nhạy cảm khỏi ảnh đóng gói):

    Tuyệt đối không lưu trữ cứng (hardcode) mật khẩu, token, hoặc API key bên trong Dockerfile hoặc mã nguồn của image. Điều này có thể dẫn đến việc lộ lọt thông tin nhạy cảm khi chia sẻ image công khai. Hãy sử dụng environment variables - biến môi trường được định nghĩa bên ngoài hoặc cơ chế Docker Secrets để nạp chúng vào runtime của container (vùng chạy cô lập).

  4. Sử dụng các image cơ sở siêu nhẹ (using slim base images - sử dụng các ảnh nền tối giản):

    Tránh sử dụng các image nền mặc định quá cồng kềnh nếu không cần thiết. Thay vì dùng image python:3.10 đầy đủ (dung lượng thô lên tới hơn 900MB), hãy cân nhắc các phiên bản tối giản như python:3.10-slim (chỉ khoảng 120MB) hoặc python:3.10-alpine (dưới 50MB) để hạn chế tối đa các lỗ hổng bảo mật tiềm ẩn và tăng tốc độ tải.

  5. Tối ưu hóa thứ tự các lớp để tái sử dụng cache (optimizing layer order for cache reuse - tối ưu thứ tự các lớp để sử dụng lại bộ nhớ đệm):

    Docker xây dựng image từ trên xuống dưới và sẽ sử dụng lại build cache - bộ nhớ đệm xây dựng cho các dòng lệnh chưa thay đổi. Hãy đặt các bước ít thay đổi (ví dụ: khai báo thư mục làm việc, cài đặt dependencies hệ thống) lên trên cùng, và đặt các bước thường xuyên thay đổi (ví dụ: sao chép mã nguồn ứng dụng liên tục được chỉnh sửa) ở dưới cùng.

So sánh cấu trúc Dockerfile: Chưa tối ưu vs Đã tối ưu

Dưới đây là sự khác biệt trực quan giữa cách viết Dockerfile của người mới bắt đầu (chưa tối ưu) và kỹ sư có kinh nghiệm (đã tối ưu):

1. Ví dụ Dockerfile chưa tối ưu (Unoptimized Dockerfile)

dockerfile 
FROM python:3.10
# Copy all project files first (including codebase and node_modules)
COPY . /app
WORKDIR /app
# Installing dependencies runs every time codebase changes because cache is busted
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

Đoạn mã trên thể hiện một Dockerfile chưa tối ưu vì sử dụng base image gốc dung lượng lớn và thực hiện copy toàn bộ tệp nguồn trước khi cài đặt dependencies, khiến build cache bị mất tác dụng mỗi khi chỉnh sửa code.

2. Ví dụ Dockerfile đã tối ưu (Optimized Dockerfile)

dockerfile 
FROM python:3.10-slim
WORKDIR /app
# Copy only dependencies file first to leverage caching
COPY requirements.txt .
# Run pip install without caching local packages downloaded to reduce image size
RUN pip install --no-cache-dir -r requirements.txt
# Copy the rest of the application files (highly volatile)
COPY . .
CMD ["python", "app.py"]

Đoạn mã trên thể hiện Dockerfile đã tối ưu sử dụng base image siêu nhẹ và chia nhỏ quá trình sao chép để tận dụng tối đa build cache của các lớp thư viện độc lập.

Chương 16: Các lỗi thường gặp và cách xử lý (Common Errors and How to Read Them)

Dưới đây là 5 lỗi kinh điển mà bất kỳ người mới bắt đầu nào trên môi trường Windows và WSL2 cũng sẽ gặp phải ít nhất một lần. Hãy học cách đọc thông báo lỗi và xử lý chúng một cách nhanh chóng:

Lỗi 1: Xung đột cấp phát cổng (port is already allocated - cổng đã được cấp phát) ▼

Thông báo lỗi đặc trưng:

Error response from daemon: Ports are not available: exposing port TCP 0.0.0.0:8080 -> 0.0.0.0:0: listen tcp 0.0.0.0:8080: bind: address already in use

Nguyên nhân:

Cổng mạng trên máy host Windows mà bạn muốn ánh xạ vào container đang bị một ứng dụng khác (như SQL Server cài trực tiếp, máy chủ web IIS hoặc một container cũ chạy ngầm) chiếm quyền sử dụng.

Cách khắc phục:

1. Tìm ra tiến trình đang chiếm dụng cổng trên Windows bằng Command Prompt (CMD) với quyền Administrator:

cmd 
netstat -ano | findstr :8080

Lệnh trên thực hiện tìm kiếm mã định danh PID của tiến trình đang lắng nghe tại cổng 8080 trên Windows.

2. Dừng tiến trình đó thông qua Task Manager hoặc đổi cổng ánh xạ bên ngoài của container sang cổng khác trống hơn (ví dụ sửa từ -p 8080:80 thành -p 8081:80).

Lỗi 2: Không thể kết nối tới Docker daemon (cannot connect to the Docker daemon - không thể kết nối tới trình nền quản lý Docker) ▼

Thông báo lỗi đặc trưng:

Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?

Nguyên nhân:

Docker CLI trên môi trường WSL2 không thể tìm thấy hoặc giao tiếp với dịch vụ nền Docker Engine. Điều này xảy ra do Docker Desktop trên Windows chưa khởi động, hoặc cấu hình đồng bộ hóa môi trường giữa Windows và WSL2 bị tắt.

Cách khắc phục:

  • Đảm bảo ứng dụng Docker Desktop trên Windows đang chạy (biểu tượng cá voi ở khay hệ thống sáng màu xanh lá).
  • Mở Docker Desktop > click vào biểu tượng bánh răng Settings > chọn mục Resources > WSL Integration và đảm bảo rằng bản phân phối Ubuntu của bạn đã được gạt nút bật (Enabled).
  • Khởi động lại dịch vụ Docker Desktop nếu cần thiết.
Lỗi 3: Container dừng ngay lập tức sau khi khởi chạy (container exits immediately upon starting - vùng chạy cô lập thoát ngay lập tức khi khởi chạy) ▼

Dấu hiệu:

Bạn chạy lệnh docker run -d my-app, hệ thống trả về mã hash container, nhưng khi chạy lệnh docker ps thì danh sách trống rỗng. Chạy docker ps -a thì thấy container có trạng thái Exited (1) hoặc Exited (0).

Nguyên nhân:

  • Ứng dụng chính được chỉ định ở lệnh CMD hoặc ENTRYPOINT gặp lỗi nghiêm trọng (crash) khi khởi động (lỗi code, thiếu biến môi trường, lỗi kết nối DB).
  • Container không có một tiến trình chạy nền liên tục (foreground service), dẫn đến việc Docker Engine tự động tắt container vì coi như nhiệm vụ đã hoàn thành.

Cách khắc phục:

1. Xem log lỗi đầu ra của container bị dừng để xác định nguyên nhân chi tiết:

bash 
docker logs <container_id_or_name>

Lệnh trên hiển thị toàn bộ nhật ký ghi nhận từ luồng ra tiêu chuẩn stdout/stderr của container đã thoát.

2. Đảm bảo cấu hình biến môi trường chính xác trong tệp tin Compose hoặc lệnh chạy. Nếu bạn chỉ muốn chạy một hệ điều hành trống để tương tác thử nghiệm, hãy bật chế độ tương tác bằng cờ -it (ví dụ: docker run -it ubuntu bash).

Lỗi 4: Từ chối quyền truy cập socket trên Linux/WSL2 (permission denied on Linux socket/WSL2 - từ chối quyền truy cập socket trên Linux/WSL2) ▼

Thông báo lỗi đặc trưng:

permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock

Nguyên nhân:

Tệp tin socket giao tiếp của Docker nằm tại đường dẫn /var/run/docker.sock yêu cầu quyền quản trị root để truy cập. User thông thường trên Ubuntu WSL2 không có quyền ghi vào tệp này nếu không nằm trong nhóm người dùng docker.

Cách khắc phục:

1. Thêm tài khoản người dùng hiện tại của bạn vào nhóm đặc quyền docker trên môi trường WSL2:

bash 
sudo usermod -aG docker $USER

Lệnh trên thực hiện thêm tài khoản người dùng WSL2 hiện tại vào nhóm quản lý docker để không cần dùng lệnh sudo mỗi khi tương tác.

2. Cập nhật nhóm quyền mới ngay lập tức mà không cần đăng nhập lại:

bash 
newgrp docker

Lệnh trên áp dụng ngay lập tức nhóm quyền mới cho phiên làm việc hiện tại của terminal.

Lỗi 5: Dữ liệu biến mất sau khi xóa container (data disappeared after container removal - dữ liệu biến mất sau khi xóa vùng chạy cô lập) ▼

Dấu hiệu:

Dữ liệu cơ sở dữ liệu (ví dụ: bảng MySQL, bài đăng ứng dụng) hoàn toàn biến mất sau khi bạn thực hiện xóa container cũ để nâng cấp phiên bản mới.

Nguyên nhân:

Bạn đã không ánh xạ dữ liệu ra ngoài máy host mà ghi trực tiếp dữ liệu vào writable layer - lớp ghi được tạm thời của container. Khi container bị xóa bằng lệnh docker rm, lớp ghi tạm này sẽ bị hệ thống dọn sạch hoàn toàn.

Cách khắc phục:

Sử dụng cơ chế lưu trữ bền vững volume (vùng lưu dữ liệu) hoặc bind mount - liên kết thư mục để tách biệt vòng đời dữ liệu khỏi vòng đời của container. Ví dụ cấu hình trong tệp tin compose.yaml:

yaml 
services:
  db:
    image: postgres:15-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:

Đoạn mã trên định nghĩa một named volume - vùng lưu dữ liệu do Docker quản lý có tên là pgdata liên kết trực tiếp vào thư mục lưu trữ cốt lõi của PostgreSQL bên trong container, đảm bảo dữ liệu không bị xóa bỏ khi container được nâng cấp hoặc làm mới.

Chương 17: Bảng tra cứu lệnh nhanh (Command Cheat Sheet)

Dưới đây là cẩm nang tra cứu nhanh các câu lệnh Docker phổ biến nhất được phân loại theo từng nhóm đối tượng tài nguyên cụ thể để giúp bạn thao tác nhanh chóng trên terminal:

1. Quản lý Image (Image Management)

Câu lệnh (Command) Ý nghĩa & Giải thích tiếng Việt
docker build -t <name>:<tag> . Xây dựng một image (ảnh đóng gói) mới từ Dockerfile - tệp công thức build image nằm ở thư mục hiện tại.
docker images Liệt kê toàn bộ các image (ảnh đóng gói) hiện đang lưu trữ trên máy host.
docker rmi <image_id> Xóa bỏ một image (ảnh đóng gói) khỏi ổ cứng máy host (chỉ thực hiện được khi không có container nào đang sử dụng image này).
docker pull <image_name> Tải một image (ảnh đóng gói) từ registry (kho lưu trữ) Docker Hub về bộ lưu trữ cục bộ.
docker image prune Xóa sạch toàn bộ các image (ảnh đóng gói) rác không còn được gắn thẻ tag (dangling images).

2. Quản lý Container (Container Management)

Câu lệnh (Command) Ý nghĩa & Giải thích tiếng Việt
docker run -d -p <host_port>:<container_port> --name <name> <image> Tạo mới và khởi chạy một container (vùng chạy cô lập) từ image (ảnh đóng gói) ở chế độ chạy ngầm (detached mode) và chuyển tiếp cổng.
docker ps Liệt kê danh sách các container (vùng chạy cô lập) đang ở trạng thái hoạt động (running).
docker ps -a Hiển thị toàn bộ danh sách container (vùng chạy cô lập) trong hệ thống bao gồm cả các container đã dừng.
docker stop <container_id> Gửi tín hiệu dừng an toàn tới một container (vùng chạy cô lập) đang chạy.
docker rm <container_id> Xóa bỏ vĩnh viễn một container (vùng chạy cô lập) đã dừng hoạt động.
docker logs <container_id> Xem toàn bộ thông tin logs đầu ra ghi nhận từ container.
docker exec -it <container_id> bash Truy cập trực tiếp vào interactive mode - chế độ tương tác dòng lệnh bash bên trong container đang chạy.

3. Quản lý Volume (Volume Management)

Câu lệnh (Command) Ý nghĩa & Giải thích tiếng Việt
docker volume create <volume_name> Khởi tạo một volume (vùng lưu dữ liệu) hoàn toàn mới do Docker quản lý.
docker volume ls Liệt kê toàn bộ các volume (vùng lưu dữ liệu) hiện có trên máy host.
docker volume inspect <volume_name> Xem thông tin chi tiết cấu hình và đường dẫn lưu trữ vật lý của volume (vùng lưu dữ liệu).
docker volume rm <volume_name> Xóa bỏ một volume (vùng lưu dữ liệu) không còn sử dụng.
docker volume prune Xóa sạch toàn bộ các volume (vùng lưu dữ liệu) mồ côi (không liên kết với bất kỳ container nào) để giải phóng ổ cứng.

4. Quản lý Network (Network Management)

Câu lệnh (Command) Ý nghĩa & Giải thích tiếng Việt
docker network create <network_name> Khởi tạo một network (mạng) ảo cục bộ để các container tự do kết nối và giao tiếp.
docker network ls Hiển thị danh sách các mạng ảo hiện có trong Docker Engine.
docker network connect <network> <container> Kết nối trực tiếp một container đang chạy vào một mạng ảo xác định.
docker network disconnect <network> <container> Ngắt kết nối của một container ra khỏi mạng ảo đã tham gia.
docker network rm <network_name> Xóa bỏ một network (mạng) ảo ra khỏi hệ thống.

5. Quản lý Docker Compose (Compose Management)

Câu lệnh (Command) Ý nghĩa & Giải thích tiếng Việt
docker compose up -d Tải image, khởi tạo mạng ảo, tạo volume và chạy toàn bộ ứng dụng đa dịch vụ ngầm.
docker compose down Dừng hoạt động và xóa sạch các container, mạng ảo được tạo bởi compose của dự án hiện tại.
docker compose down -v Tương tự như down, nhưng dọn dẹp và xóa sạch cả các volume (vùng lưu dữ liệu) đi kèm để reset hệ thống.
docker compose logs -f Theo dõi trực tiếp logs của toàn bộ các dịch vụ đang chạy trong dự án Compose.
docker compose ps Xem trạng thái chi tiết của tất cả các container thuộc dự án Compose hiện hành.

Chương 18: Thuật ngữ ôn tập (Glossary)

Để củng cố lại toàn bộ kiến thức cẩm nang, dưới đây là danh sách tổng hợp các thuật ngữ cốt lõi được định nghĩa chi tiết và dịch nghĩa sang tiếng Việt tương đương, được chia theo từng nhóm chủ đề lớn:

1. Nhóm thành phần cốt lõi (Core Artifacts) ▼
Thuật ngữ (Term) Định nghĩa & Nghĩa tiếng Việt
image (ảnh đóng gói) Một gói dữ liệu chỉ đọc, đóng vai trò làm khuôn mẫu chứa toàn bộ mã nguồn, thư viện hệ thống và cấu hình cần thiết để khởi chạy một ứng dụng.
container (vùng chạy cô lập) Một thực thể hoạt động được sinh ra từ image. Nó chạy hoàn toàn biệt lập với môi trường xung quanh, giúp bảo vệ ứng dụng khỏi xung đột.
volume (vùng lưu dữ liệu) Cơ chế lưu trữ dữ liệu vĩnh viễn được quản lý bởi Docker, nằm bên ngoài hệ thống file của container để bảo toàn dữ liệu khi xóa container.
network (mạng) Hệ thống liên kết và điều hướng giao tiếp giữa các container với nhau và giữa các container với môi trường mạng bên ngoài máy host.
2. Nhóm Xây dựng và Cấu hình (Build and Configuration) ▼
Thuật ngữ (Term) Định nghĩa & Nghĩa tiếng Việt
Dockerfile - tệp công thức build image Tệp văn bản thuần chứa danh sách các chỉ lệnh tuần tự giúp Docker Engine tự động xây dựng nên một image hoàn chỉnh.
build cache - bộ nhớ đệm xây dựng Cơ chế lưu trữ lại kết quả của các bước build Dockerfile trước đó để tái sử dụng, giúp tăng tốc đáng kể thời gian build ở những lần sau.
build context - ngữ cảnh xây dựng Toàn bộ các tệp tin và thư mục cục bộ được gửi từ client lên Docker Daemon để phục vụ cho việc build image.
base image - ảnh đóng gói nền Image cơ sở được khai báo ở dòng lệnh FROM đầu tiên của Dockerfile để làm nền tảng xây dựng các layer phía trên.
multi-stage build - xây dựng nhiều giai đoạn Kỹ thuật build nâng cao sử dụng nhiều câu lệnh FROM trong một Dockerfile để lọc bỏ các tệp tin build thừa, giữ lại thành phẩm chạy tối giản nhất.
3. Nhóm Thực thi và Lưu trữ (Execution and Storage) ▼
Thuật ngữ (Term) Định nghĩa & Nghĩa tiếng Việt
bind mount - liên kết thư mục Phương thức liên kết trực tiếp một đường dẫn thư mục cụ thể trên máy host vật lý vào bên trong container (thường dùng cho code khi code-reload).
named volume - vùng lưu dữ liệu do Docker quản lý Một volume có tên định danh rõ ràng, được Docker tự động quản lý vị trí lưu trữ thực tế trên ổ cứng nhằm mang lại hiệu năng tối đa.
port forwarding - chuyển tiếp cổng Cơ chế ánh xạ một cổng mạng từ máy host vào bên trong cổng mạng của container để bên ngoài có thể truy cập được ứng dụng dịch vụ.
interactive mode - chế độ tương tác Chế độ chạy container cho phép liên kết trực tiếp luồng nhập/xuất (stdin/stdout) với terminal của máy host (cờ -it).
detached mode - chế độ chạy ngầm Chế độ khởi chạy container chạy ẩn trong nền hệ thống mà không chiếm quyền điều khiển của terminal hiện hành (cờ -d).
4. Nhóm Điều phối dịch vụ (Orchestration) ▼
Thuật ngữ (Term) Định nghĩa & Nghĩa tiếng Việt
compose file - tệp cấu hình Compose Tệp cấu hình định dạng YAML (thường đặt tên là compose.yaml) mô tả cấu trúc, cài đặt liên kết của toàn bộ hệ thống đa dịch vụ.
service (dịch vụ ứng dụng) Một khai báo dịch vụ trong compose đại diện cho một hoặc nhiều container chạy song song có cùng chung cấu hình hoạt động.
scaling (mở rộng số lượng bản sao) Hành vi tăng hoặc giảm số lượng thực thể container giống nhau chạy song song của một dịch vụ nhằm chia sẻ tải trọng.
recreate (tái thiết lập container) Hành vi của Docker Compose tự động hủy container cũ và dựng lên một container mới hoàn toàn khi phát hiện cấu hình hoặc image có sự thay đổi.

Chương 19: Đường chạy Nâng cao: Docker trong Triển khai Điện toán Đám mây (Docker for Cloud Deployment)

Khi ứng dụng đã sẵn sàng chạy ổn định ở `local environment - môi trường cục bộ`, bước tiếp theo là thực hiện `cloud deployment - triển khai trên đám mây`. Việc dịch chuyển từ máy tính cá nhân lên hạ tầng đám mây mang lại nhiều thách thức do sự khác biệt cơ bản về mặt kiến trúc.

So sánh hành vi của container: Local vs Cloud

Để triển khai thành công, lập trình viên cần hiểu rõ sự khác biệt giữa `local container behavior - hành vi container ở môi trường cục bộ` và `cloud container behavior - hành vi container ở môi trường đám mây`:

Đặc tính (Feature) Môi trường cục bộ (Local Environment) Môi trường đám mây (Cloud Environment)
Hạ tầng lưu trữ (Storage) Sử dụng ổ cứng vật lý trực tiếp, dữ liệu được gắn kết lâu dài qua `bind mount - liên kết thư mục` hoặc `named volume - vùng lưu dữ liệu do Docker quản lý`. Sử dụng `ephemeral storage - lưu trữ tạm thời`. Dữ liệu ghi trực tiếp trong `container (vùng chạy cô lập)` sẽ bị xóa sạch khi container khởi động lại hoặc chuyển đổi host.
Địa chỉ mạng (Networking) Truy cập trực tiếp thông qua cổng localhost (máy cục bộ) hoặc địa chỉ IP tĩnh nội bộ của WSL2. Đứng sau các dịch vụ định tuyến phức tạp như `load balancer - bộ cân bằng tải`, tường lửa và cổng kết nối mạng riêng ảo.
Cấu hình tài nguyên (Resources) Phụ thuộc vào tài nguyên vật lý của máy tính cá nhân (RAM, CPU, GPU cục bộ). Được cấp phát động, tự động co giãn dựa trên `scaling configuration - cấu hình tự động mở rộng quy mô`.

Cấu hình thời gian chạy & Biến môi trường (Runtime Configuration & Environment Variables)

Một nguyên tắc tối kị trong phát triển ứng dụng hiện đại là đóng gói cứng (hardcode) các cấu hình (như cổng kết nối, chuỗi kết nối cơ sở dữ liệu, API key) vào trong `image (ảnh đóng gói)`. Thay vào đó, ta sử dụng `runtime configuration - cấu hình thời gian chạy` dựa trên `environment variable - biến môi trường`.

Bằng cách sử dụng các `environment variable - biến môi trường`, một `image (ảnh đóng gói)` duy nhất có thể được tái sử dụng linh hoạt trên nhiều môi trường khác nhau: từ máy cá nhân của lập trình viên, máy chủ thử nghiệm (staging), cho đến hệ thống vận hành thực tế (production).

Cổng kết nối mở ra ngoài & Cơ chế ghi nhận nhật ký (Exposed Ports & Logging)

Khi chạy trên đám mây, `container (vùng chạy cô lập)` chỉ cần khai báo một `exposed port - cổng kết nối được mở ra ngoài` (ví dụ cổng 8000). Hạ tầng đám mây sẽ chịu trách nhiệm chuyển tiếp traffic từ các cổng tiêu chuẩn (80 cho HTTP, 443 cho HTTPS) vào cổng nội bộ này.

⚠️
Nguyên tắc ghi log trên Cloud (Cloud Logging Best Practice)

Vì container trên đám mây hoạt động trên cơ chế `ephemeral storage - lưu trữ tạm thời`, tuyệt đối không được ghi nhật ký hoạt động (logs) vào tệp tin tĩnh bên trong container. Nếu làm vậy, log sẽ biến mất khi container gặp sự cố hoặc được cập nhật.

Giải pháp: Toàn bộ `logging mechanism - cơ chế ghi nhận nhật ký` của ứng dụng phải xuất log trực tiếp ra `standard output (luồng xuất chuẩn - stdout)` và `standard error (luồng lỗi chuẩn - stderr)`. Dịch vụ đám mây sẽ tự động thu thập các luồng này và đẩy về các công cụ quản lý log tập trung.

⚠️
Cảnh báo quan trọng về dịch vụ AWS App Runner:

Theo tài liệu chính thức từ AWS, dịch vụ AWS App Runner không còn mở cho khách hàng mới đăng ký (no longer open to new customers). Để đảm bảo tính bền vững và hiện đại cho các dự án triển khai mới:

  • Đối với người dùng mới: Hãy sử dụng dịch vụ Amazon ECS (Elastic Container Service) kết hợp với AWS Fargate làm luồng triển khai mặc định hiện đại và lâu dài. Quy trình build và đẩy image lên AWS ECR trong hướng dẫn dưới đây hoàn toàn trùng khớp; bạn chỉ cần cấu hình ECS Fargate Task Definition trỏ đến URI của ECR image thay vì tạo dịch vụ App Runner.
  • Đối với người dùng cũ (đã active App Runner trước đó): Bạn vẫn có thể tham khảo luồng cấu hình App Runner được giữ lại trong tài liệu này làm mẫu tham khảo nhanh.

Để đưa một ứng dụng AI đã được đóng gói container lên đám mây AWS, quy trình chuẩn hóa bao gồm việc đẩy `image (ảnh đóng gói)` lên AWS ECR (`Amazon Elastic Container Registry - kho lưu trữ ảnh đóng gói container của AWS`), sau đó kéo ảnh chạy serverless thông qua AWS Fargate (trong Amazon ECS) hoặc AWS App Runner (nếu tài khoản của bạn đã được kích hoạt trước đó). Quy trình này giúp tự động hóa từ việc lưu trữ image đến vận hành thực tế mà không cần quản lý máy chủ vật lý.

Các bước đẩy ảnh đóng gói lên Amazon ECR

Dưới đây là các câu lệnh chính xác thực thi trên terminal WSL2 Ubuntu để đăng nhập và đẩy `image (ảnh đóng gói)` lên AWS ECR:

  1. Tạo repository chứa image trên AWS ECR:

    bash 
    aws ecr create-repository --repository-name my-ai-research-agent --region ap-southeast-1

    Đoạn mã trên gửi yêu cầu đến AWS để tạo một `repository (kho chứa)` mới có tên my-ai-research-agent tại khu vực Singapore (ap-southeast-1).

  2. Xác thực Docker CLI với AWS ECR (Authenticate using get-login-password):

    bash 
    aws ecr get-login-password --region ap-southeast-1 | docker login --username AWS --password-stdin 123456789012.dkr.ecr.ap-southeast-1.amazonaws.com

    Đoạn mã trên sử dụng lệnh get-login-password để lấy mã thông báo xác thực dùng một lần từ AWS, sau đó truyền qua pipe (|) để đăng nhập vào Docker Registry bảo mật của AWS. Hãy thay thế 123456789012 bằng ID tài khoản AWS thực tế của bạn.

  3. Gắn nhãn (Tagging) local image:

    bash 
    docker tag my-ai-research-agent:latest 123456789012.dkr.ecr.ap-southeast-1.amazonaws.com/my-ai-research-agent:latest

    Đoạn mã trên tạo một thẻ nhãn mới cho `image (ảnh đóng gói)` từ tên cục bộ my-ai-research-agent:latest trỏ trực tiếp đến địa chỉ kho chứa trên AWS ECR vừa đăng ký.

  4. Đẩy (Pushing) image lên ECR:

    bash 
    docker push 123456789012.dkr.ecr.ap-southeast-1.amazonaws.com/my-ai-research-agent:latest

    Đoạn mã trên thực hiện tải toàn bộ các lớp dữ liệu của `image (ảnh đóng gói)` từ máy cục bộ lên AWS ECR. Các lớp dữ liệu trùng lặp với thư viện chuẩn sẽ được tối ưu tải lên tự động.

Cấu hình vận hành trên AWS App Runner

Khi `image (ảnh đóng gói)` đã nằm an toàn trên AWS ECR, ta tiến hành cấu hình AWS App Runner để tự động tạo `instance (thực thể chạy)` và vận hành dịch vụ:

1. Tự động kéo ảnh đóng gói (Pulling Image & Auto Deploy) ▼

Trong giao diện AWS App Runner Console, chọn nguồn cấp dữ liệu là ECR. Cung cấp đường dẫn ảnh đóng gói (URI ECR). Bạn có thể kích hoạt tùy chọn Automatic Deployment: mỗi khi có phiên bản image mới được đẩy lên thẻ latest, App Runner sẽ tự động kéo về và chạy quy trình nâng cấp không gián đoạn (zero-downtime deployment).

2. Ánh xạ cổng kết nối (Port Mapping & Networking) ▼

App Runner yêu cầu bạn khai báo cổng kết nối mà container của bạn đang lắng nghe để phục vụ các yêu cầu HTTP. Ví dụ, nếu ứng dụng Python FastAPI chạy trên cổng 8000, ta điền cổng 8000 vào cấu hình App Runner. Dịch vụ đám mây sẽ tự động cấu hình bộ cân bằng tải công cộng để định tuyến các cuộc gọi HTTPS ngoài internet (cổng 443) đi thẳng vào cổng 8000 của container.

3. Cấu hình tự động co giãn (Scaling Configuration) ▼

Bạn có thể cấu hình `scaling configuration - cấu hình tự động mở rộng quy mô` dựa trên số lượng request đồng thời (Concurrency limit). Ví dụ, nếu quy định tối đa 100 kết nối đồng thời trên mỗi instance, khi lưu lượng truy cập tăng vọt vượt ngưỡng, App Runner sẽ tự động nhân bản thêm các container instances mới chạy song song để gánh tải. Khi lưu lượng giảm, hệ thống sẽ tự động tắt bớt để giảm chi phí.

4. Tiêm biến môi trường chứa khóa API (Injecting Environment Variables) ▼

Thay vì lưu khóa bí mật trong mã nguồn, ta cấu hình chúng trực tiếp trong mục Environment Variables của App Runner. Các `API key environment variables - biến môi trường chứa khóa truy cập API` như GEMINI_API_KEY sẽ được hệ thống mã hóa và tiêm trực tiếp vào môi trường thực thi của container một cách an toàn nhất.

Chương 21: Đóng gói Tác nhân Trí tuệ Nhân tạo với Docker trong Triển khai Đám mây

Một `containerized AI agent - tác nhân trí tuệ nhân tạo được đóng gói trong container` có các đặc điểm hoạt động rất đặc thù so với các microservices truyền thống. Tác nhân AI thường thực hiện các tác vụ nặng về tính toán, duy trì trạng thái hội thoại dài hạn và tương tác liên tục với các mô hình ngôn ngữ lớn (LLM) bên ngoài.

Kiến trúc cấu trúc của tác nhân AI trong Container

Để hoạt động ổn định trên môi trường đám mây, một tác nhân AI cần có cấu trúc chuẩn hóa gồm các thành phần sau:

1. Giao diện máy chủ ứng dụng (App Server Interface)

Sử dụng các framework lập trình web bất đồng bộ như FastAPI phối hợp với máy chủ ASGI Uvicorn. Việc sử dụng `asynchronous programming - lập trình bất đồng bộ` là bắt buộc vì tác nhân AI dành phần lớn thời gian để chờ đợi kết quả trả về (I/O bound) từ các API mô hình lớn của Google hay OpenAI.

2. Quản lý thư viện phụ thuộc (Dependency Management)

Khai báo rõ ràng tất cả `dependency (thư viện phụ thuộc)` cần thiết cho việc vận hành tác nhân AI như thư viện kết nối mô hình (google-generativeai), thư viện xử lý dữ liệu (pydantic), thư viện kết nối mạng bất đồng bộ (httpx) vào tệp tin cấu hình. Việc khóa chặt phiên bản thư viện ngăn ngừa các lỗi không tương thích runtime khi build image.

3. Quản lý biến môi trường chứa khóa truy cập API (API Key Environment Variables)

Các tác nhân AI luôn yêu cầu các `API key environment variables - biến môi trường chứa khóa truy cập API` để xác thực với nhà cung cấp dịch vụ LLM. Các biến này phải được đọc động bằng thư viện os.getenv khi khởi chạy ứng dụng. Tuyệt đối không được nhúng cứng khóa bảo mật vào Dockerfile hay mã nguồn dự án để tránh rò rỉ thông tin.

4. Quyền truy cập mạng (Network Access Requirements)

Tác nhân AI cần `network access - quyền truy cập mạng` chiều đi (Egress internet access) thông suốt để giao tiếp với các API của bên thứ ba, thực hiện các công cụ tìm kiếm web (web search tools) hoặc thu thập dữ liệu trực tuyến. Cấu hình bảo mật mạng trên đám mây phải cho phép container kết nối ra ngoài internet.

5. Yêu cầu lưu trữ dữ liệu bền vững (Persistency Requirements)

Do container đám mây mang đặc tính `ephemeral storage - lưu trữ tạm thời`, mọi `persistency requirement - yêu cầu lưu trữ dữ liệu bền vững` (như lịch sử chat của người dùng, cơ sở dữ liệu vector lưu tài liệu tham khảo) không được phép ghi trực tiếp lên ổ đĩa của container. Mọi dữ liệu này phải được ghi nhận vào các dịch vụ lưu trữ bên ngoài như RDS, DynamoDB hoặc các dịch vụ Vector Database đám mây (Pinecone/Qdrant Cloud).

Chương 22: Triển khai Tác nhân Nghiên cứu AI với Docker, ECR (Mẫu Tham Khảo Triển Khai - Illustrative Example)

Để hoàn thiện cẩm nang này, chúng ta sẽ phân tích một mẫu tham khảo triển khai (illustrative example - ví dụ minh họa) của một ứng dụng tác nhân nghiên cứu AI (AI Research Agent) bằng Python FastAPI. Mô hình tham khảo này cung cấp một `endpoint (điểm cuối kết nối)` /api/chat thực hiện chuỗi tư duy suy luận gồm 3 bước và gọi API Google Gemini thực tế (yêu cầu nạp biến môi trường khóa API thật). Đi kèm là một `production-ready Dockerfile - tệp công thức build image tối ưu cho môi trường vận hành` được thiết kế để làm khuôn mẫu tham khảo tối giản hóa dung lượng và nâng cao bảo mật.

Mã nguồn ứng dụng Python FastAPI (main.py) - Mẫu Tham Khảo Triển Khai

Dưới đây là khung mã nguồn đầy đủ của ứng dụng để làm cấu trúc tham khảo triển khai:

python 
import os
import logging
import httpx
from fastapi import FastAPI, HTTPException, status
from pydantic import BaseModel, Field
from typing import List, Optional

# Cấu hình logging để xuất log ra luồng xuất chuẩn (stdout) phục vụ Cloud Logging
logging.basicConfig(level=logging.INFO, format="%(asctime)s - %(levelname)s - %(message)s")
logger = logging.getLogger(__name__)

app = FastAPI(
    title="AI Research Agent API",
    description="Production-ready FastAPI AI Agent simulating reasoning steps and integrating Gemini API.",
    version="1.0.0"
)

# Khai báo cấu trúc dữ liệu đầu vào sử dụng Pydantic
class ChatRequest(BaseModel):
    message: str = Field(..., description="The research topic or message for the AI Agent", example="Explain Docker container lifecycle in simple terms.")
    session_id: Optional[str] = Field(None, description="Unique identifier for the chat session", example="session-123")

# Khai báo cấu trúc bước suy luận (Thinking Step)
class ThinkingStep(BaseModel):
    step: int
    title: str
    content: str

# Khai báo cấu trúc dữ liệu trả về cho client
class ChatResponse(BaseModel):
    session_id: str
    response: str
    thinking_steps: List[ThinkingStep]
    model_used: str

@app.get("/")
def read_root():
    """Endpoint dùng để kiểm tra trạng thái hoạt động của container (Health Check)"""
    return {"status": "healthy", "service": "AI Research Agent"}

@app.post("/api/chat", response_model=ChatResponse, status_code=status.HTTP_200_OK)
async def chat_endpoint(request: ChatRequest):
    """Endpoint xử lý logic trò chuyện và chuỗi suy luận của tác nhân AI"""
    logger.info(f"Processing request for session: {request.session_id}")
    
    # Bước suy luận 1 (Thinking Step 1): Lập kế hoạch phân tích yêu cầu của người dùng
    step1 = ThinkingStep(
        step=1,
        title="Query Intent Parsing & Research Strategy Formulation",
        content=f"Analyzing user query: '{request.message}'. Mapping out key topics. Formulating plan to fetch relevant context and model the output structure."
    )
    
    # Bước suy luận 2 (Thinking Step 2): Tổng hợp kiến thức và xây dựng nội dung cốt lõi
    step2 = ThinkingStep(
        step=2,
        title="Information Synthesis & Deep Reasoning",
        content="Processing conceptual definitions. Structuring the explanation framework. Generating detailed response components for Docker execution lifecycle."
    )

    # Lấy API Key từ biến môi trường của container
    api_key = os.getenv("GEMINI_API_KEY")
    model_used = "gemini-1.5-flash (Simulated)"
    final_answer = ""
    
    # Bước suy luận 3 (Thinking Step 3): Gọi mô hình ngôn ngữ lớn để sinh câu trả lời
    if api_key:
        model_used = "gemini-1.5-flash"
        url = f"https://generativelanguage.googleapis.com/v1beta/models/gemini-1.5-flash:generateContent?key={api_key}"
        headers = {"Content-Type": "application/json"}
        payload = {
            "contents": [{
                "parts": [{
                    "text": f"You are a Senior DevOps Engineer and AI Architect. Provide a detailed, professional answer to: {request.message}"
                }]
            }]
        }
        
        try:
            logger.info("Initiating request to Google Gemini API...")
            async with httpx.AsyncClient(timeout=30.0) as client:
                response = await client.post(url, json=payload, headers=headers)
                
                if response.status_code == 200:
                    res_data = response.json()
                    final_answer = res_data["candidates"][0]["content"]["parts"][0]["text"]
                    logger.info("Successfully fetched response from Gemini API.")
                else:
                    logger.warning(f"Gemini API returned an error code: {response.status_code}. Executing fallback response.")
                    final_answer = f"[Fallback Response] Gemini API returned error: {response.text}. Prompt query: {request.message}"
        except Exception as e:
            logger.error(f"Failed to communicate with Gemini API due to exception: {str(e)}")
            final_answer = f"[Fallback Response] Connection failed. Query: {request.message}"
    else:
        logger.info("GEMINI_API_KEY is not defined. Initiating local simulator mode.")
        final_answer = (
            f"Hello! This is a simulation response because no GEMINI_API_KEY environment variable was detected. "
            f"Your request was: '{request.message}'. "
            f"Please configure your GEMINI_API_KEY in the AWS App Runner environment variables to connect this agent to the live Google Gemini API."
        )

    step3 = ThinkingStep(
        step=3,
        title="Response Refinement & Formatting",
        content="Refining markdown layout, structuring code blocks, and returning final answer response to the caller client."
    )

    return ChatResponse(
        session_id=request.session_id or "default-session",
        response=final_answer,
        thinking_steps=[step1, step2, step3],
        model_used=model_used
    )

Đoạn mã Python trên khởi tạo một ứng dụng FastAPI có hai đường dẫn truy cập. Hàm read_root xử lý yêu cầu GET tại gốc (/) để hệ thống kiểm tra sức khỏe của container. Hàm chat_endpoint xử lý yêu cầu POST gửi tới /api/chat, thực hiện giả lập chuỗi suy luận 3 bước (Thinking Steps) và gọi API thực tế tới dịch vụ Gemini của Google thông qua thư viện kết nối không đồng bộ httpx.

dockerfile

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# Cài đặt thư viện hệ thống cần thiết cho việc cài đặt gói

    build-essential \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# Giai đoạn 2: Stage chạy cuối cùng siêu gọn nhẹ (Production Environment)
FROM python:3.11-slim

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

WORKDIR /app

# Sao chép các package đã được biên dịch từ giai đoạn builder
COPY --from=builder /install /usr/local
COPY main.py .

# Tạo một user non-root để chạy container nhằm đảm bảo an toàn bảo mật thông tin
RUN useradd -u 1001 appuser && chown -R appuser:appuser /app

# Khai báo cổng kết nối container sẽ lắng nghe lúc runtime

Đoạn mã trên sử dụng kỹ thuật `multi-stage build - xây dựng nhiều giai đoạn` để phân tách quá trình biên dịch thư viện (builder) ra khỏi sản phẩm chạy cuối cùng. Phương pháp này giúp loại bỏ hoàn toàn các tệp tin rác phát sinh trong lúc cài đặt thư viện, giảm dung lượng `image (ảnh đóng gói)` xuống mức tối thiểu. Đồng thời, cấu hình khởi tạo một `non-root user (người dùng không có quyền quản trị)` có tên là appuser để chạy container, ngăn ngừa các cuộc tấn công chiếm quyền điều khiển hệ thống máy chủ đám mây.
