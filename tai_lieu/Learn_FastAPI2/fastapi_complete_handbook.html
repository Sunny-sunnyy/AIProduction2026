Part A: FastAPI Foundations

Chương 1: FastAPI là gì?

Mục tiêu

Hiểu được bản chất của framework FastAPI, các thành phần cấu tạo cốt lõi (Starlette và Pydantic) và sự khác biệt căn bản của nó so với các framework truyền thống như Django hay Flask.

1. Bản chất và Sự phổ biến

FastAPI là một Python web framework hiện đại, hiệu năng cao để xây dựng các API, được viết dựa trên các gợi ý kiểu dữ liệu (standard Python type hints) của Python 3.8+. FastAPI kế thừa sức mạnh từ hai thư viện nền tảng:

  • Starlette: Đảm nhiệm phần xử lý web trực tiếp (routing, request/response, WebSocket), mang lại hiệu năng phi thường tương đương với Node.js và Go.
  • Pydantic: Đảm nhiệm phần quản lý cấu trúc dữ liệu, tự động chuyển đổi kiểu (data serialization) và xác thực dữ liệu đầu vào (data validation).

2. So sánh FastAPI, Flask và Django

Để giúp lập trình viên mới hiểu rõ hơn về bức tranh tổng thể, dưới đây là bảng so sánh chi tiết giữa ba ông lớn của thế giới Python Web:

Tiêu chí FastAPI Flask Django
Kiểu kiến trúc ASGI (Không đồng bộ mặc định) WSGI (Đồng bộ mặc định) WSGI / ASGI (Monolithic)
Hiệu năng Cực kỳ cao (Ngang Go/NodeJS) Trung bình Trung bình
Validation Tự động qua Pydantic Phải tự viết/thư viện ngoài Django Forms/DRF Serializers
Tự động sinh tài liệu Có sẵn (Swagger UI, ReDoc) Không Không (cần cài DRF-YASG)
Khi nào nên chọn FastAPI?

FastAPI là lựa chọn hoàn hảo khi bạn cần xây dựng API hướng dịch vụ microservices, ứng dụng có lưu lượng kết nối lớn cần cơ chế non-blocking (chat, streaming dữ liệu) hoặc các ứng dụng phục vụ mô hình AI/ML cần tính toán bất đồng bộ và kiểm soát dữ liệu đầu vào nghiêm ngặt.

Recap: FastAPI = Starlette (Web/ASGI) + Pydantic (Validation) + Python Type Hints. Nó sinh ra để phát triển API nhanh gọn, có hiệu năng cực cao và tài liệu tự động.

Chương 2: HTTP và REST Basics

Mục tiêu

Trang bị kiến thức nền tảng về giao thức truyền dữ liệu siêu văn bản (HTTP) và nguyên lý kiến trúc RESTful API trước khi bắt tay vào triển khai thực tế.

1. Request và Response trong HTTP

Trong thế giới Web, mọi giao dịch được thực hiện qua chu trình Request-Response giữa Client (trình duyệt, ứng dụng di động) và Server:

  • Request (Yêu cầu): Chứa HTTP Method (GET, POST...), Target URL, Headers (chứa metadata như Content-Type, Authorization), và Request Body (JSON payload).
  • Response (Phản hồi): Chứa HTTP Status Code (200 OK, 404 Not Found...), Headers và Response Body.

2. HTTP Methods và Tính bất biến (Idempotency)

Mỗi phương thức HTTP biểu thị hành động dự kiến đối với tài nguyên:

  • GET: Truy xuất dữ liệu (Bất biến - Idempotent).
  • POST: Tạo tài nguyên mới (Không bất biến).
  • PUT: Cập nhật toàn bộ tài nguyên (Bất biến).
  • PATCH: Cập nhật một phần tài nguyên (Không bất biến).
  • DELETE: Xóa tài nguyên (Bất biến).
Bất biến (Idempotent) là gì?

Một phương thức được coi là bất biến nếu nhiều yêu cầu giống hệt nhau gửi lên server sẽ luôn đem lại kết quả giống như lần gửi đầu tiên, không làm thay đổi trạng thái tài nguyên của hệ thống nhiều lần.

3. Truyền dữ liệu: Path vs Query vs Body

  • Path Parameter: Dùng để định danh một tài nguyên cụ thể (Ví dụ: /users/42 - lấy user có ID 42).
  • Query Parameter: Dùng để lọc, sắp xếp, hoặc phân trang tài nguyên (Ví dụ: /users?limit=10&page=2).
  • Request Body: Dùng để gửi payload phức tạp lên server, thường định dạng JSON (Ví dụ: Dữ liệu đăng ký tài khoản).

Recap: Thiết kế RESTful API chuẩn mực yêu cầu sử dụng đúng các HTTP Methods và phân biệt rõ khi nào truyền tham số qua Path, Query hoặc Request Body.

Chương 3: ASGI, WSGI & Async

Mục tiêu

Tìm hiểu cơ chế chạy bất đồng bộ của FastAPI, hiểu sự khác biệt giữa chuẩn giao tiếp cũ (WSGI) và chuẩn giao tiếp bất đồng bộ mới (ASGI).

1. WSGI vs ASGI

WSGI (Web Server Gateway Interface): Chuẩn giao tiếp đồng bộ (synchronous). Mỗi khi có request, web server sẽ phân bổ một luồng (thread) xử lý riêng. Nếu luồng đó thực hiện tác vụ I/O (như gọi DB, đọc file), nó sẽ bị chặn (block), không thể xử lý request khác.

ASGI (Asynchronous Server Gateway Interface): Bản nâng cấp của WSGI hỗ trợ xử lý bất đồng bộ (asynchronous), cho phép nhiều kết nối diễn ra đồng thời trên một luồng duy nhất (Event Loop). ASGI cực kỳ phù hợp cho WebSockets, Server-Sent Events, hoặc các tác vụ I/O kéo dài.

2. Vai trò của Uvicorn

Uvicorn là một web server ASGI siêu nhanh, đóng vai trò như một người trung chuyển tiếp nhận các kết nối mạng HTTP/WebSocket thô, sau đó chuyển giao chúng thành định dạng dữ liệu ASGI tiêu chuẩn để ứng dụng FastAPI có thể xử lý.

3. Viết code: def vs async def

Quy tắc vàng khi lập trình ứng dụng với FastAPI:

  • Dùng async def khi trong logic xử lý của bạn có các lời gọi hàm I/O hỗ trợ async (có từ khóa await), hoặc khi bạn đang tương tác với các external API của các nhà cung cấp AI (như OpenAI, Anthropic) bằng thư viện HTTP client hỗ trợ async.
  • Dùng def thông thường khi bạn gọi các thư viện đồng bộ cũ (như thư viện kết nối cơ sở dữ liệu đồng bộ cổ điển, đọc file bằng hàm mặc định của Python). FastAPI sẽ tự động chạy các hàm def này trên một luồng phụ riêng (Thread Pool) để không chặn Event Loop chính.
Lỗi nghiêm trọng: Dùng async def kèm thư viện chặn (Blocking)

Nếu bạn định nghĩa hàm bằng async def nhưng bên trong lại gọi một hàm đồng bộ mất thời gian (ví dụ: time.sleep(5) hoặc gọi DB đồng bộ), Event Loop của toàn bộ ứng dụng sẽ bị đóng băng hoàn toàn. Mọi người dùng khác sẽ không thể truy cập hệ thống.

Recap: Uvicorn là server ASGI thực thi ứng dụng. Sử dụng async def đúng cách để tối ưu hóa hiệu năng, giúp server xử lý hàng ngàn request đồng thời.

Chương 4: Cài đặt & App đầu tiên

Mục tiêu

Thiết lập môi trường làm việc hiện đại bằng công cụ quản lý package `uv`, viết mã nguồn cho ứng dụng đầu tiên và chạy thử nghiệm.

Version Note: CLI FastAPI mới nhất

Trong các phiên bản FastAPI gần đây, bạn được khuyến khích sử dụng lệnh CLI chính thức của thư viện là fastapi dev (thay thế cho uvicorn main:app --reload trước đây) và fastapi run cho môi trường production.

1. Cài đặt môi trường với `uv`

Chúng ta sẽ sử dụng công cụ quản lý package uv siêu tốc của astral. Chạy các lệnh sau trong Terminal:

# Tạo môi trường ảo
uv venv
# Kích hoạt môi trường ảo (Windows)
.venv\Scripts\activate
# Cài đặt fastapi và công cụ chạy server
uv add fastapi uvicorn

2. Code: Hello World App [C01]

main.py 
from fastapi import FastAPI

# Khởi tạo instance ứng dụng FastAPI
app = FastAPI(title="FastAPI Handbook Hello App")

@app.get("/")
async def root():
    # Trả về một dictionary, FastAPI sẽ tự động chuyển thành JSON

Giải thích mã nguồn main.py
  • Mục tiêu: Tạo một API endpoint đơn giản nhất trả về dòng chữ chào mừng.
  • Luồng chạy: Khi server nhận request HTTP GET vào đường dẫn gốc /, hàm root() sẽ được thực thi bất đồng bộ và trả về một Python dictionary. FastAPI tự động chuyển đối tượng này sang định dạng JSON và trả về client kèm header Content-Type: application/json.

3. Khởi chạy ứng dụng và truy cập API Docs

Để chạy ứng dụng ở chế độ phát triển (development), chạy lệnh sau:

  • Tài liệu ReDoc: http://127.0.0.1:8000/redoc

Recap: Viết app cực nhanh với cú pháp `app = FastAPI()`, sử dụng decorator `@app.get("/")` để định tuyến và chạy nhanh bằng lệnh `fastapi dev`.

Chương 5: Tạo Endpoint cơ bản

Mục tiêu

Làm quen với các decorator định tuyến (routing) cơ bản để tiếp nhận các phương thức GET và POST, trả về các kiểu cấu trúc dữ liệu khác nhau.

1. Cú pháp định tuyến (Route Decorators)

FastAPI hỗ trợ toàn bộ các phương thức HTTP thông qua các hàm decorator trực tiếp trên thực thể ứng dụng: `@app.get()`, `@app.post()`, `@app.put()`, `@app.delete()`...

2. Code: GET & POST Basics [C02]

routes_demo.py 
from fastapi import FastAPI
from typing import List, Dict

app = FastAPI()

# Endpoint GET: Truy xuất thông tin danh sách sản phẩm
@app.get("/products")
async def get_products() -> List[Dict[str, float]]:
    return [
        {"id": 1.0, "price": 99.9},
        {"id": 2.0, "price": 49.5}
    ]

# Endpoint POST: Nhận dữ liệu để tạo mới sản phẩm
@app.post("/products")
async def create_product(payload: Dict[str, str]):
    # Xử lý tạo sản phẩm (ở đây mock phản hồi)
    return {
        "status": "created",
        "product_name": payload.get("name")
    }
Giải thích mã nguồn routes_demo.py
  • Mục tiêu: Tạo route GET để lấy danh sách sản phẩm và POST để tạo sản phẩm.
  • Cách chạy: Endpoint POST yêu cầu Client gửi dữ liệu định dạng JSON lên. FastAPI sẽ tự động phân giải request body thô thành một Python dictionary và đưa vào biến payload.

Recap: Dùng `@app.get` cho các tác vụ lấy dữ liệu và `@app.post` khi cần gửi dữ liệu thô dạng JSON lên để tạo mới tài nguyên.

Chương 6: Path, Query & Body

Mục tiêu

Phân biệt và kết hợp linh hoạt ba nguồn truyền nhận dữ liệu phổ biến: Path Parameter, Query Parameter và Request Body.

1. Định nghĩa kỹ thuật

  • Path Parameter: Là một phần cố định của URL, được định vị bằng cặp ngoặc nhọn {param}.
  • Query Parameter: Các tham số tùy chọn hoặc bổ sung nằm sau dấu ? trên URL.
  • Body: Dữ liệu JSON gửi ngầm trong request payload.

2. Code: Tích hợp Path, Query & Body [C03, C04, C05]

params_demo.py 
from fastapi import FastAPI
from typing import Optional

app = FastAPI()

# Endpoint tổng hợp nhận cả 3 loại tham số
@app.put("/items/{item_id}")
async def update_item(
    item_id: int,                      # Path parameter (bắt buộc)
    q: Optional[str] = None,          # Query parameter (tùy chọn)
    payload: dict = None              # Request body
):
    return {
        "item_id": item_id,
        "query_filter": q,
        "updated_data": payload
    }
Phân tích hoạt động của params_demo.py
  • Item ID: Kiểu int. Nếu client cố ý truyền chuỗi không thể convert thành số (ví dụ: `/items/abc`), FastAPI sẽ trả về lỗi HTTP 422 Validation Error ngay lập tức.
  • Tham số q: Là một Query Parameter tùy chọn vì có giá trị mặc định là None.
Request Example (PUT)
PUT /items/105?q=search_term
Content-Type: application/json

{
  "name": "Sách lập trình Python",
  "price": 250000
}
Response Example
{
  "item_id": 105,
  "query_filter": "search_term",
  "updated_data": {
    "name": "Sách lập trình Python",
    "price": 250000
  }
}

Recap: FastAPI đủ thông minh để tự động phân tích: nếu biến trùng tên với định tuyến thì là Path param, nếu có kiểu dữ liệu nguyên thủy (primitive) thì là Query param, nếu là class phức tạp thì là Request Body.

Chương 7: Validation với Pydantic

Mục tiêu

Sử dụng thư viện Pydantic để định nghĩa các mô hình dữ liệu (schema) chặt chẽ, ràng buộc kiểu dữ liệu, chiều dài, giá trị tối thiểu/tối đa và cấu trúc lồng nhau.

Version Note: Pydantic v2 trong FastAPI mới

FastAPI hiện tại đã sử dụng Pydantic v2. Khai báo schema kế thừa từ pydantic.BaseModel. Định nghĩa ràng buộc nâng cao sử dụng hàm Field.

1. Khai báo BaseModel & Field Constraints

Pydantic bảo vệ ứng dụng của bạn tại "cửa ngõ". Nếu dữ liệu client gửi lên không đúng định dạng khai báo, request sẽ bị từ chối kèm mô tả lỗi chi tiết, giúp backend không bao giờ nhận dữ liệu "bẩn".

2. Code: Nested Model & Constraints [C06]

pydantic_models.py 
from fastapi import FastAPI
from pydantic import BaseModel, Field, EmailStr
from typing import List, Optional

app = FastAPI()

class Tag(BaseModel):
    name: str = Field(..., min_length=2, max_length=20)
    category: str

class UserProfile(BaseModel):
    email: EmailStr  # Tự động validate định dạng email hợp lệ
    username: str = Field(..., min_length=3, description="Tên đăng nhập")
    age: Optional[int] = Field(None, ge=18, le=100) # Tuổi từ 18 đến 100
    tags: List[Tag] = []  # Nested model: Chứa danh sách các đối tượng Tag

@app.post("/users/profile")
async def create_profile(profile: UserProfile):
    return {"message": "Dữ liệu hợp lệ!", "profile": profile}
Giải thích luồng chạy của pydantic_models.py
  • EmailStr: Yêu cầu cài thêm package email-validator để xác thực chuỗi email hợp lệ.
  • Field(..., min_length=3): Dấu ba chấm ... biểu thị đây là trường bắt buộc (required).
  • Nested validation: Khi nhận dữ liệu, Pydantic duyệt qua danh sách `tags`, validate từng phần tử xem có đủ trường `name` và `category` hay không trước khi truyền đối tượng vào hàm xử lý chính.
Best Practice: Tái sử dụng Schema

Không dùng chung một Schema cho tất cả thao tác. Hãy tách biệt rõ ràng các schemas: UserCreate (cần mật khẩu), UserRead (không có mật khẩu), UserUpdate (tất cả các trường đều là Optional).

Recap: Pydantic chuyển đổi kiểu và validate dữ liệu cực mạnh. Sử dụng nested model giúp mô tả các cấu trúc JSON phức tạp từ client một cách dễ dàng.

Chương 8: Response Models và Thiết kế Response

Mục tiêu

Sử dụng tham số `response_model` để ẩn các thông tin nhạy cảm của hệ thống trước khi gửi kết quả về cho client, đồng thời định nghĩa chuẩn hóa cấu trúc HTTP status codes.

1. Tầm quan trọng của việc lọc dữ liệu

Thông thường, đối tượng người dùng lưu trong DB sẽ chứa mật khẩu đã băm (password hash) hoặc các token quản trị. Chúng ta tuyệt đối không được trả trực tiếp các thông tin này về phía người dùng cuối. response_model hoạt động như một màng lọc dữ liệu.

2. Code: Response Model & Filtering [C07]

response_demo.py 
from fastapi import FastAPI, status
from pydantic import BaseModel, EmailStr
from typing import Optional

app = FastAPI()

# Schema dữ liệu đầu vào (Client gửi lên)
class UserIn(BaseModel):
    username: str
    email: EmailStr
    password: str  # Chứa mật khẩu thô

# Schema dữ liệu đầu ra (Server trả về)
class UserOut(BaseModel):
    username: str
    email: EmailStr
    is_active: bool = True  # Không chứa password

@app.post("/users", response_model=UserOut, status_code=status.HTTP_201_CREATED)
async def register_user(user: UserIn):
    # Mock xử lý lưu vào Database
    # Trả về toàn bộ thông tin (kèm password), FastAPI sẽ tự động lọc chỉ giữ lại trường trong UserOut
    return {
        "username": user.username,
        "email": user.email,
        "password": "hashed_secret_password_123",
        "is_active": True
    }
Giải thích luồng chạy response_demo.py
  • Mục tiêu: Tạo user và tự động ẩn mật khẩu ở phản hồi.
  • Cơ chế lọc: Hàm register_user trả về một dictionary chứa cả trường password. Nhưng vì chúng ta định nghĩa response_model=UserOut tại decorator, FastAPI sẽ parse dữ liệu qua UserOut và loại bỏ triệt để trường password trước khi trả về.
  • Status Code 201: Trả về mã 201 Created thay vì 200 mặc định, tuân thủ chặt chẽ REST API standard khi tạo mới tài nguyên.

Recap: `response_model` là vũ khí bảo mật quan trọng của FastAPI giúp định hình chính xác cấu trúc dữ liệu trả về và bảo vệ thông tin mật.

Chương 9: Error Handling và API Contract

Mục tiêu

Xử lý ngoại lệ (exceptions) một cách có hệ thống, không để lộ lỗi crash hệ thống (internal stack trace) cho client, trả về cấu trúc lỗi JSON nhất quán.

1. Sử dụng HTTPException

FastAPI cung cấp lớp HTTPException kế thừa từ các lớp lỗi cơ bản để bạn kích hoạt lỗi kèm HTTP status code và nội dung tùy chọn tại bất kỳ vị trí nào trong ứng dụng.

2. Code: Custom Exception Handler [C08, C09]

error_handling.py 
from fastapi import FastAPI, HTTPException, Request, status
from fastapi.responses import JSONResponse

app = FastAPI()

# Định nghĩa một Class Exception tự chế
class DatabaseConnectionException(Exception):
    def __init__(self, name: str):
        self.name = name

# Đăng ký handler xử lý toàn cục cho DatabaseConnectionException
@app.exception_handler(DatabaseConnectionException)
async def db_exception_handler(request: Request, exc: DatabaseConnectionException):
    return JSONResponse(
        status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
        content={
            "error_code": "DATABASE_ERROR",
            "message": f"Hệ thống cơ sở dữ liệu tạm thời gián đoạn: {exc.name}"
        }
    )

@app.get("/items/{item_id}")
async def read_item(item_id: int):
    if item_id == 0:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="Item ID không thể bằng 0."
        )
    if item_id == 999:
        # Giả lập lỗi kết nối database
        raise DatabaseConnectionException(name="PostgreSQL-Primary")
    return {"item_id": item_id}
Giải thích luồng xử lý lỗi
  • Mục tiêu: Bắt lỗi client (400 Bad Request) và lỗi hệ thống (503 Service Unavailable) một cách an toàn.
  • Exception Handler: Khi DatabaseConnectionException bị kích hoạt ở bất cứ đâu, hàm db_exception_handler sẽ hứng lại lỗi đó và trả về định dạng JSON thống nhất, ngăn ngừa rò rỉ mã lỗi Python thô ra màn hình của client.

Recap: Sử dụng `HTTPException` cho các lỗi logic nghiệp vụ thông thường, và `@app.exception_handler()` để xử lý tập trung các lỗi đặc thù của hệ thống.

1. Upload file và Form data trong FastAPI

headers_files.py 
from fastapi import FastAPI, Header, Cookie, Form, File, UploadFile
from typing import Optional

app = FastAPI()

@app.post("/upload")
async def handle_assets(
    user_agent: Optional[str] = Header(None),         # Đọc User-Agent từ Header
    session_token: Optional[str] = Cookie(None),     # Đọc session_token từ Cookie

    uploaded_file: UploadFile = File(...)              # Nhận file tải lên

Chương 11: Dependency Injection

Mục tiêu

Hiểu sâu sắc về cơ chế Tiêm phụ thuộc (Dependency Injection) trong FastAPI, cách tái sử dụng logic xử lý, giải phóng tài nguyên một cách an toàn.

1. Dependency Injection là gì?

Dependency Injection (DI) là một mô hình thiết kế phần mềm cho phép một hàm hoặc class nhận các phụ thuộc (các dịch vụ, kết nối cơ sở dữ liệu) từ bên ngoài thay vì tự khởi tạo chúng bên trong thân hàm. Trong FastAPI, DI hoạt động như một **đồ thị phụ thuộc có hướng, không có chu trình (Directed Acyclic Graph - DAG)**. FastAPI sẽ tự phân tích và giải quyết các dependency lồng nhau theo đúng thứ tự trước khi đưa tham số cuối cùng vào hàm route chính.

2. Code: Basic DI & Yield Dependency [C12, C13]

di_demo.py 
from fastapi import FastAPI, Depends, Header, HTTPException
from typing import Generator

app = FastAPI()

# 1. Dependency kiểm tra API Key từ Header
async def verify_api_key(x_api_key: str = Header(...)):
    if x_api_key != "secret-handbook-key":
        raise HTTPException(status_code=403, detail="API Key không hợp lệ!")
    return x_api_key

# 2. Dependency quản lý tài nguyên sạch (sử dụng yield)
async def get_db_session() -> Generator[str, None, None]:
    try:
        # Bắt đầu vòng đời kết nối
        db_connection = "SQLAlchemy-Session-Connected"
        print("Mở DB Connection Session")
        yield db_connection  # Cung cấp session cho endpoint sử dụng
    finally:
        # Điểm dọn dẹp (luôn chạy sau khi endpoint hoàn thành gửi response)
        print("Đóng DB Connection Session an toàn")

# Route sử dụng cả 2 dependencies
@app.get("/secure-data")
async def get_data(
    api_key: str = Depends(verify_api_key),
    db: str = Depends(get_db_session)
):
    return {
        "status": "authorized",
        "api_key_used": api_key,
        "db_session_status": db
    }
Giải thích cơ chế vận hành DI
  • Mục tiêu: Bảo vệ route bằng API key và cấp phát session DB tự động dọn dẹp.
  • yield cơ chế: Hàm get_db_session thực hiện phần code phía trước lệnh yield trước khi route chạy, sau đó tạm dừng tại `yield`. Sau khi route get_data trả về kết quả cho client, phần code trong khối finally mới tiếp tục chạy để đóng kết nối, đảm bảo không bao giờ rò rỉ (leak) tài nguyên DB.
Production Note: Dependency Injection và Unit Testing

DI là nhân tố quan trọng nhất giúp viết test dễ dàng trong FastAPI. Bạn có thể dễ dàng hoán đổi cơ sở dữ liệu thật sang cơ sở dữ liệu thử nghiệm trong các bài kiểm thử tự động bằng tính năng override dependencies của FastAPI mà không cần sửa một dòng code nào trong route.

Recap: Dùng `Depends` để tiêm logic. Sử dụng `yield` bên trong các dependency để tự động đóng kết nối DB hoặc giải phóng tài nguyên mạng.

Chương 13: Database Foundations

Mục tiêu

Làm quen với thư viện SQLAlchemy ORM, hiểu cấu trúc kết nối cơ sở dữ liệu và quản lý vùng nhớ Connection Pool trong ứng dụng Web.

1. Vai trò của ORM

Object-Relational Mapping (ORM) giúp lập trình viên tương tác với cơ sở dữ liệu quan hệ (SQLite, PostgreSQL, MySQL) bằng cách sử dụng các đối tượng Python thuần túy thay vì phải viết các câu lệnh SQL thô dễ gây lỗi cú pháp.

Để quản lý kết nối cơ sở dữ liệu đúng cách, chúng ta khởi tạo engine dùng chung và một nhà máy sản xuất các session phục vụ các giao dịch đơn lẻ:

database_config.py 
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base

# Kết nối database thử nghiệm SQLite
DATABASE_URL = "sqlite:///./app_database.db"

# Khởi tạo engine dùng chung. Đối với PostgreSQL, pool_size sẽ cấu hình tại đây
engine = create_engine(
    DATABASE_URL, 
    connect_args={"check_same_thread": False}  # Chỉ cần thiết đối với SQLite
)

# Tạo một sessionmaker để khởi tạo các phiên transaction độc lập
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# Base class để các DB Models kế thừa

Giải thích database_config.py
  • Mục tiêu: Thiết lập cấu hình kết nối DB ban đầu.
  • check_same_thread=False: Cho phép nhiều luồng bất đồng bộ trong FastAPI cùng truy cập database SQLite an toàn.

Recap: Sử dụng ORM như SQLAlchemy giúp trừu tượng hóa các câu lệnh SQL. Cần kiểm soát chặt chẽ vòng đời của Session để tránh lỗi rò rỉ bộ nhớ cơ sở dữ liệu.

Chương 14: CRUD Application

Mục tiêu

Xây dựng một hệ thống CRUD (Create, Read, Update, Delete) hoàn chỉnh, tương tác trực tiếp với cơ sở dữ liệu thông qua Repository pattern.

1. Tạo cấu trúc Database Model & Schema

Chúng ta tách biệt Database Model (SQLAlchemy) và Data Schema (Pydantic) để đảm bảo tính an toàn dữ liệu.

2. Code: CRUD & DB Integration [C16]

db_crud.py 
from fastapi import FastAPI, Depends, HTTPException, status
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker, Session
from pydantic import BaseModel
from typing import Generator

# Cấu hình Database SQLite
SQLALCHEMY_DATABASE_URL = "sqlite:///./test_handbook.db"
engine = create_engine(SQLALCHEMY_DATABASE_URL, connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()

# 1. SQLAlchemy DB Model (Cấu trúc bảng lưu trong DB)
class BookDB(Base):
    __tablename__ = "books"
    id = Column(Integer, primary_key=True, index=True)
    title = Column(String, index=True)
    author = Column(String)

# Tạo bảng tự động (chỉ dùng cho SQLite thử nghiệm)
Base.metadata.create_all(bind=engine)

# 2. Pydantic Schemas
class BookCreate(BaseModel):
    title: str
    author: str

class BookOut(BaseModel):
    id: int
    title: str
    author: str
    class Config:
        from_attributes = True # Cho phép chuyển đổi từ thực thể ORM SQLAlchemy

# 3. Dependency lấy DB Session
def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db  # Đưa session vào route
    finally:
        db.close() # Đóng kết nối an toàn sau khi hoàn thành request

app = FastAPI()

# POST: Tạo sách mới
@app.post("/books", response_model=BookOut, status_code=status.HTTP_201_CREATED)
def create_book(book: BookCreate, db: Session = Depends(get_db)):
    new_book = BookDB(title=book.title, author=book.author)
    db.add(new_book)
    db.commit()
    db.refresh(new_book) # Lấy dữ liệu mới nhất kèm ID tự sinh từ DB
    return new_book

# GET: Đọc thông tin sách
@app.get("/books/{book_id}", response_model=BookOut)
def read_book(book_id: int, db: Session = Depends(get_db)):
    db_book = db.query(BookDB).filter(BookDB.id == book_id).first()
    if not db_book:
        raise HTTPException(status_code=404, detail="Không tìm thấy cuốn sách này!")
    return db_book
Giải thích luồng chạy CRUD
  • Mục tiêu: Triển khai việc lưu trữ và truy xuất thực thể `Book` vào DB SQLite.
  • from_attributes = True: (Trong Pydantic v2) Cho phép Pydantic đọc trực tiếp các trường từ DB Model trả về (như book.title) thay vì bắt buộc dữ liệu phải là định dạng dict.
  • Vòng đời DB Session: Được bao bọc hoàn hảo trong dependency `get_db`. Lệnh db.close() luôn chạy cuối cùng để giải phóng kết nối về Pool.

Recap: Tiến trình CRUD chuẩn mực: Nhận dữ liệu đầu vào qua Pydantic -> Lưu trữ qua SQLAlchemy ORM -> Trả về dữ liệu được chuẩn hóa qua `response_model`.

Mục tiêu

Part E: Security and Authentication

Chương 16: Security Fundamentals

Mục tiêu

Phân biệt rõ ràng giữa hai khái niệm: Xác thực (Authentication) và Phân quyền (Authorization), tìm hiểu cơ chế băm mật khẩu mật mã một chiều (Password Hashing).

1. Authentication vs Authorization

  • Authentication (Xác thực): Là bước xác định danh tính của người dùng ("Bạn là ai?"). Thường thực hiện qua đăng nhập với Email/Password hoặc API Keys.
  • Authorization (Phân quyền): Là bước xác định xem người dùng đã được xác thực đó có được quyền truy cập tài nguyên cụ thể hay không ("Bạn được làm gì?").

2. Cơ chế băm mật khẩu an toàn

Tuyệt đối không lưu trữ mật khẩu thô của người dùng trong DB. Sử dụng các thuật toán băm chậm, có bổ sung muối muối ngẫu nhiên (salt) như bcrypt để chống lại các đòn tấn công vét cạn (brute-force) hoặc tra cứu bảng cầu vồng (rainbow tables).

3. Code: Password Hashing [C18]

Yêu cầu cài đặt thư viện mã hóa: uv add "passlib[bcrypt]"

hashing_util.py 
from passlib.context import CryptContext

# Thiết lập ngữ cảnh mã hóa sử dụng thuật toán bcrypt
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    # Sinh muối ngẫu nhiên và băm mật khẩu thô
    return pwd_context.hash(password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    # Đối sánh mật khẩu thô nhập vào với mật khẩu đã băm trong Database
    return pwd_context.verify(plain_password, hashed_password)

# Chạy thử nghiệm kiểm tra tính đúng đắn
if __name__ == "__main__":
    my_pass = "admin1234"
    hashed = hash_password(my_pass)
    print(f"Mật khẩu thô: {my_pass}")
    print(f"Bản băm lưu Database: {hashed}")
    print(f"Kết quả khớp: {verify_password(my_pass, hashed)}")

Recap: Không bao giờ lưu mật khẩu dạng plaintext. Sử dụng `CryptContext` của thư viện `passlib` để băm và đối sánh mật khẩu an toàn.

Chương 17: OAuth2 và JWT Thực Chiến

Mục tiêu

Triển khai quy trình đăng nhập cấp mã thông báo bảo mật dạng JWT (JSON Web Token), xây dựng các middleware và dependencies để bảo vệ các tuyến đường dẫn (protected routes).

1. JSON Web Token (JWT) hoạt động như thế nào?

JWT là mã thông báo tự chứa thông tin (stateless token). Nó bao gồm 3 phần phân tách bởi dấu chấm: Header.Payload.Signature. Phần chữ ký được tạo ra bằng khóa bí mật (Secret Key) phía server để đảm bảo dữ liệu không bị sửa đổi.

2. Code: Đăng nhập & Xác thực JWT [C19, C20]

Yêu cầu cài đặt: uv add pyjwt python-multipart

auth_server.py 
from fastapi import FastAPI, Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm
import jwt
import os
from datetime import datetime, timedelta, timezone
from typing import Optional

# Cấu hình bảo mật JWT
# WARNING: Tuyệt đối không hardcode khóa bảo mật trong mã nguồn!
# Đọc trực tiếp từ biến môi trường của hệ điều hành. Ứng dụng sẽ báo lỗi ngay nếu thiếu khóa.
SECRET_KEY = os.environ.get("SECRET_KEY")
if not SECRET_KEY:
    raise RuntimeError("Biến môi trường SECRET_KEY chưa được cấu hình. Ứng dụng không thể khởi chạy an toàn!")
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30

# Công cụ phân tích mã token từ Header Authorization Bearer
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="login")

app = FastAPI()

# 1. Hàm tạo JWT Token
def create_access_token(data: dict, expires_delta: Optional[timedelta] = None):
    to_encode = data.copy()
    if expires_delta:
        expire = datetime.now(timezone.utc) + expires_delta
    else:
        expire = datetime.now(timezone.utc) + timedelta(minutes=15)
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

# 2. Endpoint Đăng nhập: Cấp token
@app.post("/login")
async def login(form_data: OAuth2PasswordRequestForm = Depends()):
    # Xác thực người dùng (ở đây giả định xác thực cứng)
    if form_data.username != "hieu" or form_data.password != "password":
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Tên đăng nhập hoặc mật khẩu không chính xác"
        )
    access_token = create_access_token(data={"sub": form_data.username})
    return {"access_token": access_token, "token_type": "bearer"}

# 3. Dependency lấy thông tin User hiện tại từ JWT Token
async def get_current_user(token: str = Depends(oauth2_scheme)):
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Không thể xác thực thông tin đăng nhập",
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        username: str = payload.get("sub")
        if username is None:
            raise credentials_exception
    except jwt.PyJWTError:
        raise credentials_exception
    return {"username": username}

# 4. Route được bảo vệ (Protected Route)
@app.get("/users/me")
async def read_current_user(current_user: dict = Depends(get_current_user)):
    return {"message": "Truy cập thông tin thành công!", "user": current_user}
Giải thích luồng xác thực JWT
  • Mục tiêu: Tạo route đăng nhập và bảo vệ dữ liệu nhạy cảm của người dùng hiện tại mà không để lộ khóa bí mật.
  • os.environ.get: Đảm bảo khóa mật mã của hệ thống luôn được nạp từ biến cấu hình ngoài, ngăn chặn hoàn toàn việc rò rỉ mã bảo mật khi đẩy code lên Github.
Cảnh báo: BOLA / IDOR Vulnerability

Lỗi thiết kế bảo mật phổ biến nhất (Broken Object Level Authorization): Endpoint dạng /users/{user_id}/invoice đọc dữ liệu trực tiếp dựa trên biến user_id mà quên không kiểm tra xem user_id này có trùng khớp với danh tính giải mã ra từ JWT Token hay không. Người dùng A có thể đổi ID thành của người dùng B để đọc lén dữ liệu.

Recap: Đăng nhập -> Cấp Token JWT ký bằng Secret Key -> Client đính kèm Header `Authorization: Bearer ` -> Sử dụng `Depends(get_current_user)` bảo vệ API.

Chương 18: Middleware và CORS

Mục tiêu

Hiểu cơ chế hoạt động của Middleware, đăng ký CORS (Cross-Origin Resource Sharing) để cho phép ứng dụng frontend kết nối API an toàn.

1. Middleware là gì?

Middleware là một hàm trung gian chặn (intercept) tất cả các request đi vào ứng dụng và các response đi ra trước khi gửi về client, thích hợp cho việc ghi log, đo lường thời gian đáp ứng hệ thống (performance tracing).

2. Code: Middleware ghi log & CORS config [C21, C22]

middleware_demo.py 
from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
import time

app = FastAPI()

# 1. Cấu hình CORS Middleware
origins = [
    "http://localhost:3000", # Cho phép frontend React chạy trên port 3000 kết nối
    "https://myproductionapp.com"
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=True,
    allow_methods=["*"], # Cho phép toàn bộ các methods GET, POST, PUT, DELETE...
    allow_headers=["*"], # Cho phép toàn bộ các Custom Headers gửi lên
)

# 2. Tự viết Middleware đo lường thời gian chạy (Timing Middleware)
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.time()
    # Chuyển giao request cho các router xử lý chính
    response = await call_next(request)
    # Đo đạc thời gian sau khi router xử lý xong
    process_time = time.time() - start_time
    response.headers["X-Process-Time"] = f"{process_time:.4f} seconds"
    return response

@app.get("/ping")
async def ping():
    return {"message": "pong"}

  • Mục tiêu: Hỗ trợ gọi API chéo tên miền (CORS) và gán thêm header đo tốc độ xử lý vào response.
  • call_next: Là hàm nhận vào request và trả về đối tượng response sau khi đã đi qua toàn bộ các lớp xử lý của API.

Recap: CORS chặn các trang web ngoài tên miền gọi API của bạn. Đăng ký CORSMiddleware đúng cách để tháo gỡ rào cản chặn tên miền của trình duyệt.

Chương 19: Lifespan và Events

Mục tiêu

Sử dụng cơ chế lifespan mới để thực hiện các thao tác khởi động (startup) và dọn dẹp (shutdown) tài nguyên hệ thống dùng chung.

Version Note: Lifespan thay thế on_event

Các hàm decorator cũ như @app.on_event("startup") đã bị loại bỏ trong các phiên bản mới của FastAPI. Hãy sử dụng context manager lifespan truyền trực tiếp vào thực thể ứng dụng.

1. Ý nghĩa của Lifespan

Khi ứng dụng chạy, chúng ta cần nạp các tài nguyên nặng và giữ lại bộ nhớ của nó (như kết nối Pool cơ sở dữ liệu, nạp các mô hình mạng neural học máy lớn, kết nối HTTP client dùng chung). Khi tắt server, ta phải giải phóng các bộ nhớ này một cách an toàn để tránh thất thoát hoặc hỏng hóc dữ liệu.

2. Code: Lifespan Manager [C23]

lifespan_demo.py 
from fastapi import FastAPI
from contextlib import asynccontextmanager
import httpx

# Mock class đại diện cho Model AI lớn cần nạp sẵn
class MachineLearningModel:
    def load(self):
        print("Đang nạp mô hình AI nặng vào GPU...")
    def predict(self, text: str) -> str:
        return f"Mô hình AI dự đoán cho từ: {text}"
    def unload(self):
        print("Giải phóng mô hình khỏi GPU bộ nhớ")

# Định nghĩa Lifespan Manager
@asynccontextmanager
async def lifespan(app: FastAPI):
    # [STARTUP STATE] - Thực hiện khi ứng dụng bắt đầu khởi chạy
    ai_model = MachineLearningModel()
    ai_model.load()
    shared_client = httpx.AsyncClient()
    
    # Gán trực tiếp vào app.state trước khi yield để các route truy cập an toàn
    app.state.ai_model = ai_model
    app.state.shared_client = shared_client
    
    yield  # Nhường quyền điều khiển chạy ứng dụng chính
    
    # [SHUTDOWN STATE] - Thực hiện khi tắt ứng dụng
    await shared_client.aclose()
    ai_model.unload()
    print("Dọn dẹp hệ thống hoàn tất.")

app = FastAPI(lifespan=lifespan)

@app.get("/predict")
async def get_prediction(text: str):
    # Truy xuất tài nguyên dùng chung từ state của app
    ai_model = app.state.ai_model
    result = ai_model.predict(text)
    return {"prediction": result}
Giải thích hoạt động của Lifespan
  • Mục tiêu: Quản lý tài nguyên nặng (ML Model, HTTP Client) xuyên suốt chu kỳ sống của Server.
  • app.state: Các tài nguyên nạp vào app.state trong giai đoạn startup sẽ tồn tại ổn định, giúp các route truy cập dễ dàng mà không bị khởi tạo lại ở mỗi request.

Recap: Dọn dẹp hệ thống khi tắt, khởi tạo dùng chung khi bật thông qua lifespan context manager để tránh rò rỉ bộ nhớ GPU/RAM.

Chương 20: Background, SSE & WebSockets

Mục tiêu

Khám phá thế giới kết nối realtime thông qua các công nghệ truyền phát dữ liệu (Streaming Response), Server-Sent Events (SSE), WebSockets và xử lý tác vụ chạy nền (Background Tasks).

1. Khi nào chọn công nghệ nào?

  • Background Tasks: Chạy các tác vụ tốn thời gian phụ (ví dụ: Gửi email xác nhận sau khi đăng ký) ngay trong luồng xử lý web mà không chặn response trả về.
  • Streaming Response: Gửi từng mảnh dữ liệu thô (chunk) về client liên tục thay vì nén dữ liệu gửi một lần (ví dụ: Token streaming từ OpenAI).
  • SSE (Server-Sent Events): Kết nối một chiều từ Server truyền sự kiện thời gian thực tới Client, cực kỳ nhẹ nhàng (qua giao thức HTTP thông thường).
  • WebSockets: Kết nối hai chiều liên tục (full-duplex) giữa Client và Server (ví dụ: Ứng dụng chat trực tuyến).

2. Code: Tổng hợp realtime & Background [C24, C25, C26, C27]

realtime_demo.py 
from fastapi import FastAPI, BackgroundTasks, WebSocket, WebSocketDisconnect
from fastapi.responses import StreamingResponse
import asyncio
import json
import time

app = FastAPI()

# 1. Background Tasks: Giả lập tác vụ nặng
def write_log_file(message: str):
    time.sleep(2) # Giả lập ghi file chậm
    print(f"[BACKGROUND LOG]: {message}")

@app.post("/trigger-task")
def trigger(message: str, background_tasks: BackgroundTasks):
    # Đăng ký tác vụ chạy nền, response trả về ngay lập tức
    background_tasks.add_task(write_log_file, message)
    return {"message": "Tác vụ chạy nền đã được đăng ký!"}

# 2. StreamingResponse & SSE: Gửi từng mảnh dữ liệu
async def generator_stream():
    for i in range(5):
        await asyncio.sleep(1)
        # Định dạng chuẩn SSE bắt đầu bằng "data: "
        yield f"data: {json.dumps({'message': f'Dữ liệu mảnh {i}'})}\n\n"

@app.get("/sse")
def get_stream():
    return StreamingResponse(generator_stream(), media_type="text/event-stream")

# 3. WebSockets: Giao tiếp hai chiều
@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept() # Chấp nhận kết nối
    try:
        while True:
            # Nhận tin nhắn gửi lên từ client
            data = await websocket.receive_text()
            # Gửi phản hồi ngược lại client thời gian thực
            await websocket.send_text(f"Server đã nhận tin nhắn: {data}")
    except WebSocketDisconnect:
        print("Client ngắt kết nối WebSocket")
Giải thích luồng chạy realtime_demo.py
  • Mục tiêu: Triển khai đồng thời cơ chế chạy nền, streaming dữ liệu (SSE) và kết nối socket.
  • generator_stream: Sử dụng cơ chế async generator (async def với yield) để liên tục phát dữ liệu mà không làm tốn tài nguyên server.

Recap: Dùng BackgroundTasks cho các logic phụ. Dùng StreamingResponse (SSE) cho Token LLM. Dùng WebSocket cho các ứng dụng tương tác hai chiều.

from fastapi.testclient import TestClient
import httpx
from httpx import ASGITransport
import pytest

app = FastAPI()

# Giả lập một dependency kiểm tra quyền quản trị
def get_user_role():
    return "guest"

@app.get("/admin-zone")
def admin_area(role: str = Depends(get_user_role)):
    if role != "admin":
        return {"status": "denied"}
    return {"status": "welcome"}

def test_guest_access():

def test_admin_access_with_override():
    # Thay thế hàm get_user_role bằng một hàm giả lập trả về "admin"

    response = client.get("/admin-zone")
    assert response.status_code == 200
    assert response.json() == {"status": "welcome"}
    
    # Dọn dẹp override sau khi hoàn thành test để không ảnh hưởng test khác
    app.dependency_overrides.clear()

async def test_async_endpoint():

        assert response.json() == {"status": "denied"}

Chương 22: Debugging và Quality Mindset

Mục tiêu

Xây dựng tư duy phát triển phần mềm chất lượng cao, định vị và giải quyết các lỗi nhập tuần hoàn (Circular Imports), lỗi bất đồng bộ bị khóa hay lỗi cấu trúc dữ liệu không khớp (Schema Mismatch).

1. Lỗi Import tuần hoàn (Circular Imports)

Triệu chứng: Server crash khi khởi chạy kèm thông báo lỗi: cannot import name 'X' from partially initialized module.

Nguyên nhân: Module A import module B và Module B lại import ngược lại module A để lấy đối tượng. Thường xảy ra khi khai báo cấu trúc DB models lồng nhau hoặc dependencies.

2. Code: Giải quyết Circular Import

Để giải quyết lỗi import tuần hoàn, ta thực hiện import cục bộ bên trong hàm xử lý:

# File: auth_service.py

  • Mục tiêu: Tránh lỗi nạp chéo module.
  • Cơ chế: Khi Python khởi chạy file, nó sẽ không nạp ngay class UserManager, giải quyết hoàn toàn vấn đề vòng lặp nạp thư viện.

Recap: Đọc log lỗi từ dưới lên. Tách biệt cấu trúc dữ liệu và logic module để tránh lỗi import vòng và lỗi kiểu dữ liệu.

Chương 23: Settings và Environment Variables

Mục tiêu

Quản lý cấu hình ứng dụng (DATABASE_URL, API_KEYS) tập trung thông qua file biến môi trường `.env` sử dụng thư viện `pydantic-settings`.

Chương 24: Observability và Production Operations

Mục tiêu

Giám sát sức khỏe ứng dụng khi vận hành (observability) thông qua việc phân định ghi nhật ký (logs) có cấu trúc và thiết lập API đo lường trạng thái sống/chết (Health Checks).

1. Liveness vs Readiness Probes

Trong hệ thống vận hành thực tế (như chạy Kubernetes), hệ thống giám sát cần phân biệt:

  • Liveness Probe: Cho biết ứng dụng còn sống hay không. Nếu kiểm tra thất bại, Kubernetes sẽ lập tức khởi động lại container.
  • Readiness Probe: Cho biết ứng dụng đã sẵn sàng tiếp nhận lượng truy cập mạng hay chưa (đã kết nối DB xong chưa, nạp AI model xong chưa). Nếu thất bại, hệ thống sẽ tạm thời ngắt kết nối mạng đến container này chứ không khởi động lại nó.

2. Code: Liveness & Readiness Endpoints [C32]

Chương 25: Deployment và Docker

Mục tiêu

Đóng gói ứng dụng FastAPI thành một Container Docker chuẩn hóa, hiểu được thiết lập mạng khi đặt sau máy chủ chuyển tiếp (Reverse Proxy) và cách chạy uvicorn workers hiệu quả.

1. Kiến trúc Production thực tế

Trong thực tế, người ta không bao giờ phơi trực tiếp server Uvicorn ra ngoài Internet. Uvicorn sẽ được đặt sau một máy chủ Reverse Proxy (Nginx, Traefik, Cloudflare) để quản lý chứng chỉ HTTPS, giới hạn lưu lượng (rate limiting) và chống tấn công từ chối dịch vụ (DDoS).

Production Note: Chuyển tiếp Headers (Proxy Headers)

Khi ứng dụng đặt sau Nginx, địa chỉ IP của client truy cập vào ứng dụng sẽ luôn hiển thị là địa chỉ IP của Nginx (`127.0.0.1`). Để ghi nhận đúng IP thật và giao thức kết nối HTTPS của người dùng cuối, Nginx cần truyền các header: X-Forwarded-For và X-Forwarded-Proto. Ứng dụng FastAPI cần cấu hình Uvicorn nhận các header này bằng tùy chọn: --proxy-headers và --forwarded-allow-ips="*".

Chương 26: Scaling Basics

Mục tiêu

Nắm vững tư duy thiết kế ứng dụng không lưu trạng thái (Stateless API Design) để mở rộng hệ thống theo chiều ngang (Horizontal Scaling) không giới hạn.

1. Stateless API là gì?

Một API được gọi là Stateless nếu mỗi request gửi lên chứa đầy đủ thông tin để tự thực thi mà không phụ thuộc vào bộ nhớ tạm thời (RAM session) của riêng server đó. Nếu server A bị sập, server B có thể tiếp quản xử lý request thay thế mà không gây mất dữ liệu của người dùng.

2. Ứng phó áp lực kết nối DB (Connection Pressure)

Khi mở rộng quy mô chạy hàng chục Container ứng dụng song song, số lượng kết nối đồng thời đến Database sẽ bùng nổ vượt quá giới hạn của máy chủ DB. Giải pháp là thiết lập một máy chủ quản lý pool kết nối tập trung (như PgBouncer cho PostgreSQL) hoặc điều chỉnh cấu hình Connection Pool của SQLAlchemy ở mức vừa phải.

3. Code: Cấu hình Connection Pool cho SQLAlchemy

from sqlalchemy import create_engine

    pool_size=10,          # Số kết nối mặc định được mở sẵn và giữ lại

Part J: Common Mistakes, Best Practices, and Learning Roadmap

Chương 29: Sai lầm thường gặp

1. Không đóng DB Session

Mô tả: Khởi tạo DB Session bên trong route nhưng quên không gọi db.close() ở khối lệnh finally. Dẫn đến rò rỉ kết nối, làm treo server DB sau vài giờ chạy.

Giải pháp: Luôn dùng dependency Injection dạng yield để quản lý mở/đóng tự động.

2. Lạm dụng async cho các hàm blocking

Mô tả: Định nghĩa async def nhưng bên trong gọi các hàm đồng bộ nặng như requests.get() hoặc time.sleep().

Giải pháp: Đổi sang dùng def thông thường để FastAPI tự động điều hành chạy trên luồng phụ Thread Pool, hoặc chuyển đổi các thư viện này sang thư viện async tương ứng (ví dụ: dùng httpx.AsyncClient và asyncio.sleep()).

3. Trả về trực tiếp thực thể ORM

Mô tả: Trả về trực tiếp object Database Model trong route mà không định cấu hình lọc qua Pydantic `response_model`, dễ làm rò rỉ các trường thông tin nhạy cảm của người dùng.

4. Quên đặt Timeout cho các API AI ngoài

Mô tả: Gọi API của OpenAI hay Anthropic bằng httpx client mà không cấu hình Timeout. Nếu máy chủ của nhà cung cấp AI bị nghẽn, server của bạn sẽ bị treo chờ vô thời hạn.

Giải pháp: Luôn đặt timeout=10.0 khi thực hiện các cuộc gọi API.
